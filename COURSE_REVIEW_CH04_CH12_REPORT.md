# ch04-ch13 课程优化审查报告

审查范围：`src/content/docs/zh-cn/ch04-ai-math` 到 `src/content/docs/zh-cn/ch13-open-source-llm`。

审查方式：先递归盘点中文 Markdown 文件，再做结构扫描和重点抽读。重点检查：技术背景入口、实验是否支撑目标、抽象概念是否有直觉铺垫、工具/平台是否说明存在原因、是否保留评估证据和失败案例。

本报告是课程优化建议，不直接修改课程正文。

## 总体结论

整体上，ch08 RAG、ch09 Agent、ch12 多模态、ch13 开源大模型工程中的很多章节已经很好地体现了“先讲为什么存在，再讲机制和代码”的教学方式。尤其是 RAG 基础、LangChain 基础、Agent 基础、ReAct、MCP、扩散模型、开源模型 runbook 等章节，可以作为后续优化其他章节的模板。

主要待优化方向不是“补更多代码”，而是统一补齐三类内容：

1. 技术存在背景：首次出现新模型、新工具、新工程模式时，先回答它解决什么现实问题。
2. 实验任务边界：明确实验证明了什么、没有证明什么，避免学习者误以为玩具实验完整代表真实能力。
3. 证据与失败案例：项目和工程章节要让读者看到指标、失败样本、排查路径和下一步动作。

## 优先级建议

### P0：全课程统一增加“为什么存在”入口

涉及范围：ch04-ch13 所有首次引入核心技术的章节。

建议模板：

```text
现实中遇到什么问题？
不用这个技术时哪里不够？
这个技术新增了什么能力或取舍？
什么场景下应该想到它，什么场景下不一定需要它？
```

说明：这个问题已经在陪学过程中多次出现。建议最终形成课程写作规范，尤其应用到 CNN、RNN、Transformer、GAN、VAE、RAG、Agent、微调、压缩、部署、平台工具章节。

### P0：工具/生态类章节必须先讲平台背景

重点文件：

- `src/content/docs/zh-cn/ch07-llm-principles/ch01-nlp-crash/04-huggingface-quickstart.md`
- `src/content/docs/zh-cn/ch08-rag/ch03-app-dev/02-langchain-basics.md`
- `src/content/docs/zh-cn/ch09-agent/ch06-frameworks/*`
- 后续首次出现 OpenAI SDK、Docker、Kubernetes、向量数据库等工具章节

问题：学习者如果不知道 Hugging Face、LangChain、MCP、Docker 这类工具为什么存在，直接看 API 会觉得碎。

建议：首次出现时先说明它解决的工程痛点。例如 Hugging Face 可先解释为“模型、tokenizer、config、权重、数据集和训练工具的共享生态”，再进入 `AutoTokenizer` / `AutoModel`。

## 逐章建议

### ch04 AI 数学

优点：向量、信息论、链式法则等章节已经有较强直觉铺垫，能把数学和 AI 场景连起来。

建议：

- 在各小节开头统一增加“这个数学工具后面会在哪里用”的小框。例如向量对应 embedding/RAG，链式法则对应 PyTorch `backward()`，KL 对应 VAE/RLHF。
- 对公式密集章节增加“公式前直觉”标识，避免读者先被符号挡住。
- 数学工作坊可增加一张“从数学概念到第 6/7/8 章应用”的索引表，帮助学习者知道为什么学这些。

### ch05 机器学习

重点文件：

- `ch04-evaluation/03-bias-variance.md`
- `ch04-evaluation/04-hyperparameter-tuning.md`

建议：

- 偏差-方差和超参数调优已有实验，但可在开头补更明确的现实场景：模型效果不好时，为什么不能只换模型或盲目调参。
- 在超参数调优中加入“调参事故”示例：看 final holdout 后继续改 grid，导致 holdout 被污染。
- 项目章节可统一要求 baseline、失败样本、下一步动作，和 ch06/ch08 的证据包风格对齐。

### ch06 深度学习

已记录的重点优化：

- RNN 基础实验 3 不够体现顺序重要性。
- LSTM 需要先用“长期笔记 / 擦 / 写 / 读”铺垫。
- Sinusoidal 位置编码需要“多根钟表指针 / 多尺度位置身份证”铺垫。
- VAE 需要先讲现实用途：生成相似样本、平滑插值、异常检测、学习隐藏分布。

新增建议：

- 所有深度学习实验都建议固定增加“这个实验证明什么 / 没证明什么”。
- GAN/VAE/扩散等生成模型章节可统一加对照表：适用场景、训练稳定性、生成速度、可控性、常见失败模式。
- 模型压缩章节已经强调部署约束，建议加入一个 before/after 报告模板作为项目交付标准。

### ch07 大模型原理、Prompt 与微调

重点文件：

- `ch01-nlp-crash/04-huggingface-quickstart.md`
- `ch06-finetuning/02-lora-qlora.md`

建议：

- Hugging Face 章节需要先补“它是什么、为什么存在、解决什么工程重复劳动”。这是当前最明确的改进点之一。
- LoRA/QLoRA 章节已有不错的背景和矩阵直觉，可以作为微调相关章节模板。
- ch07 与 ch11 有 tokenizer、embedding、BERT/GPT 等内容重叠。建议在 ch07 开头明确定位：ch07 是 LLM 应用心智模型和工程选择；ch11 是传统 NLP 到预训练 NLP 的任务体系。避免读者觉得重复。
- Prompt 章节建议统一保留固定 cases、版本、输出、评分、失败样本，强化“Prompt 是可测试资产”的主线。

### ch08 RAG

优点：RAG 基础、向量数据库、LangChain 基础的背景入口很好。它们已经体现“先讲为什么需要，再讲组件”的方式。

建议：

- 把 RAG 基础的故障定位表抽象成全章统一排查卡：切块、召回、重排、上下文拼接、生成利用证据。
- 向量数据库章节可补“普通数据库 + 全文搜索 + 向量库”的对比表，帮助学习者判断什么时候不需要向量库。
- LangChain 章节可明确“适合原型和线性链路，不一定适合所有复杂生产系统”，避免框架崇拜。
- 文档解析、模板生成、Docker/API 章节建议同样套用平台背景模板。

### ch09 Agent

优点：Agent、ReAct、MCP 的背景入口很强，能清楚区分工作流、聊天机器人、Agent、函数调用、协议。

建议：

- 框架章节可统一增加“为什么要用框架，什么时候不用框架”的判断表。
- 多 Agent 章节建议增加最小失败案例：角色太多导致沟通成本高、任务分配不清、输出互相覆盖。
- 部署章节可进一步强调观测性和回放：Agent 出错时要能重放 trace、工具调用、状态变化。
- 对 ReAct/Plan-and-Execute 等模式，建议统一增加“适合 / 不适合”表，避免读者把所有任务都 Agent 化。

### ch10 计算机视觉

重点文件：

- `ch01-cv-basics/03-image-processing.md`
- `ch03-detection/01-detection-overview.md`
- `ch04-segmentation/01-semantic-segmentation.md`

建议：

- 图像处理章节需要更明确的现实背景：OCR 前处理、工业质检、医学影像、传统视觉系统为什么仍然用 OpenCV 和规则处理。
- 检测和分割章节的任务边界讲得较好，可作为 CV 任务引入模板。
- 图像处理代码会生成图片文件，建议加“如何查看输出图、每张图应该观察什么”的视觉检查清单。
- 目标检测/分割项目可增加错误类型图库：误检、漏检、框偏移、类别混淆、小目标失败、边界失败。

### ch11 NLP

优点：文本预处理章节很好地讲了“不是越重越好，而是任务驱动”；BiLSTM+CRF 也清楚说明了标签转移约束。

建议：

- ch11 和 ch07 有内容重叠，建议增加章节定位提示：ch11 更偏 NLP 任务谱系、传统方法和序列标注；ch07 更偏 LLM 应用和适配决策。
- 文本表示、词嵌入、上下文 embedding 可复用 ch07 的“token id 只是编号，embedding 才是语义坐标”解释方式。
- NER/Seq2Seq/CTC 等抽象机制建议统一加“非法输出或失败样本”示例，让读者理解为什么需要 CRF、attention、CTC。
- 文本分类项目可强化 error analysis：错分样本、类别不平衡、阈值、宏/微 F1。

### ch12 多模态

优点：扩散模型和 Stable Diffusion 背景入口较强，能把 GAN/VAE/扩散的关系讲清楚。

建议：

- `ch01-multimodal/01-multimodal-basics.md` 中 CLIP 背景有语义重复，建议精炼成一次完整说明，减少三段相近表述。
- 多模态基础可增加更多“为什么单模态不够”的现实任务，例如客服截图问答、医学图文、工业巡检视频+传感器。
- Stable Diffusion 章节已讲 VAE/U-Net/text encoder/cross-attention，建议增加一张“训练时 vs 推理时”的对照图。
- 图像生成应用和伦理/法规章节可增加“使用前检查清单”：版权、肖像、敏感内容、模型来源、生成标识。

### ch13 开源大模型部署与微调

优点：章节已经非常强调工程证据包，包括计算路线、环境报告、模型/运行时决策、固定评估集、API 契约、停止和回滚步骤。这很适合作为部署类章节模板。

建议：

- 在章节入口继续强化“不要从模型排行榜开始”，先看许可证、硬件、隐私、运行时、评估集和回滚边界。
- 计算路线、模型运行时、发布 runbook 已经较完整，后续可统一增加“失败边界卡”：显存不足、许可证不匹配、模型能跑但不能服务、评估不过、GPU 未关机。
- 微调相关内容如果继续扩展，应保持“先 Prompt/RAG/量化/运行时，再考虑 LoRA，全参训练最后”的判断顺序。

## 可复制的优秀模板

以下章节当前做法值得复用到较弱章节：

- `ch08-rag/ch01-rag/01-rag-basics.md`：先用闭卷/开卷类比，再讲 RAG 流程和故障定位。
- `ch08-rag/ch03-app-dev/02-langchain-basics.md`：先解释链式抽象为什么自然出现，再讲最小链。
- `ch09-agent/ch01-agent-basics/01-what-is-agent.md`：先区分工作流、聊天机器人、Agent，避免概念泛化。
- `ch09-agent/ch05-mcp/01-mcp-overview.md`：从“工具接入很乱”讲到协议价值，背景入口清楚。
- `ch12-multimodal/ch02-image-gen/01-diffusion-models.md`：先讲生成为什么难，再讲加噪/去噪主线。
- `ch13-open-source-llm/index.md`：把开源模型使用组织成计算路线、运行时、评估、发布和回滚证据包，而不是模型名字清单。

## 建议落地顺序

1. 先改写课程写作规范：每章首次引入核心技术时必须有背景入口。
2. 优先修 ch06 已经在陪学中暴露的卡点：RNN 顺序实验、LSTM、位置编码、VAE。
3. 补 ch07 Hugging Face 背景入口，并同步到后续所有平台/框架章节。
4. 处理 ch07/ch11 重叠定位，避免读者重复学习但不知道区别。
5. 补 CV 图像处理现实场景和视觉输出检查清单。
6. 精简 ch12 多模态基础里重复的 CLIP 背景段落。
7. 纳入 ch13 开源大模型部署，继续补齐模型选择、运行时、服务化、评估和回滚的失败边界。
