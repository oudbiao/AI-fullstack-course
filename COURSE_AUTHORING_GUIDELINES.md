# 课程写作优化规范

这份文档用于指导后续课程正文优化。它不是学习进度记录，而是给作者和陪学过程使用的写作标准。

## 1. 新技术先给背景入口

每章首次引入核心技术、模型、框架、平台或工程模式时，先回答四个问题，再进入术语、公式和代码。

```text
现实中会遇到什么问题？
不用这个技术时，原方法哪里不够？
这个技术新增了什么能力或取舍？
什么场景下应该想到它，什么场景下不一定需要它？
```

背景入口要短，但必须让学习者知道“为什么会有这个东西”。例如 VAE 不是先讲 `mu` / `logvar`，而是先讲普通 autoencoder 能重建，却不一定能从 latent space 随机采样生成有意义的新样本。

## 2. 实验要说明证明边界

每个实验都要写清楚两件事：

```text
这个实验证明了什么？
这个实验没有证明什么？
```

如果实验只是跑通训练闭环，就不要把它说成已经证明模型优势。比如 RNN 的“整体偏正 / 偏负”分类可以展示 many-to-one 流程，但不能充分证明顺序建模能力，因为求和或均值也能解决。

## 3. 代码讲解按功能块组织

代码讲解不要机械逐行翻译，优先按功能块、数据流和 shape 变化解释。

```text
这块代码负责什么？
它吃进去什么输入？
它吐出来什么输出？
它连接前后哪两步？
关键行为什么重要？
```

PyTorch 章节尤其要讲清：

```text
输入 shape
模型每一层输出 shape
loss 比较的对象
backward 计算的梯度
optimizer.step 更新的参数
eval / no_grad 的作用
```

## 4. 工具和生态先讲它封装了什么

遇到 Hugging Face、LangChain、OpenAI SDK、Docker、Kubernetes、向量数据库等工具时，不能只讲 API。

首次出现时要说明：

```text
它解决什么重复工作？
它把哪些底层概念标准化了？
高层封装适合什么时候用？
出问题时要回到哪个底层对象排查？
```

例如 Hugging Face 可以先解释成“模型和数据集共享平台 + transformers 工具库”。`from_pretrained` 不是魔法，它是在读取模型权重、tokenizer、config，并组装成可调用对象。

## 5. 抽象机制先用直觉图景

gate、attention、position encoding、optimizer、RAG、Agent、diffusion 等机制，不要先堆术语。先给直觉，再映射变量。

推荐顺序：

```text
普通方法怎么做
问题出在哪里
新方法多了什么控制点
数据或状态怎样流动
最后对应到代码变量和公式
```

例如 LSTM 先讲“长期笔记：擦掉旧内容、写入新内容、读出当前摘要”，再对应 forget gate、input gate、output gate、cell state 和 hidden state。

## 6. 输出要有观察方法

实验输出不能只贴结果，还要告诉学习者看什么。

```text
指标变好代表什么？
哪些输出是正常现象？
哪些输出说明实验失败？
失败后优先检查什么？
```

视觉、多模态和生成章节尤其要加入“人工检查清单”，例如边缘图是否只保留轮廓、二值图是否把前景背景分开、生成图是否存在文字错乱或身份漂移。
