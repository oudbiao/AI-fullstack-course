## 待评估优化点

### 0. 全课程需要统一增加“技术存在背景”入口

位置：全课程，尤其是首次引入新技术、新模型、新架构、新工程模式的章节。

问题：部分章节较快进入概念、公式、API 或实验，学习者可能知道“怎么写”，但不清楚“为什么这个技术会存在”“它解决什么现实问题”“遇到什么场景时应该想到它”。

为什么影响学习：如果缺少问题背景，技术会显得像孤立知识点。学习者容易机械记 API 或公式，却难以建立工程判断：什么时候用、为什么用、和旧方法相比解决了什么、不适合什么场景。

可能优化方向：为每个重要技术点增加统一的背景入口，放在术语和实验之前。建议使用四问模板：现实中会遇到什么问题？不用这个技术时原方法哪里不够？这个技术新增了什么能力或取舍？什么场景下应该想到它，什么场景下不一定需要它？尤其关注 CNN、RNN、LSTM/GRU、Attention、Transformer、GAN、VAE、RAG、Agent、微调、模型压缩、部署等章节。

当前处理：已将“背景入口模板”加入 `LEARNING_EXPLANATION_METHOD.md`，并新增 `COURSE_AUTHORING_GUIDELINES.md` 作为课程正文优化时的项目级写作规范。后续章节优化按该规范执行。

### 1. RNN 基础实验 3 不够体现顺序的重要性

位置：`src/content/docs/zh-cn/ch06-deep-learning/ch04-rnn/01-rnn-basics.md`，实验 3：训练一个小型序列分类器

问题：当前实验用“整体偏正 / 整体偏负”的短数值序列做分类。这个任务可以用均值、求和或普通线性模型解决，不一定需要 RNN。

为什么影响学习：本节主题是 RNN 和顺序建模，但这个实验更适合展示 RNN 训练闭环、many-to-one 分类和 shape 流程，不足以证明“顺序本身是信息”。

可能优化方向：保留原实验用于讲训练闭环，同时增加或替换一个对照实验：使用相同数字集合，但顺序不同、标签不同的序列。例如“先负后正 -> 1，先正后负 -> 0”。这样能更直接展示顺序对分类结果的影响。

候选最小实验：使用 4 条长度为 4 的序列，所有序列均值都为 0，但标签取决于“第一个值和最后一个值是否相同”。例如 `[1, -1, -1, 1] -> 1`，`[1, -1, 1, -1] -> 0`。这个任务更能说明 RNN 通过时间步更新 hidden state 读取顺序信息，而不是只看整体正负。

当前处理：已在课程正文补充原实验的证明边界，并加入“均值相同但顺序不同”的最小对照样本，用来强调顺序信息。

### 2. LSTM 概念讲解可以先用“长期笔记”直觉铺垫

位置：`src/content/docs/zh-cn/ch06-deep-learning/ch04-rnn/02-lstm-gru.md`，LSTM 概念与 gate 讲解部分

问题：直接介绍 `cell state`、`hidden state`、forget/input/output gate 对学习者来说仍然偏抽象，容易感觉像在背术语。

为什么影响学习：LSTM 的核心不是先记住门的名字，而是理解它比普通 RNN 多了“长期记忆通道”和“选择性擦、写、读”的机制。如果先给术语，学习者不容易把 gate 和实际动作对应起来。

可能优化方向：在正式介绍 gate 名称前，先加入一个生活化直觉：LSTM 像一本长期笔记。每个时间步先擦掉旧笔记中不重要的部分，再写入当前输入中重要的部分，最后读出一部分作为当前输出。然后再映射：长期笔记 = `cell state`，擦 = forget gate，写 = input gate，读 = output gate，当前摘要 = `hidden state`。

当前处理：已在课程正文 LSTM 入口补入“长期笔记 / 擦 / 写 / 读”的直觉映射，再对应 cell state、hidden state 和三个 gate。

### 3. Sinusoidal 位置编码需要更直观的铺垫

位置：`src/content/docs/zh-cn/ch06-deep-learning/ch05-transformer/02-transformer-architecture.md`，位置信息与 positional encoding 实验部分

问题：直接给出 `positions`、`dims`、`angle_rates`、sin/cos 填充矩阵时，学习者很难理解为什么要这样设计，容易只是在机械抄公式。

为什么影响学习：位置编码是 Transformer 理解顺序的关键桥，但公式形式抽象。如果学习者不能先理解“稳定、连续、多尺度的位置身份证”，后续 RoPE、相对位置、长上下文都会更难接上。

可能优化方向：在公式前加入更直观的解释：用“多根钟表指针”或“多尺度日期表示”类比位置编码。快频率维度负责区分近距离，慢频率维度负责覆盖长距离；sin/cos 成对表示同一个频率下的周期状态。再用一个小表对比 one-hot、直接编号和 sin/cos 多频率编码的优缺点。

当前处理：已在课程正文 positional encoding 公式前补入“多尺度位置身份证 / 多根钟表指针”的解释。

### 4. VAE 需要先讲清“为什么需要它”

位置：`src/content/docs/zh-cn/ch06-deep-learning/ch06-generative/02-vae.md`，VAE 概念入口

问题：当前内容较快进入 encoder、`mu`、`logvar`、reparameterization 和 KL，学习者容易先看到机制细节，却不清楚 VAE 的目的和实际作用。

为什么影响学习：如果没有先建立“普通 autoencoder 能重建但不一定能生成”的问题意识，VAE 的分布、采样和 KL 会显得像额外公式，而不是为了解决 latent space 可采样、连续、可插值的问题。

可能优化方向：在正式介绍 VAE 前增加一个现实背景入口：先说明遇到“想生成相似但不重复的新样本、做平滑变化、检测异常、学习数据隐藏分布”这类问题时，可以想到 VAE。例如头像/字体/服装款式生成、从不笑到微笑的平滑插值、用正常设备传感器数据检测异常。然后再引入对比：普通 autoencoder 像把每个样本压成孤立坐标，重建可以，但随机抽一个坐标不一定有意义；VAE 像把样本压成一片有范围的区域，并约束这些区域靠近标准正态分布，所以可以从 latent space 随机采样、插值和生成新样本。

当前处理：已在课程正文 VAE 入口补入“什么时候会想到 VAE”，强调生成、插值、异常检测和 latent space 可采样地图。

### 5. 平台/生态工具章节也需要先补背景

位置：`src/content/docs/zh-cn/ch07-llm-principles/ch01-nlp-crash/04-huggingface-quickstart.md`，以及后续首次引入 LangChain、OpenAI SDK、Docker、Kubernetes、向量数据库等平台或工具生态的章节。

问题：课程进入 Hugging Face 等工具时，如果直接讲 `AutoTokenizer`、`AutoModel`、`pipeline`、`Trainer` 等 API，学习者可能不知道 Hugging Face 本身是什么、为什么存在、解决了什么工程问题。

为什么影响学习：平台和生态类工具承载的是复用、分发、标准化和工程协作。如果缺少背景，学习者容易把它们当成一堆神秘 API，而不是理解为“模型、tokenizer、config、权重、数据集和训练工具的共享生态”。

可能优化方向：首次介绍平台/工具库时，先回答：它解决什么工程痛点？不用它时开发者要自己做哪些重复工作？它封装了哪些底层概念？什么时候用高层封装，什么时候要回到底层张量/模型接口调试？例如 Hugging Face 可先解释为“机器学习模型和数据集的共享平台 + transformers 工具库”，再进入 `AutoTokenizer` 和 `AutoModel`。

当前处理：已将“平台、生态或工具库也要讲背景”的规则加入 `LEARNING_EXPLANATION_METHOD.md`，并在 Hugging Face 快速上手页补入平台与 Transformers 工具库的背景入口。
