# 第九阶段：AI Agent 与智能体

| 信息 | 说明 |
|---|---|
| **预估学时** | 150～200 小时 |
| **前置要求** | 完成第八A+8B阶段 |

## 阶段概述

掌握 Agent 架构、推理、工具使用、记忆与多 Agent 系统

## 阶段导读

这一阶段最容易学乱，因为名词非常多。  
更稳的顺序是沿着 Agent 的结构来学：

1. 先知道什么是 Agent
2. 再补推理与工具
3. 再补记忆、协议、框架、多 Agent
4. 最后再看评估、安全、部署和项目

## 第八 B 阶段是怎么自然流到这里的

如果你刚学完第八 B 阶段，最值得先看清的一件事是：

- 第八 B 已经教会你知识怎么接进系统、模型怎么被调用、应用怎么工程化
- 第九阶段开始回答：如果系统不只是“按固定流程调用模型”，而是要围绕目标动态决定下一步，那它会长成什么样

更直白地说：

- 第八 B 更像“大模型应用系统”
- 第九阶段开始进入“智能体系统”

所以这里不是突然换题，而是在继续回答：

> **当系统开始具备目标、状态、工具和多步动作时，它和普通应用到底差在哪里。**

## 这一阶段的教学安排是否由浅入深？

整体上是顺的，而且对第一次系统学 Agent 的人来说，这种“先结构、再能力、再系统化”的路线是合理的。

更适合新人的理解主线是：

```mermaid
flowchart LR
    A["Agent 基础"] --> B["推理与工具"]
    B --> C["记忆"]
    C --> D["协议与框架"]
    D --> E["多 Agent"]
    E --> F["评估与安全"]
    F --> G["部署与项目"]
```

也就是说：

- **前四章在回答 Agent 到底是什么、靠什么运转**
- **第五到第七章在回答 Agent 系统怎样扩展成更复杂网络**
- **第八到第十章在回答怎样让它可评估、可上线、可交付**

## 这一阶段最适合新人的理解方式

可以先把第九阶段理解成一条不断加层的系统主线：

1. 先回答“Agent 是什么”
2. 再回答“它怎么想”
3. 再回答“它怎么做”
4. 再回答“它怎么记”
5. 最后才回答“它怎么协作、怎么评估、怎么上线”

这样读，会比直接按工具、框架和热点词去记更稳。

## 这一阶段真正新增了什么能力

从第八 B 走到这里，真正新增的不是“再多几个框架”，而是：

1. 开始从固定工作流走向动态决策
2. 开始把推理、工具、状态和记忆看成一个整体系统
3. 开始认真面对多步执行、失败恢复和安全边界

### 建议学习顺序

1. 第一章：Agent 基础
2. 第二章：推理
3. 第三章：工具
4. 第四章：记忆
5. 第五章：MCP
6. 第六章：框架
7. 第七章：多 Agent
8. 第八章：评估与安全
9. 第九章：生产部署
10. 第十章：项目实践

## 更适合新人的学习节奏

如果你是第一次系统学 Agent，更稳的节奏通常是：

1. 先把第一章学完  
   先搞清 Agent 和普通聊天系统到底差在哪。

2. 再学第二、三章  
   先把推理、工具和 Function Calling 串起来。

3. 再学第四章  
   这时再去理解短期记忆、长期记忆和状态管理，会更自然。

4. 再看第五、第六章  
   先知道 MCP 和各种框架在系统里各自扮演什么角色。

5. 然后学第七章  
   再进入多 Agent，才不会一上来就把复杂度拉满。

6. 最后学第八到第十章  
   把评估、安全、部署和项目真正收口。

## 如果你是第一次系统学 Agent，最该先守住什么

最值得先守住的是这三个判断：

- 先把单 Agent 做稳，再谈多 Agent
- 先把成功路径和失败路径都看见，再谈自动化
- 先把评估和日志补上，再谈“看起来很聪明”

## 如果你是第一次系统学 Agent，最稳的起步方式

建议先不要急着追：

- 多 Agent
- 最复杂框架
- 自动自循环系统

更稳的顺序通常是：

1. 先把 `Agent 基础 + 推理 + 工具` 看顺  
   先知道 Agent 到底靠什么运转。

2. 再看记忆  
   先把“状态怎么存、怎么回忆、怎么接续任务”看清楚。

3. 再看 MCP、框架和多 Agent  
   这时你更容易理解这些东西为什么会自然长出来。

4. 最后再看评估、安全和部署  
   这时才会意识到为什么 Agent 不是“跑起来就算完成”。

## 本阶段章节地图

| 章节 | 主题 | 主要解决什么问题 |
|---|---|---|
| 第一章 | Agent 基础 | 搞清 Agent 的边界、能力层级与系统结构 |
| 第二章 | 推理 | 学 ReAct、Plan-and-Execute 与规划方法 |
| 第三章 | 工具 | 学工具描述、调度、安全和代码 Agent |
| 第四章 | 记忆 | 学短期、长期、情景与程序性记忆 |
| 第五章 | MCP | 学协议化工具接入与能力网络 |
| 第六章 | 框架 | 学主流 Agent 框架的抽象和选型 |
| 第七章 | 多 Agent | 学协作、通信、协调和失败模式 |
| 第八章 | 评估与安全 | 学评测、护栏、观测和风险控制 |
| 第九章 | 生产部署 | 学运行时、恢复、成本与生产化实践 |
| 第十章 | 项目实践 | 把研究、分析和开发团队型 Agent 做成系统 |

### 学这一阶段最该带走什么

- 能说清 Agent 和普通聊天系统的差别
- 能判断什么时候要工具、什么时候要记忆、什么时候要多 Agent
- 能看懂一个 Agent 系统为什么能上线或为什么会失控

## 学这一阶段最容易犯的错

- 工具一多就想上多 Agent，而不是先把单 Agent 做稳
- 只演示成功路径，不设计失败恢复
- 没有评估、日志和权限控制就直接拼系统

## 这一阶段最该优先建立的系统习惯

1. 每加一个能力，都先问系统复杂度是不是明显上升了
2. 每多一个工具，都先问失败时怎么恢复
3. 每做一个 Agent，都先设计观察点，而不是只看最后答得像不像

## 学这一阶段最该养成的 3 个习惯

1. 每加一层能力，都先问它到底是在补推理、补工具、补记忆，还是补系统编排
2. 每做一个 Agent，都先设计失败路径，而不只设计成功路径
3. 每遇到“效果不好”，先分清问题出在规划、工具、状态还是运行时

## 这一阶段最值得优先补强的能力

- 能把“推理、工具、记忆、状态、评估”讲成一条系统链
- 能判断什么时候需要单 Agent、什么时候才值得多 Agent
- 能从项目目标反推：问题出在规划、工具、状态还是运行时

### 学完后的出口能力

- 能设计一个最小可用 Agent 系统
- 能为 Agent 增加评估、安全和运行时约束
- 能讲清楚 Agent 项目的模块边界和工程权衡

## 学完第九阶段后，你应该能自己回答什么

- 什么样的系统才值得叫 Agent，而不是普通工作流
- 一个问题到底更需要推理、工具、记忆，还是多 Agent
- Agent 出错时，我该优先查规划、工具、状态还是运行时
- 一个 Agent Demo 距离“可交付系统”还差哪几层
