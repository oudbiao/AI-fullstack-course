# Teaching and Self-Study Guide

![Learning pace cards](/img/course/intro-learning-pace-cards-en.png)

Use the same course content differently depending on the setting. The order stays mostly the same; the pace, homework, and review method change.

## 1. Choose the Mode

| Mode | Weekly focus | Evidence |
|---|---|---|
| Self-study | Run one small result and write one review | README, command, screenshot, failure note |
| Instructor-led | Teach the frame, demo the loop, review homework | slides or notes, demo code, rubric |
| Study group | Share deliverables, not just reading progress | peer-reviewed README and sample output |

## 2. Recommended Rhythm

| Step | Self-learner | Instructor | Study group |
|---|---|---|---|
| Preview | Read the map and goal | Pick the misconception to explain | Assign roles |
| Practice | Run the smallest code | Demo once, then let learners change it | One member demos |
| Project | Keep a tiny artifact | Set basic and challenge tasks | Review artifacts |
| Review | Record one failure | Discuss common failures | Collect blockers |

## 3. Homework Rule

Good homework should be runnable, checkable, and reviewable. Avoid only saying “read chapter X.” Better tasks:

| Stage | Homework shape |
|---|---|
| Python | CLI script with sample input/output |
| Data | Notebook or report with chart conclusion |
| ML/DL | baseline or training run with metric and failure |
| RAG | Q&A demo with retrieval logs and citations |
| Agent | tool-calling task with trace and stop condition |

## 4. Acceptance Rule

![Portfolio acceptance ladder](/img/course/intro-portfolio-acceptance-ladder-en.png)

Every project review should ask four questions: can it run, can others reproduce it, can the result be evaluated, and can the learner explain a failure and next step?

Keep the course alive by updating maps, projects, evaluation standards, and common mistakes first. Do not chase every new model name unless it changes the teaching path.
