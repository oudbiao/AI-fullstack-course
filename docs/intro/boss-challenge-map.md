# Stage Checkpoint Cards

![Stage checkpoint cards](/img/course/intro-stage-checkpoint-cards-en.png)

This page replaces the old “Boss battle” idea with a simpler rule:

> Every stage should leave one runnable artifact, one piece of evidence, and one short reflection.

## The Checkpoint Pattern

| Part | Meaning | Example |
| --- | --- | --- |
| Artifact | What you built | script, notebook, API, RAG demo, Agent trace |
| Evidence | How others know it worked | screenshot, CSV, log, metric table, citation check |
| Reflection | What you learned or fixed | limitation, failure case, next step |

## Stage Cards

| Stage | Minimum artifact | Evidence |
| --- | --- | --- |
| 1 Tools | A project folder that runs | README and Git commit |
| 2 Python | A small CLI or API | Sample input/output and error handling |
| 3 Data | A cleaned dataset report | Charts, quality notes, limitations |
| 4 Math | A runnable mini experiment | Formula-to-code explanation |
| 5 Machine learning | A baseline model | Metrics and error samples |
| 6 Deep learning | A training run | Loss curve and checkpoint |
| 7 LLM/Prompt | A stable prompt test | Fixed inputs and output checks |
| 8 RAG | A cited Q&A demo | Retrieval logs and citation table |
| 9 Agent | A controlled task run | Tool trace and stop condition |
| 10-12 Extensions | One direction project | Before/after examples and evaluation |
| Graduation | A complete AI product | Demo, evaluation report, deployment notes |

## Keep the Reflection Short

Use this template after each stage:

```md
## Stage Checkpoint

Artifact:
Evidence:
One failure:
Fix or next step:
```

If the card is complete, move on. If it is empty, read less and build the minimum artifact first.
