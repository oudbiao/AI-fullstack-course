## What You Are Building Toward

<div class="course-evidence-panel">
  <p>By the end of the main line, your strongest project should explain one AI application end to end. The goal is not to memorize buzzwords. The goal is to prove that you can turn model behavior into a usable product workflow.</p>
  <ul>
    <li><strong>Setup and README commands:</strong> another person can run the work.</li>
    <li><strong>Data, prompt, retrieval, and tool traces:</strong> the application has observable behavior.</li>
    <li><strong>Metrics, comparison notes, and failure cases:</strong> you can judge quality instead of trusting one demo.</li>
    <li><strong>Safety, privacy, cost, and latency notes:</strong> you understand product constraints.</li>
    <li><strong>Screenshots or short demos:</strong> a reviewer can understand the user experience quickly.</li>
  </ul>
</div>

## One Rule

Read briefly, run something, keep evidence. At the end of each stage, you should have something another person can inspect: a README command, a saved output, a metric table, a trace, a failure note, or a small demo. This is also how you turn learning into a portfolio story.

```text
target_role: AI application engineer, AI full-stack builder, or AI-enabled product engineer
project_thread: one assistant, automation, analysis, or multimodal idea to improve across chapters
first_route: quick experience -> minimum setup -> capability map -> Chapter 1
first_evidence: screenshot, saved output, README command, or short observation note
expected_output: one main-route plan and one concrete portfolio-grade artifact to start
```
