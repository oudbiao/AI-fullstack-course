## 最終的に作るもの

<div class="course-evidence-panel">
  <p>メインラインを終えるころには、最も強いプロジェクトで1つの AI アプリケーションを端から端まで説明できる状態を目指します。目標は流行語の暗記ではありません。モデルの振る舞いを使えるプロダクトワークフローに変えられることを示すことです。</p>
  <ul>
    <li><strong>セットアップと README コマンド:</strong> 他の人が作業を再実行できます。</li>
    <li><strong>データ、Prompt、検索、ツールのトレース:</strong> アプリの挙動を観察できます。</li>
    <li><strong>指標、比較メモ、失敗例:</strong> 1回のデモではなく品質を判断できます。</li>
    <li><strong>安全性、プライバシー、コスト、遅延メモ:</strong> プロダクト上の制約を理解しています。</li>
    <li><strong>スクリーンショットまたは短いデモ:</strong> レビューする人がユーザー体験をすばやく理解できます。</li>
  </ul>
</div>

## 1つのルール

短く読み、動かし、証拠を残す。各ステージの終わりには、他の人が確認できるものを残します。README コマンド、保存した出力、指標表、トレース、失敗メモ、小さなデモなどです。これが学習をポートフォリオの物語に変える方法でもあります。

```text
目標職種: AIアプリケーションエンジニア、AIフルスタックビルダー、またはAIプロダクトエンジニア
プロジェクト軸: 章をまたいで改善する assistant、automation、analysis、または multimodal の案
最初のメインルート: クイック体験 -> 最小環境 -> 能力マップ -> 第1章
最初の証拠: スクリーンショット、保存した出力、README コマンド、または短い観察メモ
期待される成果: メインルート計画と、ポートフォリオの起点になる具体的な成果物
```
