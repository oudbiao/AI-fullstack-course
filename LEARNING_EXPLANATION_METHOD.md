# 学习讲解方式草稿

这份文档记录之后陪学时使用的讲解方式。它不是课程进度表，也不记录学到了哪一节；它只记录“怎样讲更容易学会”。后续讲解过程中，如果发现某种表达更清楚，就更新这里，直到形成稳定定稿。

## 当前状态

状态：草稿

目标：让代码讲解从“逐行翻译”升级为“按功能块、数据流和关键机制理解”。

## 核心原则

不要机械逐行讲。逐行解释容易把整体逻辑切碎，尤其是深度学习、数据流、模型训练这类代码。

讲任何新技术前，先补“存在理由”和使用背景。学习者需要先知道它为什么被发明、解决什么现实痛点、遇到什么问题时会想到它，再进入术语、公式和代码。背景不是额外故事，而是理解机制的入口。

优先按下面顺序讲：

1. 先说这个技术为什么存在，解决什么痛点。
2. 再说这段代码整体要完成什么。
3. 再把代码分成几个功能块。
4. 每个功能块讲清目的、输入、输出和 shape 变化。
5. 对关键语句局部放大解释。
6. 最后把执行流程串成一条完整故事线。

## 推荐讲解结构

讲一节新内容时，默认先讲概念，再做实验，再讲代码，最后总结。不要一开始就进入代码讲解。

```text
1. 背景入口：现实中遇到什么问题时会想到这个技术。
2. 先讲简单概念：这一节解决什么问题，为什么需要这个技术。
3. 再讲理解框架：核心机制、直觉、和上一节的关系。
4. 学习者自己写实验代码：先动手，不急着逐段解释。
5. 写完后讲实验代码：按功能块、数据流、shape 和关键行放大讲。
6. 最后总结：本节最该记住的点、实验证明了什么、还有哪些边界。
```

讲实验代码时，默认使用这个结构：

```text
1. 这段代码解决什么问题
2. 整体执行流程
3. 功能块拆解
4. 数据流 / shape 变化
5. 关键代码放大
6. 常见误区
7. 你现在最该记住的 3 个点
```

如果代码很短，可以合并部分环节；如果代码很复杂，可以先画出数据流，再逐块进入。

## 背景入口模板

讲一个新技术时，先用下面四个问题建立需求感：

```text
现实中会遇到什么问题？
不用这个技术时，原来的方法哪里不够？
这个技术新增了什么能力或取舍？
什么场景下应该想到它，什么场景下不一定需要它？
```

例子：

```text
VAE：
现实问题：想生成相似但不重复的新样本、做平滑插值、检测异常。
原方法不足：普通 autoencoder 会重建，但 latent space 不一定适合随机采样。
新增能力：把样本编码成分布，并约束 latent space 更规整。
适用场景：生成、插值、异常检测、表示学习；如果只做分类，不一定需要 VAE。
```

以后讲 CNN、RNN、Transformer、RAG、Agent、微调、压缩、部署等内容时，也先回答“技术为什么存在”，再进入机制。

平台、生态或工具库也要讲背景。遇到 Hugging Face、LangChain、OpenAI SDK、PyTorch Lightning、Docker、Kubernetes 这类名字时，不能只说 API 怎么用；先说明它在工程里解决了什么协作或复用问题，它和底层概念的关系是什么，以及什么时候需要直接用底层接口而不是高层封装。

## 功能块讲解法

讲一个功能块时，不要马上逐行翻译，先回答四个问题：

```text
这块代码负责什么？
它吃进去什么输入？
它吐出来什么输出？
它在整个程序里连接前后哪两步？
```

然后只挑关键行解释。普通赋值、导入、格式化打印等代码，除非学习者卡住，否则简短带过。

## 实验例子的边界说明

讲实验时，先判断这个例子真正证明了什么、没有证明什么。能跑通训练闭环，不等于它已经很好地展示了对应模型的核心优势。

例如 RNN 示例如果只是判断序列整体偏正或偏负，其实均值、求和或普通 MLP 也能解决，不足以体现“顺序”的重要性。讲解时应明确：

```text
这个例子证明了：RNN 训练闭环、输入输出 shape、many-to-one 分类流程。
这个例子没充分证明：模型必须利用时间顺序。
更好的对照实验：使用相同数字集合，但顺序不同、标签不同的序列。
```

以后遇到类似情况，优先补一个对照样本或替代小实验，让模型能力和任务设计真正对应。

## 深度学习代码讲解模板

遇到 PyTorch / 深度学习代码时，优先讲 shape 和训练闭环。

```text
数据 shape 是什么？
模型每一层怎样改变 shape？
loss 比较的是哪两个东西？
backward 计算了什么？
optimizer.step 真正更新了什么？
预测时为什么要 no_grad？
```

例如 RNN 序列分类器应先讲：

```text
X:                [batch, seq_len, input_size]
RNN out:          [batch, seq_len, hidden_size]
最后时间步:       out[:, -1, :]
Linear 后 logits: [batch, num_classes]
```

再解释为什么 many-to-one 任务通常取最后时间步或最终 hidden state。

## 关键行放大方式

关键行不是孤立解释，而是放回数据流中解释。

坏方式：

```text
这一行调用 rnn，下一行 return。
```

好方式：

```text
这一行把整段序列交给 RNN。RNN 会在每个时间步更新 hidden state，所以输出 out 里保留了每一步的状态。后面 out[:, -1, :] 取的是最后一步，也就是模型读完整段序列后的总结。
```

## 讲解时的语气和节奏

先建立整体感，再补细节。学习者不需要一次吃下所有术语。

优先使用：

- “这块代码的角色是……”
- “这里的 shape 从……变成……”
- “这行真正关键的是……”
- “你可以把它想成……”
- “现在先记住这一点，后面再展开……”

避免：

- 长时间逐行翻译。
- 用一堆术语解释另一个术语。
- 在学习者还没建立流程前展开太多底层细节。
- 把代码讲成 API 文档。

## 图解优先规则

遇到 gate、attention、backprop、optimizer、RAG 检索链路、Agent 执行循环等抽象机制时，优先用竖向流程图讲，再补公式或代码。

图解顺序建议：

```text
普通版本怎么做
问题出在哪里
改进版本多了什么控制点
数据或状态怎样一步步流动
最后再对应到代码变量
```

如果机制涉及“旧状态 -> 新状态”的变化，优先画竖图或上下流程图；如果涉及两个模型对比，再画左右对照图。

难概念不要先报术语名。先给一个具体任务场景，再用生活动作解释机制，最后再映射到术语。例如讲 LSTM 时，先讲“长期笔记怎样擦掉、写入、读出”，再对应 forget gate、input gate、output gate、cell state 和 hidden state。

## 每次讲解后的更新规则

讲解过程中如果发现新的更好方式，就更新本文档。

可以更新：

- 更清楚的讲解顺序。
- 某类代码的专门模板。
- 学习者容易卡住的点。
- 更好的类比或 shape 表达。
- 应该避免的讲法。

不要更新：

- 当前学到哪一节。
- 某次实验是否完成。
- 临时问题排查日志。
- 课程内容本身的总结。

这些属于学习进度或项目证据，不属于讲解方式。

## 暂定定稿标准

当这种讲解方式能稳定适用于以下内容时，可以考虑定稿：

- Python 基础代码
- 数据分析代码
- sklearn 训练代码
- PyTorch 训练循环
- CNN / RNN / Transformer shape 讲解
- LLM / RAG / Agent 应用代码

定稿前，继续保持“草稿”状态，并根据实际讲解体验微调。
