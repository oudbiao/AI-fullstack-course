# 作品集验收清单

![作品集验收阶梯](/img/course/intro-portfolio-acceptance-ladder.png)

作品集项目不需要堆很多功能，但必须有证据：能运行、别人能复现、输出能评估、你能讲清楚取舍。

## 1. 验收阶梯

| 等级 | 最低证据 | 如果这样就还不够 |
|---|---|---|
| 练习级 | 代码能在本地运行，并打印结果 | 只有截图，没有代码或命令 |
| 项目级 | 有 README、运行命令、示例输入输出 | 别人按说明跑不起来 |
| 作品集级 | 有评估集、日志、失败样本、演示截图 | 只展示成功案例 |
| 面试级 | 能讲架构、指标、限制、成本、安全和替代方案 | 说不清为什么这样选 |

每个阶段项目至少做到“项目级”。毕业项目建议冲到“作品集级”。

## 2. 一分钟快检

分享项目前，先确认这 8 项：

| 项目 | 达标标准 |
|---|---|
| README | 目标、用户、功能列表清楚 |
| 运行命令 | 新读者能跑通最小流程 |
| 依赖记录 | 有 `requirements.txt`、`pyproject.toml` 或 `package.json` |
| 示例 | 有 1-3 组真实输入输出 |
| 截图或演示 | 能看出项目到底做了什么 |
| 评估 | 有固定问题集、指标或人工评分规则 |
| 失败样本 | 记录坏案例和原因 |
| 下一步 | 改进方向具体，而不是“继续优化” |

如果缺 3 项以上，先把它当练习代码继续打磨。

## 3. 不同 AI 项目的证据

| 项目类型 | 应保留的证据 |
|---|---|
| 机器学习 | Baseline、训练/测试划分、指标、错误样本 |
| 深度学习 | 训练日志、验证曲线、过拟合检查 |
| Prompt 助手 | Prompt 版本、输入输出、失败案例 |
| RAG | 文档来源、切分策略、检索日志、引用检查 |
| Agent | 工具 schema、执行 trace、停止条件、权限边界 |
| 多模态 | 输入格式、处理流程、人工复核、质量标准 |

RAG 项目的核心不是界面好看，而是检索和引用可靠。Agent 项目的核心不是步骤多，而是执行可追踪、动作可控制。

## 4. 三分钟演示顺序

展示项目时按这个顺序讲：

1. 用户是谁，有什么问题。
2. 最小可运行功能做了什么。
3. AI 在流程里的位置。
4. 你怎样验证输出。
5. 以前失败在哪里，你改了什么。
6. 下一步准备怎么改。

## 5. 最小 README 结构

```md
# 项目名

## 目标

## 如何运行

## 示例输入输出

## 评估

## 已知失败

## 下一步
```

短而可复现，比长而空泛更有价值。
