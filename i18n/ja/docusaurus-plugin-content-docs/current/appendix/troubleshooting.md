# 学習のつまずき救援

![学習のつまずき排障マップ](/img/course/appendix-troubleshooting-rescue-map.png)

![つまずきの最小再現と相談のフローチャート](/img/course/appendix-debug-mre-help-flow.png)

:::tip 図の見方
問題が起きたら、まず慌てて別のやり方に変えないでください。「問題を再現する -> 環境を集める -> 入力を小さくする -> エラーを記録する -> 具体的な質問をする」という順で進めると、多くのつまずきは感情的な問題から、原因を特定できる工学的な問題に変わります。
:::

このページの目的は理論を説明することではなく、「つまずいたときにできるだけ早く再び前に進める」ことを助けることです。学習が止まる原因の多くは、難しすぎるからではなく、小さな問題が長引きすぎるからです。

## 1. まず、どの種類のつまずきかを判断する

| 現象 | いちばんありそうな問題の種類 | 最初にやること |
|---|---|---|
| `ModuleNotFoundError` | 環境や依存関係が正しく入っていない | まず現在の Python 環境を確認する |
| コードは動くのに結果が変 | 入力、ラベル、または評価の理解が間違っている | まず中間結果を print する |
| 学習が遅い、またはメモリ不足で落ちる | batch が大きすぎる、モデルが重い、デバイスが合っていない | まず実験規模を小さくする |
| 1章読んでも問題が解けない | 概念がコードと結びついていない | 最小例に戻ってもう一度実行する |
| プロジェクトの始め方がわからない | テーマが大きすぎて分解できていない | まず最小の閉ループを描く |
| たくさん学んだのに覚えられない | 振り返りとアウトプットが足りない | ノートと短いまとめを作る |

## 2. 環境の問題をどう調べるか

環境の問題は人をいちばん混乱させますが、実は似たような問題が何度も出ます。

まず、次のコマンドを実行してください。

```bash
python --version
which python
pip --version
pip list
pwd
ls
```

GPU がある場合は、これも追加します。

```bash
nvidia-smi
```

### 2.1 いちばんよくある環境の問題

#### 問題1：インストールしたのに import できない

よくある原因は次のとおりです。

- 別の Python 環境にインストールしている
- ターミナルと IDE で同じ interpreter を使っていない
- `pip` が今の環境を指していない

より安全なやり方は次の通りです。

```bash
python -m pip install numpy
python -m pip install pandas
```

これで「違う環境に入れてしまう」可能性を減らせます。

#### 問題2：パスが見つからない

まず確認すること：

- 今いる作業ディレクトリは、想定した場所か
- 相対パスは現在のディレクトリを基準にしているか
- ファイル名の大文字・小文字を間違えていないか

一時的に次を print して確認できます。

```python
from pathlib import Path

print(Path.cwd())
print(Path("data").exists())
```

#### 問題3：バージョンの衝突

依存関係の問題が何度も起きるなら、まず次の2つをやるのが先です。

1. よく使う依存関係を統一した環境ファイルに書く
2. 大きなテーマごとに独立した仮想環境を使う

すべての実験をひとつの汚れた環境に詰め込まないでください。

## 3. コードは動くのに、正しいかどうかわからない

この種類の問題は、「エラーが出る」より危険です。

### 3.1 まず入力を確認する

少なくとも次を確認しましょう。

- データ件数は合っているか
- 入力 shape は正しいか
- ラベルの範囲は正しいか
- 空値、文字化け、重複サンプルはないか

例：

```python
print("サンプル数:", len(texts))
print("最初の2件のテキスト:", texts[:2])
print("最初の2件のラベル:", labels[:2])
print("ラベル集合:", sorted(set(labels)))
```

### 3.2 次に中間結果を確認する

最終的な精度だけを見ないでください。もっと大事なのは次の点です。

- モデルは実際に何を出力しているか
- 確率は妥当か
- 中間特徴がすべて 0、またはすべて同じになっていないか

### 3.3 最後に評価方法を確認する

よくあるミスは次の通りです。

- 学習データとテストデータが混ざっている
- クラス不均衡なのに accuracy だけ見ている
- テキストタスクで1件ずつしか見ず、全体の誤り分布を見ていない

## 4. 学習が収束しないときはどうするか

最初からモデル構造を疑いすぎないでください。多くの場合、問題はそこまで複雑ではありません。

次の順番で確認するとよいです。

1. まず極小データセットで過学習できるか試す
2. 次に learning rate が大きすぎないか、小さすぎないかを見る
3. 次に loss 関数とラベル形式が合っているかを見る
4. 最後にモデル構造やより複雑なテクニックを疑う

もしモデルがごく小さいデータすら学習できないなら、まず次を確認します。

- 入力を間違えていないか
- ラベルがずれていないか
- loss 関数を間違えていないか
- optimizer と learning rate が極端すぎないか

## 5. メモリ不足のときはどうするか

いちばん直接的な順番は次の通りです。

1. `batch size` を小さくする
2. 入力サイズを小さくする
3. 不要なログやキャッシュを止める
4. より小さいモデルを使う、または head だけを学習する
5. 必要なら gradient accumulation や mixed precision を使う

初心者は「もっと大きい GPU に替える」ことを考えがちですが、まず実験規模を小さくするほうが効果的なことが多いです。

## 6. プロジェクトをどう形にすればいいかわからないとき

「いろいろ少しずつわかるけれど、どうプロジェクトにすればいいかわからない」と感じるなら、たいていテーマが大きすぎます。

より安定した分け方は次の通りです。

### 6.1 まず一文で目標を書く

例えば次のようにします。

- 社内ルールの質問に答えられるナレッジベースのアシスタントを作る
- 画像のカテゴリを判定する小さなシステムを作る
- 記事の要約を生成するツールを作る

### 6.2 次に最小閉ループに分解する

最小閉ループは、だいたい次の4つだけで構成します。

1. 明確な入力
2. 実行できる処理フロー
3. 観察できる出力
4. 基本的な評価方法

### 6.3 最後に少しずつ機能を足す

よく追加するものは次の通りです。

- ログ
- エラー処理
- キャッシュ
- ユーザーインターフェース
- 評価用データセット
- デプロイ

## 7. 焦ってしまう、追いつけない気がするときは

まず覚えておいてほしいのは、AI 分野は更新が速いですが、基礎能力の更新はそんなに速くないということです。

本当に安定させたい能力は次のものです。

- コードを読む
- エラーを調べる
- 問題を分解する
- 小さな閉ループを作る
- 自分の案を説明する

この5つが少しずつできるようになっているなら、遅れているわけではありません。

## 8. 質問するとき、どうすれば助けてもらいやすいか

「エラーが出ました。見てください」だけではなく、次のように書くとよいです。

```text
何をしているか：
何を期待しているか：
実際に何が起きたか：
完全なエラーの最後の20行：
すでに試した方法：
最小再現コード：
```

こうすると、相手が助けやすくなりますし、自分でも先に問題を整理できます。

## 9. 最小再現テンプレート

大きなプロジェクトが複雑すぎると感じたら、まず問題を次のくらいの大きさに縮めます。

```python
def predict(x):
    return x * 2

data = [1, 2, 3]
preds = [predict(x) for x in data]
print(preds)
```

そのあとで、少しずつ本来のロジックを戻していきます。このやり方の意味は、どの層を追加したところで問題が出始めたのかを、すばやく特定できることです。

## 10. いつ休むべきか、いつ強く進めるべきか

休むのに向いている状況：

- 30分以上、ずっとむやみに試している
- 何をしているのか分からないまま、機械的にコマンドをコピペしている
- エラーメッセージが読めなくなっている

続けて粘るのに向いている状況：

- 問題が 1〜2 個の検証可能な仮説まで小さくなっている
- 次に何を試すか分かっている
- 変更のたびに、その目的を説明できる

本当に効率のよい学習は、ずっと押し続けることではなく、いつ止まって整理するかを知ることです。
