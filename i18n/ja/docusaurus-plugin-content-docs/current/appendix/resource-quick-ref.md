## よくあるタスクで最初に選ぶ baseline

| タスク | 最初におすすめする baseline | 理由 |
|---|---|---|
| 表形式データの分類 / 回帰 | 線形モデル、木系モデル | シンプルで速く、説明しやすい |
| テキスト分類 | `TF-IDF + LogisticRegression` | 強い baseline で、理解しやすい |
| 画像分類 | 転移学習 | 小規模データでは現実的 |
| 固有表現抽出 | ルール / 辞書ベースの baseline の後に系列モデル | まずタスク定義を確認するため |
| 文書 QA | キーワード検索や BM25 の後に RAG | まず検索が有効かを確認するため |
| 企業知識ベース | 単発 QA + 参照元の提示 | まず検証できる閉ループを作るため |
| Agent のツール呼び出し | 単一 Agent + 少数のツール | まず安定させてから複雑化するため |

多くのプロジェクトが進まない理由は、モデルが弱いからではなく、baseline が固まっていないからです。

## よく使う評価指標の早見表

### 分類タスク

| 指標 | いつ見るか | 注意点 |
|---|---|---|
| Accuracy | クラスの偏りが比較的小さいとき | 不均衡データでは誤解を招きやすい |
| Precision | 誤報のコストが高いとき | 「出した結果のうち、本当に正しい割合」を見る |
| Recall | 見逃しのコストが高いとき | 医療や安全系で特に重要 |
| F1 | Precision と Recall の両方を重視したいとき | よく使う総合指標 |

### 回帰タスク

| 指標 | いつ見るか | 注意点 |
|---|---|---|
| MAE | 平均絶対誤差を見たいとき | 外れ値に比較的強い |
| MSE / RMSE | 大きな誤差をより強く罰したいとき | 外れ値に敏感 |
| R² | 全体の当てはまりの説明度を見たいとき | 単独では使わない |

### 検索と QA

| 指標 | 役割 |
|---|---|
| Hit@K | 正しい文書が上位 K 件に入っているか |
| MRR | 正しい答えがどれくらい上位にあるか |
| 引用の正確率 | 回答が実際に召回した内容に基づいているか |
| 人手評価 | 最終的に使える品質かどうか |

## 学習の途中でよく見るシグナル

| 現象 | まず疑うべきもの |
|---|---|
| 学習 loss が下がらない | 学習率、ラベル、損失関数、入力形式 |
| 学習は良いのに検証が悪い | 過学習、データ漏洩、分布の不一致 |
| Accuracy がずっと変わらない | 特徴量が弱い、ラベルに問題がある、モデルが学べていない |
| GPU メモリが足りない | batch が大きすぎる、入力が長すぎる、モデルが大きすぎる |
| 結果が非常に不安定 | データが少ない、乱数の影響が大きい、実験で seed を固定していない |

## RAG を作るときに優先して確認すること

### 最小閉ループ

1. 文書を正しく分割できているか
2. 検索で正しい断片を召回できているか
3. 生成時に出典が付いているか
4. 回答が本当に召回内容を使っているか

### よくある落とし穴

| 問題 | 最初に確認すること |
|---|---|
| 回答がでたらめに見える | 検索が関連断片を召回できていない |
| 回答がぼんやりしている | chunk が大きすぎて情報が散らばっている |
| 権限が必要な内容をよく間違える | 権限フィルタがない |
| 速度が遅い | 検索層、モデル層、ネットワーク呼び出しを分けて測る |
| コストが高すぎる | キャッシュ、文脈短縮、不要な呼び出しの削減 |

## Agent を作るときに優先して確認すること

### まずはシンプルな構成から始める

より安定しやすい順番は、一般的に次の通りです。

1. 単発 QA
2. 単一 Agent + 単一ツール
3. 単一 Agent + 複数ツール
4. 状態を持つワークフロー
5. 複数 Agent の協働

### ツール層で確認すべき 5 つのこと

1. ツール説明が明確か
2. パラメータ schema が厳密か
3. エラーを返せるか
4. 高リスク操作に制限があるか
5. ログで追跡できるか

## Prompt 設計の早見表

より安定した Prompt には、少なくとも次の要素が入っています。

- 役割: モデルに何を演じてほしいか
- タスク: 具体的に何をしてほしいか
- 入力: どんな内容を受け取るか
- 出力: どんな形式で出すか
- 制約: 何をしてはいけないか

簡単なテンプレート：

```text
あなたは____です。
あなたのタスクは____です。
入力には____が含まれます。
次の形式で____を出力してください。
情報が不足している場合は、推測せずにその旨を明確に伝えてください。
```

## 最小の学習ループの骨組み

```python
for epoch in range(num_epochs):
    model.train()
    for x, y in train_loader:
        optimizer.zero_grad()
        pred = model(x)
        loss = criterion(pred, y)
        loss.backward()
        optimizer.step()
```

この 6 行で特に理解したいのは次の点です。

- データがどこから来るか
- 順伝播で何をしているか
- 損失をどう定義しているか
- 勾配をどう逆伝播しているか
- パラメータをいつ更新しているか

## プロジェクトが進まなくなったら、この図に戻る

| 今の問題 | 先に補うとよいもの |
|---|---|
| コードが読めない | Python とデバッグ |
| 結果が安定しない | データと評価 |
| どのモデルを選べばいいかわからない | baseline とタスク定義 |
| 完全なシステムにできない | 最小閉ループ、インターフェース、状態設計 |
| たくさん学んだのに覚えられない | 小さなプロジェクト、振り返り、アウトプット |
