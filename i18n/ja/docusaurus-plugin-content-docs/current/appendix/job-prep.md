# A.10 就職準備チェックリスト

![就職準備ファネル図](/img/course/appendix-job-prep-funnel-ja.png)

![AI 就職ポートフォリオのストーリーライン図](/img/course/appendix-job-portfolio-storyline-map-ja.png)

:::tip 図の読み方
就職用のプロジェクトは、技術用語をただ並べるだけではいけません。図を見るときは「職種目標 -> ユーザーの課題 -> 技術解決策 -> 指標結果 -> 失敗の振り返り -> README / 面接での説明」とつなげてストーリーとして捉えると、面接官に本当の実力が伝わりやすくなります。
:::

このページは「学び終わってから就職活動を始める」ためのものではありません。学習の過程を、応募できて、面接で話せて、内容をきちんと説明できる成果に少しずつ変えていくためのものです。

## まず、どの職種に応募したいかを決める

方向によって、準備する重点は違います。

| 職種方向 | より重視されること | 重点的に準備すること |
|---|---|---|
| AI / アルゴリズムエンジニア | モデル理解、学習、評価 | 機械学習、深層学習、実験プロジェクト |
| LLM アプリケーションエンジニア | プロダクトの一連の流れ、RAG、Agent、バックエンド開発 | プロジェクトの完成度、API設計、ログと評価 |
| データサイエンス / データ分析 | データ処理、統計、モデリング、可視化 | SQL、Python、分析レポート、ビジネス表現 |
| AI プロダクト / テクニカルプロダクト | シナリオ理解、要件分解、評価と協働 | プロジェクト案、プロダクト判断力、コミュニケーション力 |

初心者なら、すべての方向を同時に準備しようとしないでください。まずは主目標を1つ決めるほうが効果的です。

## 履歴書で一番重みがあるものは何か

多くの初心者にとって、最も価値があるのは「何コース見たか」ではなく、次の3つです。

1. 2～3個の、しっかり説明できるプロジェクト
2. 安定した技術スタックとツールチェーン
3. 明確に検証できる結果

プロジェクトは「XXX を学んだ、XXX を知っている」よりも説得力があります。なぜなら、プロジェクトでは次の力を同時に見せられるからです。

- 実践力
- トラブル対応力
- 取捨選択の力
- 説明力

## どんなプロジェクトを選べば就職に役立つか

大きな題材を作ることよりも、「境界がはっきりした題材」を作ることのほうが重要です。

より適したプロジェクトは、次の条件を満たすことが多いです。

- 入力と出力が明確
- baseline がある
- 評価方法がある
- 失敗ケースがある
- 自分なりの改善点がある

例えば、次のようなものです。

- 画像分類システム
- テキスト分類 / 感情分析プロジェクト
- 企業ナレッジベースQ&Aシステム
- スマートアシスタントや Agent のワークフロー
- データ分析 Agent

## 履歴書のプロジェクト欄はどう書くべきか

「某フレームワークを使って某プロジェクトを完成させた」だけではなく、もっと良い書き方があります。

### まず課題を書く

そのプロジェクトは何を解決するのか。誰向けか。入力と出力は何か。

### 次に自分の解決策を書く

どんなモジュールを作ったのか。なぜその設計にしたのか。

### 最後に結果を書く

結果として書けるのは、たとえば次のようなものです。

- 指標の改善
- 処理速度の向上
- 召回品質の改善
- ユーザー体験の向上
- デプロイや保守コストの削減

### 履歴書の表現例

あまり良くない書き方：

```text
Python と LangChain を使ってナレッジベースQ&Aシステムを作成した。
```

より良い書き方：

```text
Python を基盤に企業ナレッジベースQ&Aシステムを構築し、文書分割、ベクトル検索、権限フィルタリング、回答の引用元追跡を実装；
評価用データセットを設計して異なる chunk 戦略を比較し、誤召回を減らしつつ回答の追跡可能性を向上させた。
```

2つ目の書き方のほうが、本当に取り組んだことが伝わります。

## GitHub のプロジェクトページはどう準備するか

評価されやすいプロジェクトリポジトリには、少なくとも次の内容が必要です。

- `README`
- 実行方法
- プロジェクト構成の説明
- 入出力例
- スクリーンショットや効果図
- 既知の問題と今後の予定

他の人があなたのリポジトリを開いたとき、3分以内に次のことが分かるべきです。

- 何をするものか
- どうやって動かすか
- どこが重要な実装か
- だいたいどんな効果があるか

## 面接前に補っておくべき基礎

### 共通基礎

- Python の基礎
- データ構造と、よく使う計算量の感覚
- Git、環境管理、よく使うコマンドライン操作
- HTTP / API の基礎

### 機械学習 / 深層学習の基礎

- 代表的なタスクと指標
- 過学習と未学習
- 訓練データ、検証データ、テストデータ
- 最適化手法、学習率、損失関数

### LLM / Agent 方向の基礎

- Prompt、RAG、Fine-tuning の違い
- ベクトル検索と chunking の役割
- ツール呼び出しに schema と検証が必要な理由
- Agent に状態、記憶、監視が必要な理由

## プロジェクト面接でよく聞かれること

頻出する質問は、たいてい「どのフレームワークを使いましたか」ではなく、次のようなものです。

- なぜこの方法を選んだのか
- baseline はあったか
- どんな失敗があったか
- どうやって効果を評価したか
- 本番で問題が起きたら、まず何を確認するか

なので、準備するときは次の5つをきちんと話せるようにしておきましょう。

1. プロジェクトの目的
2. 最小限の閉ループ
3. 重要な設計
4. 評価方法
5. 失敗事例と改善の方向性

## 応募前のセルフチェックリスト

- 履歴書を30秒で読んで主軸が分かるか
- 深く話せるプロジェクトが少なくとも2つあるか
- GitHub に README と実行説明があるか
- 自分のプロジェクト構成を口頭で説明できるか
- 一度は失敗したが、その後修正した問題を1つ話せるか
- 目指す職種が何を重視するか分かっているか

この6項目のうち、ほとんどがまだ不安定なら、無限に知識をため込むよりも、まずプロジェクトと説明力を補強するほうが優先です。

## 4週間で進める就職ラッシュ対策の提案

### 第1週

- 職種方向を決める
- 既存プロジェクトを整理する
- 2～3個を主力プロジェクトに選ぶ

### 第2週

- README、スクリーンショット、実行方法を補う
- 履歴書のプロジェクト説明を書き直す
- プロジェクト説明の原稿づくりを始める

### 第3週

- 共通基礎問題を解く
- 職種に合わせて ML / LLM / エンジニアリング基礎を補う
- 模擬面接を行う

### 第4週

- 応募を始める
- よく聞かれる質問を記録する
- フィードバックに応じて、プロジェクトと説明をさらに改善する

## 最も大事なこと

初心者にとって、就職準備とは知識を全部暗記することではなく、「学習の痕跡」を「他人が理解できる成果」に整理し直すことです。

最終的に競争に出すのは、どれだけ資料を読んだかではなく、次のようなことです。

- 何をやったか
- どの程度理解しているか
- きちんと説明できるか
- その後もやりきれるか
