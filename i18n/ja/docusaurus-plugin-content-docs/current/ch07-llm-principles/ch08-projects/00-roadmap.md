# 学習ガイド：総合プロジェクト、この章はどう学ぶべきか

この章では、単に用語を増やすのではなく、これまで学んだ大規模モデルの概要、事前学習、Prompt、微調整、アラインメント、評価を、具体的なプロジェクトの中に本当に落とし込みます。

大規模モデルプロジェクトでいちばん大切なのは、問題を見たらすぐ微調整することでも、すべてのタスクを Prompt だけで無理やり書くことでもありません。大事なのは、その問題が「タスクの表現があいまいなのか」「分野知識が足りないのか」「振る舞いのスタイルが安定しないのか」「評価基準が不足しているのか」を見分ける力です。問題ごとに、選ぶべき方法は変わります。

## この章の全体カリキュラムでの位置づけ

第八段階のテーマは、大規模モデルの能力がどこから来るのか、そして Prompt、微調整、アラインメントによってモデルの振る舞いをどう変えるかを理解することです。総合プロジェクトはこの段階の出口であり、これらの概念を検証可能なエンジニアリング判断に変える場です。

あなたは、明確な分野タスクを中心に、baseline を設計し、Prompt、RAG、または微調整の案を比較し、サンプルを準備し、効果を記録し、なぜそのルートを選んだのかを説明できることを示す必要があります。

![大規模モデル総合プロジェクトのロードマップ](/img/course/ch07-projects-route-map.png)

## この章で本当に解決したい問題

この章では、次の5つの問いに答えます。どうやって1つの大規模モデルプロジェクトを、明確な分野とタスクに絞るのか。どうやっていきなり複雑な手法に進まず、まず Prompt baseline を作るのか。どうやって Prompt、RAG、微調整のどれを使うべきか見極めるのか。どうやってデータと評価セットを準備するのか。どうやってモデルの効果、失敗例、手法の選択理由を見せるのか。

初心者がよくやる間違いは、「モデルの答えがよくない」ことをそのまま「微調整が必要」と考えてしまうことです。実際のプロジェクトでは、多くの問題は、資料不足、Prompt の表現が不明確、出力形式に制約がない、評価サンプルが足りない、あるいはアプリケーションの流れが適切でないことが原因です。

## 初心者におすすめの学習順序

まずは、小さくて明確な分野タスクを1つ選ぶのがおすすめです。たとえば、講義Q&Aの分類、章要約の構造化、カスタマーサポートの意図判定、契約条項の分類、学習アドバイスの生成などです。次に Prompt baseline を作り、テスト用サンプルを集めて、成功例と失敗例を記録します。それから失敗の種類を判断します。もし文脈資料が足りないなら RAG を検討し、固定フォーマットや文体が安定しないなら微調整を検討し、タスク自体があいまいなら、まずタスク定義と評価基準を書き直します。

![大規模モデルプロジェクトの手法選択ループ図](/img/course/ch07-project-method-choice-loop.png)

## この章を学ぶときに押さえる主線

この章の主線は、次のようにまとめられます。大規模モデルプロジェクトは「最強のモデルを選ぶ」ことではなく、タスク、データ、手法、評価の間で取捨選択することです。

この線を理解すると、プロジェクトレポートに「なぜ別の案を使わないのか」を書く理由がわかります。こうした選択理由の説明は、単に結果を見せるよりも、あなたの大規模モデルエンジニアリング能力をよく示してくれます。

## このプロジェクトで本当に鍛えること

このプロジェクトで鍛えるのは、次の4つです。タスクを明確な分野に絞ること、まず Prompt baseline を作ること、Prompt をさらに改善するべきか、RAG をつなぐべきか、微調整をするべきか判断すること、そして最後に評価セットと失敗例で手法の有効性を示すことです。

分野微調整プロジェクトを選ぶ場合は、特にデータ品質、学習/検証の分割、フォーマットの安定性、baseline との比較に注意してください。RAG 方式を選ぶ場合は、資料の出どころ、分割、検索、引用、そして答えがない場合の扱いに注意してください。Prompt 方式を選ぶ場合は、構造化出力、例の設計、バージョンの反復に注意してください。

## この章と次の段階の関係

この章は、第八B段階の LLM アプリ開発と RAG に直接つながります。ここで身につける手法選択の力は、後半で知識ベースQ&A、スマートアシスタント、ツール呼び出し、Agent システムへと広がっていきます。

この章が固まっていないと、後でよくある問題は次のようになります。問題を見るとすぐフレームワークを積み上げる。微調整、RAG、Prompt を同じものとして混ぜてしまう。評価セットがない。手法がなぜ有効なのか説明できない。プロジェクト発表で最終回答しか見せず、失敗分析や取捨選択の過程がない。

## 初心者と上級学習者はどう読むか

初心者がこの章を初めて学ぶときは、まず主線と最小の動く例に集中しましょう。すべての細部を一度に理解する必要はありません。この章が何を解決するのか、入力と出力は何か、最小プロジェクトをどう動かすのかが説明できれば、次に進んで大丈夫です。

経験のある学習者は、この章を穴埋めとエンジニアリング練習として使えます。境界条件、失敗例、評価方法、コードの再現性、そして前後の段階とのつながりに注目してください。読み終えたら、本章の内容を自分の作品 README や実験記録にまとめるのがおすすめです。

## 学習時間と難易度の目安

| 学習方法 | 推奨時間 | 目標 |
|---|---|---|
| ざっと確認 | 20～30 分 | この章が何を解決するのかを理解し、後でどこで使うかを知る |
| 最小クリア | 1～2 時間 | 最小例を動かし、この章の小プロジェクト出口まで到達する |
| しっかり練習 | 半日～1日 | エラー分析、比較実験、または README の記録を補強する |

## 本章の自己チェック問題

| 自己チェック問題 | 合格基準 |
|---|---|
| この章は何を解決するのか？ | コース全体の中での位置づけを1文で説明できる |
| 最小の入力と出力は何か？ | 例に必要な入力と、何が出力されるかを説明できる |
| よくある失敗点はどこか？ | エラー、精度の悪さ、理解のずれの原因を少なくとも1つ挙げられる |
| 学んだあとに何を残せるか？ | 本章の成果を README、実験記録、ポートフォリオに書ける |

## 本章の小プロジェクト出口

この章を学び終えたら、「分野タスクの大規模モデル手法比較プロジェクト」を1つ完成させるのがおすすめです。プロジェクトには少なくとも、タスク定義、サンプルデータ、Prompt baseline、失敗分析、改善案、評価結果、結論の説明を含めましょう。

最小版では Prompt を2種類比較するだけでも構いません。発展版では Prompt、RAG、小規模微調整を比較できます。ポートフォリオ版では、手法の取捨選択、評価指標、失敗例、次のステップをはっきり書く必要があります。

## Debug 事件簿

| 案件 | 内容 |
|---|---|
| 案件名 | JSON ドリフト事件 |
| 現場 | LLM の出力が、あるときはフィールド不足で、あるときは説明が多すぎて、構造化結果を安定して解析できない。 |
| 調査手順 | 10個の入力を固定し、元の出力を保存し、schema 検証を行い、Prompt バージョンを比較する。 |
| 終結証拠 | prompt_eval_cases.csv、Prompt バージョン表、schema 通過率。 |

プロジェクト練習では、成功したスクリーンショットだけを残さないでください。少なくとも1つは本当の失敗サンプルを選び、「現象、手がかり、疑わしい原因、調査手順、修復アクション、回帰確認」の形で `reports/failure_cases.md` に書きましょう。そうすると、プロジェクトがより実際のエンジニアリング作品らしくなります。

## プロジェクト成果物の基準

それぞれの大規模モデル総合プロジェクトは、1回のモデル回答を見せるだけでなく、同じ作品集基準で提出するのがおすすめです。最小の成果物には、README、再現可能な実行コマンド、サンプル入出力、Prompt バージョン記録、1件の失敗サンプル分析、次の改善計画を含めるべきです。

| 成果物 | 最低要件 | 発展要件 |
|---|---|---|
| README | プロジェクト目標、実行方法、モデル、サンプルを明記する | 手法の取捨選択、コスト見積もり、評価、振り返りを追加する |
| サンプル入出力 | 少なくとも1つの固定テストサンプルを残す | Prompt、RAG、微調整、またはルール方式の比較サンプルを残す |
| 評価記録 | 出力の良し悪しを何で判断したかを明記する | 固定評価セット、人手スコア、失敗タイプ統計を追加する |
| Prompt/データ記録 | Prompt バージョンや学習サンプル形式を保存する | schema、検証、データ品質、安全境界の説明を追加する |
| 発表素材 | 動作することがわかるスクリーンショットや短い GIF | 説明しやすい大規模モデル手法選択の事例にする |

大規模モデルプロジェクトで最も重要なのは、「モデルの答えがどれだけそれっぽいか」ではありません。問題の原因が、タスク表現、知識不足、フォーマットの安定性、それとも評価不足のどれなのか、そしてなぜ今の技術ルートを選んだのかを、きちんと説明できることです。

## 合格基準

この章が終わるとき、あなたは1つの分野タスクを中心に baseline を作れ、Prompt、RAG、微調整がそれぞれ何に向いているか判断でき、小規模な評価セットを準備でき、失敗例でモデルの限界を説明でき、技術案の取捨選択をプロジェクトレポートとして書けるはずです。

「なぜここで直接微調整しないのか」「なぜここでは RAG が必要なのか」「なぜこの Prompt の変更が効いたのか」をはっきり説明できれば、大規模モデルの原理と微調整段階の作品集出口基準を満たしています。

## バージョン別ルートのおすすめ

| バージョン | 目標 | 成果物の重点 |
|---|---|---|
| 基礎版 | 最小の閉ループを動かす | 入力できる、処理できる、出力できる、そして1組のサンプルを残す |
| 標準版 | 見せられるプロジェクトにする | 設定、ログ、エラー処理、README、スクリーンショットを追加する |
| 挑戦版 | 作品集品質に近づける | 評価、比較実験、失敗サンプル分析、次のルートを追加する |

まずは基礎版を完成させることをおすすめします。最初から大きく全部そろえようとしないでください。1つバージョンを上げるたびに、「何が新しくできるようになったのか、どう検証したのか、まだ何が問題なのか」を README に書き足しましょう。
