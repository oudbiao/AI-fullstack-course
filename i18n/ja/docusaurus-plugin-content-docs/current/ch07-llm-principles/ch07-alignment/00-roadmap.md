# 学習ガイド：アライメントこの章では何を学ぶのか

## 本章の位置づけ

事前学習によってモデルは汎用的な言語能力を獲得し、ファインチューニングによってタスクに適応します。しかし、これだけではモデルが人間の期待する形で回答するとは限りません。アライメントの章で扱うのは、モデルをどうすればより役立ち、より正直で、より安全にし、利用者の意図や制約により合うようにできるか、という点です。

事前学習が「モデルは何を知っているか」、ファインチューニングが「モデルはどんなタスクが得意か」を答えるのに対して、アライメントは「モデルはどう振る舞うべきか」を答えます。ChatGPT 以降、アライメントが研究テーマから大規模モデル製品体験の中核課題へと変わったのも、このためです。

## 本章が大規模モデルの学習ルートの中で占める位置

![大規模モデルのアライメント章の関係図](/img/course/ch07-alignment-chapter-flow-ja.png)

アライメントは独立した小技ではなく、モデル能力・ユーザー体験・安全治理をつなぐ一連の方法です。後で RAG、Agent、ツール呼び出しを扱うときも、アライメントの問題に引き続き出会います。たとえば、モデルはいつ拒否すべきか、いつ確認を求めるべきか、いつ出典を捏造してはいけないのか、いつ勝手に操作を実行してはいけないのか、という点です。

## 本章の学習主線

| 小節 | 重点課題 | 学習後に説明できるようになること |
|---|---|---|
| アライメントの問題 | なぜ能力のあるモデルでも使いにくいことがあるのか | 幻覚、迎合、権限逸脱、偏見、不安定な出力 |
| RLHF | 人間の好みを使ってモデルの振る舞いをどう学習させるか | SFT、報酬モデル、強化学習のおおまかな流れ |
| 代替手法 | なぜ DPO、RLAIF などの手法が登場したのか | アライメント手法の工学コストと代替ルート |

学習するときは数式の細部に入り込みすぎず、次の点をつかむことが大切です。人間の好みはどう集めるのか、モデルの振る舞いはどう比較するのか、安全の境界はどう組み込むのか、評価はアライメントが本当に改善したかをどう判断するのか。

## アライメントとアプリケーション開発の関係

多くのアプリケーション問題は Prompt だけでは解決できません。たとえば、カスタマーサポート用のボットはポリシーを捏造してはいけませんし、医療アシスタントは権限を超えて診断してはいけません。Agent はファイルを直接削除したり、支払いを実行したりしてはいけません。システムプロンプト、ツール権限、RAG の引用、人間による確認を使って振る舞いを制約できますが、根本のモデルが指示を守る傾向があるか、分からないことを認められるか、危険な出力を避けられるかは、やはりアライメントに関係しています。

![アライメントとアプリケーションの安全境界図](/img/course/ch07-alignment-app-safety-map-ja.png)

## 本章の小プロジェクトの出口

本章では RLHF を自分で学習させることまでは求めません。代わりに、「モデルの振る舞い比較記録表」を作るのがおすすめです。あいまいな要件、矛盾する指示、出典不足、権限逸脱のツール要求、安全境界に関する要求など、問題が起きやすい質問を 10 個設計し、異なる Prompt や異なるモデルの回答差を比較します。基礎版は Markdown 表にします。標準版では、有用性、誠実性、境界意識、引用の信頼性などの評価軸を加えます。チャレンジ版では、これを後半の RAG や Agent の評価セットに接続します。

## よくある誤解

1つ目の誤解は、アライメントを「モデルをもっと素直にすること」と同一視してしまうことです。本当のアライメントには、やるべきでないことを拒否すること、分からないと認めること、誤解を招かないこと、ユーザーを守ることも含まれます。  
2つ目の誤解は、アライメントはモデルの学習段階でしか起こらないと思うことです。アプリケーション層でのシステムプロンプト、ツール権限、人間による確認、ログ監査も、広い意味でのアライメントの一部です。  
3つ目の誤解は、1回の回答だけを見て、体系的な行動評価をしないことです。

## 合格基準

この章を学び終えたら、事前学習・ファインチューニング・アライメントの違いを説明でき、RLHF の基本的な流れを説明でき、DPO などの代替手法がなぜ出てきたのかを理解し、「有用性・誠実性・安全境界」を自分の LLM アプリケーション評価チェックリストに書き込めるようになっているはずです。
