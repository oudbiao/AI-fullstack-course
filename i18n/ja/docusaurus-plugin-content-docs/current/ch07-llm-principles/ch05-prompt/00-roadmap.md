# 学習ガイド：Prompt エンジニアリングのこの章では何を学ぶのか

この章で解決するのは、モデルのパラメータを変えない前提で、どうすればモデルにタスクをより安定して理解させ、制約を守らせ、製品要件に合った結果を出させられるか、ということです。

初心者が Prompt を初めて学ぶとき、「もっとそれっぽく聞くための質問テクニック」と捉えがちです。でも実際の大規模モデルアプリケーションでは、Prompt はアプリケーション層とモデル層のあいだのインターフェース設計に近いものです。つまり、モデルに対してタスクの目標、入力 सामग्री、出力形式、制約条件、例、そして失敗の境界を伝える必要があります。

## この章がコース全体の中でどこにあるか

あなたはすでに大規模モデルの概要、Transformer、事前学習を学び、モデルの能力がどこから来るのかを大まかに理解しています。Prompt エンジニアリングの章に入ると、コースはより実用に近い問いに答え始めます。つまり、モデルに能力があるなら、それをどうやって安定して引き出すのか、です。

Prompt は「モデルの能力」と「製品機能」のあいだにあります。モデルのパラメータは変えませんが、タスクの理解、推論の組み立て、文脈の参照、出力の構造、境界ケースの処理に大きな影響を与えます。

## この章で本当に解決したい問題

この章では、次の5つの問いに答えます。あいまいな要求をどうやって明確なタスクに言い換えるか。背景資料、役割、手順、制約をどう整理するか。どうすれば JSON、表、Markdown などの安定した形式で出力させられるか。例示、段階的な指示、自動チェックでどうやってミスを減らすか。感覚で何度も書き換えるのではなく、Prompt をどう評価して改善していくか。

初心者が見落としやすいのは、Prompt は一度書いたら終わりではない、という点です。実際のアプリケーションでは、Prompt にテストセット、エラー例、バージョン管理、反復の記録が必要です。そうでないと、ある修正で本当にシステム全体が良くなったのか、それともたまたまある例だけが良くなったのかを判断しにくくなります。

## 初心者におすすめの学習順序

まずは基本の Prompt から学び、タスク、文脈、制約、出力形式をはっきり書けるようになりましょう。次に、few-shot の例示、段階的な分解、役割設定、自己チェック、境界条件の説明といった高度なテクニックを学びます。その後は構造化出力を重点的に学びます。多くの製品機能は自然言語の文章だけではなく、安定したフィールド、形式、解析可能な結果を必要とするからです。最後に Prompt の実践として、実データでテストし、失敗を記録し、改善を繰り返します。

![Prompt 反復テストの閉ループ図](/img/course/ch07-prompt-iteration-loop-ja.png)

## この章で押さえるべき主線

この章の主線は、Prompt は「一度質問すること」ではなく、「再利用できるモデル呼び出しを設計すること」だ、という点にまとめられます。

この考え方は、「会話で思いついたままモデルに聞く」のと「製品の中で安定してモデルを呼び出す」の違いを見分ける助けになります。前者はその場で答えが返れば十分ですが、後者には予測可能性、解析可能性、テスト可能性、保守性が必要です。

## この章と後続章との関係

Prompt は、ファインチューニング、RAG、Agent に進むための前提スキルです。ファインチューニングの前には、まず Prompt だけでは要件を満たせないのかを確認する必要があります。RAG では、検索した断片をどのように文脈へ入れるか、そしてモデルがどのように情報源に基づいて答えるかを Prompt が決めます。Agent では、タスク計画、ツール選択、観察の要約、最終出力に Prompt が影響します。

この章をしっかり学ばないまま進むと、よくある問題として、RAG で資料は取れているのに回答が発散する、Function Calling のパラメータが安定しない、構造化出力がよくパースに失敗する、すべての問題をファインチューニングで解決しようとしてしまい、タスク表現やテスト用サンプルを先に整えない、といったことが起こります。

## 初心者と上級学習者はどう読むか

初心者がこの章を初めて学ぶときは、主線と最小実行例をまず押さえましょう。すべての細かい点を一度で理解する必要はありません。この章が何を解決するのか、入力と出力は何か、最小のプロジェクトをどう動かすのかを説明できれば、先へ進んで大丈夫です。

経験のある学習者は、この章を知識の穴埋めと実践的な工学練習として使えます。境界条件、失敗例、評価方法、コードの再現性、そして前後の段階とのつながりに注目しましょう。読み終えたら、本章の内容を自分の作品の README や実験記録に残しておくとよいです。

## 学習時間と難易度の目安

| 学習方法 | 推奨時間 | 目標 |
|---|---|---|
| ざっと読む | 20～30 分 | この章が何を解決するのかを理解し、後でどこで使うかを把握する |
| 最小到達 | 1～2 時間 | 最小例を動かし、本章の小プロジェクトの出口まで進む |
| しっかり練習 | 半日～1 日 | エラー分析、比較実験、またはプロジェクト README の記録を補う |

## 本章の自己チェック問題

| 自己チェック問題 | 合格基準 |
|---|---|
| この章は何を解決するのか？ | コース全体の中での位置づけを1文で説明できる |
| 最小の入力と出力は何か？ | 例に必要な入力と、得られる結果を説明できる |
| よくある失敗点はどこか？ | 少なくとも1つ、エラー・効果の悪さ・理解のずれの原因を挙げられる |
| 学び終えたら何を残せるか？ | 本章の成果をプロジェクト README、実験記録、作品集に書ける |

## 本章の小プロジェクトの出口

この章を学び終えたら、「コース内容の構造化抽出 Prompt」を作るのがおすすめです。1つの講義文書を入力として与え、モデルに章のテーマ、学習目標、前提知識、重要概念、練習の提案、注意すべきリスクを出力させます。出力形式は JSON または Markdown の表にします。

このプロジェクトの重点は、モデルがきれいに答えられるかではありません。出力が安定しているかを確認することです。つまり、フィールドがすべてそろっているか、形式が解析可能か、情報が欠けているときに正直にその旨を示せるか、複数の文書に変えても一貫性を保てるかをテストします。

## 合格基準

この章の終わりには、あいまいな要求をタスク目標、入力材料、制約条件、出力形式に分解できるようになっているはずです。また、基本 Prompt、例示 Prompt、構造化出力 Prompt を設計でき、テスト用サンプルのセットで Prompt の安定性を判断できるようになっている必要があります。

失敗例をもとに、感覚ではなく方向性を持って Prompt を修正できるなら、ファインチューニング、RAG、Agent のアプリケーション開発に進むための基礎条件は満たせています。
