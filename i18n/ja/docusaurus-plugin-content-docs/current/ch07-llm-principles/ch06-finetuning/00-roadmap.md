# 7.6.1 学習ガイド：微調のこの章では何を学ぶのか

この章で解決したいのは、Prompt だけではモデルの振る舞いを安定して変えられなくなったとき、どう訓練してモデルを特定のタスク、形式、または分野にもっと合うようにするか、ということです。

微調は「モデルを何でも強くする」ための魔法のボタンではありません。スタイル、形式、分野表現、固定されたタスクパターン、特定の振る舞いのくせを改善するのに向いています。知識の更新が必要な問題の多くは、実は RAG のほうが向いています。単発の作業の多くは、実は Prompt のほうが向いています。安定したタスク、十分なサンプル、明確な評価基準があるときだけ、微調を本気で検討する価値があります。

## この章がコース全体の中でどこにあるか

あなたはすでに大規模モデルの全体像、事前学習、Prompt エンジニアリングを学んできました。事前学習はモデルの汎用能力がどこから来るのかを説明し、Prompt はパラメータを変えずに能力を呼び出す方法を説明します。微調は別の道に入ります。つまり、すでにあるモデルを土台にして、タスクデータでさらに訓練し、モデルの振る舞いを目標に近づけるのです。

![大規模モデル微調の章の関係図](/img/course/ch07-finetuning-chapter-flow-ja.png)

## この章が本当に解決したい問題

この章では次の5つの問いに答えます。どんなときに微調すべきで、どんなときにすべきでないのか。微調データをどう集め、どうクリーニングし、どうラベル付けし、どう分割するのか。LoRA、QLoRA、その他の PEFT 手法がなぜ訓練コストを下げられるのか。微調の訓練には大まかにどんな手順があるのか。どう評価して、微調が本当に効いているのか、それとも訓練サンプル上で良く見えるだけなのかを見分けるのか。

初心者が一番誤解しやすいのは、モデルが分野知識を間違えたら、すぐ微調したくなることです。実際には、問題が「情報が新しすぎる、知識が私有すぎる、引用できる出典が必要」であれば、RAG のほうが合っていることが多いです。問題が「出力形式、口調、タスクの型が長期的に不安定」であれば、微調のほうが価値を発揮しやすいです。

## 初心者におすすめの学習順序

まず微調の概要を見て、「なぜ微調するのか」「いつ微調しないのか」の境界をつかみましょう。次に LoRA/QLoRA を学びます。これらは今もっとも一般的で、コストも低めな入門用微調の道だからです。その後、他の PEFT 手法を学んで、フル微調以外にもさまざまなパラメータ効率の高い方法があることを知りましょう。最後に微調の実践とデータラベル付けを見て、データ準備、訓練設定、検証集、評価サンプル、公開時のリスクをつなげて理解します。

![微調の意思決定と評価のループ図](/img/course/ch07-finetuning-decision-loop-ja.png)

## この章で押さえるべき主線

この章の主線はこうまとめられます。微調はゼロからモデルを作ることではなく、すでにあるモデルに高品質なサンプルを与えて振る舞いを形作ることです。

この線が見えるようになると、微調プロジェクトで一番高くつくのは GPU とは限らず、データと評価だと分かります。よいデータがなければ、微調は間違いをより安定して覚えてしまいます。評価がなければ、モデルが汎化したのか、それとも単にサンプルを覚えただけなのか判断できません。

## この章と後続の章の関係

微調は、RAG、Prompt、アラインメント、モデルデプロイと一緒に、大規模モデルアプリケーションの技術選択フレームを作ります。Prompt は呼び出し方を解決し、RAG は外部知識を解決し、微調は安定した振る舞いを解決し、アラインメントは人間の好みと安全境界を解決し、デプロイはコスト、遅延、可用性を解決します。

この章をしっかり学ばないと、後でよくある問題は次のようになります。微調を万能解決策だと思ってしまう。検証集がないのに効果を判断してしまう。訓練データが乱雑で、モデルの形式がかえって不安定になる。微調で、本来は RAG で解決すべき私有知識の更新問題を解こうとしてしまう。訓練 loss だけを見て、実際のタスク性能を見ない。

## 初心者と上級学習者はどう読むか

初心者がこの章を初めて学ぶときは、まず主線と最小限の実行例をつかみましょう。すべての細部を一度で理解する必要はありません。この章が何を解決するのか、入出力は何か、最小プロジェクトをどう動かすかを説明できれば、先へ進めます。

経験のある学習者は、この章を抜け漏れチェックと工程化の練習として使えます。境界条件、失敗例、評価方法、コードの再現性、そして前後の段階とのつながりに注目しましょう。読み終わったら、本章の内容を自分の作品の README や実験記録に残せると理想です。

## 学習時間と難易度の目安

| 学習方法 | 推奨投入時間 | 目標 |
|---|---|---|
| ざっと読む | 20〜30分 | この章が何を解決するかを理解し、後でどこで使うかを知る |
| 最小通過 | 1〜2時間 | 最小例を動かし、この章の小プロジェクトの出口まで進む |
| 深く練習する | 半日〜1日 | エラー分析、比較実験、またはプロジェクト README の記録を補う |

## この章の自己チェック問題

| 自己チェック問題 | 合格基準 |
|---|---|
| この章は何を解決するのか？ | コース全体の中での位置を一言で説明できる |
| 最小の入力と出力は何か？ | 例に何が必要で、どんな結果が出るか説明できる |
| よくある失敗点はどこか？ | エラー、性能不足、理解のずれの原因を少なくとも1つ挙げられる |
| 学んだ後に何を残せるか？ | 本章の成果をプロジェクト README、実験記録、または作品集に書ける |

## この章の小プロジェクトの出口

この章を学び終えたら、小さな指示微調の実験をしてみるのがおすすめです。たとえば「講義章節の要約を固定の JSON 構造に書き換える」や「ユーザーの質問を学習パス、概念説明、プロジェクト提案、環境問題に分類する」といった、はっきりしたタスクを選びます。数十から数百件のサンプルを用意し、まず Prompt の baseline を作り、それから LoRA/QLoRA で小規模な微調を行い、最後に形式の安定性、正確率、または人手評価を比較します。

このプロジェクトで大事なのは、タスク定義、データ準備、訓練、評価、baseline との比較、失敗例の記録までを一通り回すことです。大きなモデルを作ることが目的ではありません。

## 合格基準

この章が終わるころには、ある問題が微調に向いているか判断でき、LoRA、QLoRA、PEFT の基本的な役割を説明でき、微調データがなぜ訓練スクリプトより重要なのかを言え、最小の微調評価案を設計できるようになっているはずです。

Prompt、RAG、微調を同じ意思決定図の中に並べて、さまざまな場面でどの道を優先すべきか説明できれば、かなり成熟した大規模モデル工程の判断力が身についていると言えます。
