# 7.2.1 学習前ガイド：LLM 概要のこの章で何を学ぶのか

この章で解決したいのは、初めて大規模モデルを体系的に学ぶときに、まず地図を頭の中に作ることです。最初からすべてのモデル名を追いかける必要はありませんし、すぐにパラメータ規模やランキング、フレームワーク比較に入り込む必要もありません。まずは、大規模モデルがどこから来たのか、コア概念は何か、能力の境界はどこか、そしてなぜ AI アプリ開発の新しい土台になったのかを知ることが大切です。

## この章がコース全体の中でどこにあるか

ここまでに、あなたは NLP の速習と Transformer の基礎を学んできました。この章では、token、embedding、attention、事前学習といった概念を「大規模モデル時代」の全体像の中に置き直していきます。

この章は、特定のモデルを1つ取り上げて説明する章ではありません。むしろ、判断のためのフレームを作る章です。GPT、Claude、Gemini、LLaMA、Qwen、DeepSeek、あるいはほかのモデルを見たときに、訓練方式、コンテキスト能力、推論能力、ツール利用能力、デプロイ方法、アプリケーションエコシステムの観点から理解できるようになることが目的です。名前やパラメータ数だけで判断しないようにしましょう。

![LLM 概要章の関係図](/img/course/ch07-llm-overview-chapter-flow-ja.png)

## この章で本当に解決したい問題

この章では、まず次の4つの問題を解決します。なぜ大規模モデルは、従来の NLP や事前学習モデルから発展してきたのか。パラメータ数、コンテキストウィンドウ、token、embedding、推論、アラインメントという言葉は、それぞれどこに位置づくのか。オープンソースモデル、クローズドモデル、ローカルデプロイ、クラウド API にはどんな違いがあるのか。そして、大規模モデルのアプリケーションはなぜチャットだけではなく、RAG、ツール呼び出し、Agent へと進んでいくのか、という点です。

初心者にとって、この章で最も大事なのは、各モデルの公開時期を覚えることではありません。「モデルを見るための座標軸」を作ることです。この座標軸ができて初めて、次に学ぶ Prompt、ファインチューニング、アラインメント、RAG、Agent が混乱しにくくなります。

## 初心者におすすめの学習順序

まずは発展の歴史から見ましょう。大規模モデルは突然出てきたのではなく、統計的 NLP、単語ベクトル、Transformer、事前学習、指示微調整、人間のフィードバックによるアラインメントを、少しずつ経て発展してきたことを理解します。次にコア概念を見て、パラメータ、token、コンテキスト、embedding、推論、幻覚、アラインメントといった言葉を正しい位置に置きます。その後で産業構造を見て、オープンソース/クローズド、クラウド/ローカル、汎用/特化モデルの間で何を選ぶのかを理解しましょう。最後に LLM 呼び出しワークベンチを実践し、これらの概念を実行可能で検証可能なリクエストと再試行の流れに変えます。

![大規模モデル能力スタックとアプリケーションエコシステム図](/img/course/ch07-llm-capability-stack-ja.png)

## この章で押さえるべき主線

この章の主線は、次のようにまとめられます。大規模モデルとは、大規模データ、Transformer 構造、事前学習目標、指示アラインメント、アプリケーション API を組み合わせてできた、新しい能力の土台です。

この線をたどると、「モデルそのものの能力」と「アプリケーションシステムが補う能力」を区別しやすくなります。たとえば、モデルのパラメータには一般的な知識が一部入っていますが、最新の企業文書を使うには通常 RAG が必要です。モデルは文章を生成できますが、信頼性高くタスクを実行するには、通常ツール呼び出しや状態管理が必要です。モデルは質問に答えられますが、本番システムでは評価、監視、権限管理、コスト管理も必要になります。

## この章と後続章の関係

この章は、第7段階の地図ページです。後の Transformer 深掘りでは土台の構造を説明し、事前学習の章では能力がどこから来るのかを説明します。Prompt の章では能力をどう引き出すか、ファインチューニングの章ではモデルの振る舞いをどう変えるか、アラインメントの章ではなぜモデルを人間の意図により合うようにする必要があるのかを説明します。

この章をしっかり学ばないと、後で次のようなつまずきが起きやすくなります。Prompt、ファインチューニング、RAG、Agent を全部「モデルを強くする方法」と同じものだと思ってしまう。モデルのランキングだけを見て、自分のアプリケーションに本当に必要なのが知識更新なのか、出力形式の安定なのか、推論能力なのか、ツール実行なのかが分からない。オープンソースかクローズドかを早く気にしすぎて、先にシナリオ要件を定義していない、などです。

## 初心者と中級者はどう読むか

初心者がこの章を初めて学ぶときは、まず主線と最小実行例をつかみましょう。すべての細部を一度に理解する必要はありません。この章が何を解決するのか、入出力は何か、最小プロジェクトはどう動かすのかを説明できれば、先へ進んで大丈夫です。

経験のある学習者は、この章を「抜け漏れの確認」と「実践的な練習」の場として使えます。境界条件、失敗例、評価方法、コードの再現性、そして前後の章とのつながりに注目しましょう。読み終わったら、本章の内容を自分の作品 README や実験記録にまとめておくとよいです。

## 学習時間と難易度の目安

| 学習方法 | 推奨時間 | 目標 |
|---|---|---|
| ざっと読む | 20～30 分 | この章が何を解決するのかを理解し、後でどこで使うかを知る |
| 最小合格 | 1～2 時間 | 最小例を動かし、この章の小さなゴールを達成する |
| じっくり練習 | 半日～1 日 | エラー分析、比較実験、またはプロジェクト README の記録を補う |

## この章の自己チェック問題

| 自己チェック問題 | 合格基準 |
|---|---|
| この章は何を解決するのか？ | コース全体の中での位置を一文で説明できる |
| 最小の入出力は何か？ | 例に何を入力し、どんな結果が出るのかを説明できる |
| よくある失敗ポイントはどこか？ | 少なくとも1つ、エラー・悪い結果・理解のずれの原因を挙げられる |
| 学び終えた後に何を残せるか？ | 本章の成果をプロジェクト README、実験記録、作品集に書ける |

## この章の小プロジェクトの出口

この章を学び終えたら、「モデル選択カード」を1枚作るのがおすすめです。よく聞く大規模モデルを3つ選び、モデルの種類、呼び出し方法、コンテキスト長、適した場面、制限、コストの観点で比較してみましょう。この小プロジェクトでは、完全さよりも、統一された座標軸でモデルを見る練習をすることが大切です。

また、最小の LLM API 呼び出し例を書いて、入力 prompt、出力結果、token またはコスト情報を記録し、この呼び出しは「大規模モデルアプリケーションシステム」のモデル層にすぎず、完全な製品ではないことを説明してもよいです。この章の呼び出しワークベンチは安全な出発点になります。まずオフラインシミュレーターを動かし、その後で fake model response を実際の Responses API 呼び出しに置き換えます。

## 合格ライン

この章の終わりには、LLM と従来の NLP モデルの違いを説明でき、token、コンテキストウィンドウ、パラメータ数、事前学習、指示微調整、アラインメントのおおまかな意味を説明でき、オープンソースモデル、クローズドモデル、ローカルデプロイ、クラウド API の基本的な違いを区別できるようになっているはずです。

さらに、ある問題を Prompt、ファインチューニング、RAG、Agent のどれで解くのがよいか判断できるなら、あなたはすでに大規模モデルアプリケーションのシステム的な視点を身につけ始めています。
