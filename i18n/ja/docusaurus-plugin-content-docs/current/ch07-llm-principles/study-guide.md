# 学習ガイドとタスクシート：大規模モデルの原理、Prompt とファインチューニングはどう学ぶと一番混乱しにくいか

`7 大規模モデルの原理、Prompt とファインチューニング` に入って、用語が多いと感じたら、まずはモデルのランキングを追わなくて大丈夫です。先に、大規模モデルの能力はどこから来るのか、そしてその能力をどう制御し、どう適応させるのかを理解しましょう。

## この段階の基本原則

大規模モデルを最初に学ぶときは、1 本の進化の流れをつかむことが大切です。テキストが token に分かれ、token が embedding になり、Transformer が文脈をモデリングし、事前学習で汎用能力を獲得し、Prompt・ファインチューニング・アラインメントによって能力がより使いやすくなります。

![大規模モデル学習ガイドの進化の流れ](/img/course/ch07-study-guide-evolution-line-ja.png)

## この段階で必ず完了するタスク

Prompt 作業は工程入力として扱います。バージョンを記録し、固定サンプルでテストし、構造化出力を検証し、失敗サンプルを残します。

| タスク | 成果物 | 合格基準 |
|---|---|---|
| 大規模モデルの基本原理を理解する | LLM ワークフロー図またはノート | token、文脈、確率生成、Transformer の関係を説明できる |
| Prompt 基礎演習を終える | Prompt 比較サンプル一式 | 役割、文脈、例、制約が出力に与える影響を比較できる |
| 構造化出力を実装する | JSON 出力デモ 1 つ | フィールドを定義し、型を検証し、解析失敗を処理できる |
| RAG と微調整の境界を理解する | 技術選定表 1 枚 | Prompt、RAG、微調整、ツール、ルールをいつ使うべきか説明できる |
| 第 7 章フルワークショップを実行する | `llm_stage_workshop.py` のターミナル出力 | 1 回の実行から token、Prompt 版、検証失敗、手法ルートを説明できる |
| 段階プロジェクトを 1 つ完成する | Prompt アシスタント、または振り返りカード生成器 | Prompt のバージョン、固定入力、出力比較、失敗サンプルがある |

## おすすめの学習順序

1 回目はまず NLP の核心を速習で補いましょう。少なくとも tokenizer、embedding、言語モデル、HuggingFace の基本的な使い方は理解しておく必要があります。

2 回目は LLM の全体像と発展の歴史を学びます。大事なのはモデル名を覚えることではなく、規模、データ、アーキテクチャ、アラインメントがどのように連携して能力の変化を生み出したのかを理解することです。

3 回目は Transformer と事前学習を学びます。Attention、コンテキストウィンドウ、学習データ、計算資源がなぜ重要なのかを知る必要があります。

4 回目は Prompt を学びます。Prompt は最も軽量な挙動制御の方法であり、その後の構造化出力、RAG、Agent の基礎でもあります。

5 回目はファインチューニングとアラインメントを学びます。どんなときにファインチューニングが必要なのか、LoRA/QLoRA は何を解決するのか、RLHF がなぜモデルの挙動に関係するのかを重点的に理解しましょう。

6 回目に [実践：第 7 章フルワークショップ](./ch08-projects/03-stage-hands-on-workshop.md) を実行します。この段階では、ワークショップは復習ラボになります。token、Prompt 評価、構造化検証、手法選択を 1 回の実行でつなぎます。

## 学習ペースの目安

| 内容の種類 | 目安時間 | 学習目標 |
|---|---|---|
| NLP 速習 | 4～8 時間 | token、embedding、言語モデルを理解する |
| LLM 全体像 | 3～5 時間 | 発展の歴史と能力の境界をつかむ |
| Transformer / 事前学習 | 8～16 時間 | 能力の源を理解する |
| Prompt | 4～8 時間 | 構造化されたタスク用プロンプトを設計できる |
| ファインチューニング / アラインメント | 8～16 時間 | ファインチューニングが必要かどうかと、そのリスクを判断できる |

## この段階のプロジェクトの進め方

最初のプロジェクトは、Prompt の比較実験をするのがおすすめです。1 つのタスクを選び、通常の指示、役割付きプロンプト、段階的な指示、少数例プロンプト、構造化出力を比較してみましょう。

どこから始めればよいか迷う場合は、第 7 章フルワークショップをスタータープロジェクトにしてください。出力を保存し、各 Prompt 版がなぜ合格または失敗するのか説明してから、自分の AI 学習アシスタント用のテストケースへ広げます。

2 つ目のプロジェクトは、構造化出力のタスクを作るのがおすすめです。たとえば、自然言語を JSON、表、または関数パラメータに変換します。

3 つ目のプロジェクトでは、特定分野向けのファインチューニング方案を設計できます。すぐに大規模モデルを学習しなくてもかまいませんが、データ形式、ラベル付け方法、学習方法、評価指標、リスクを説明できるようにしておきましょう。

## よくあるつまずき

最もよくあるつまずきは、Prompt、RAG、ファインチューニングを混同してしまうことです。Prompt は入力の組み立て方を変え、RAG は外部知識を補い、ファインチューニングはモデルの挙動の傾向を変えます。解決する問題はそれぞれ違います。

2 つ目のつまずきは、大規模モデルをデータベースのように扱ってしまうことです。大規模モデルは幻覚を起こすことがあり、知識は古くなっていることもあり、回答がそのまま事実とは限りません。

3 つ目のつまずきは、早すぎるファインチューニングです。多くの問題は、まず Prompt、RAG、ツール呼び出し、またはプロダクトのロジックで解決すべきです。

## 段階ポートフォリオ成果物

この段階の成果を作品集に残したいなら、少なくとも次のファイル、または同等の資料を残すことをおすすめします。

| 成果物 | 説明 |
|---|---|
| `prompts/` | Prompt テンプレート、バージョン番号、適用タスク、変更説明を保存する |

これらの材料があると、Prompt プロジェクトは「プロンプトを書ける」から「モデル出力を安定して、テスト可能で、保守しやすいアプリのインターフェースにできる」へと進化します。

## 段階通過質問

この段階を学び終えたら、Token、Embedding、Attention、事前学習、Prompt、ファインチューニング、アラインメントの関係を説明できるようになっているはずです。

第 8 章へ進む前に、次の質問に答えられるか確認してください。

- token と文脈制限はなぜ重要なのか。
- Prompt 内の例はなぜ出力に影響するのか。
- 構造化出力にはなぜ schema 検証が必要なのか。
- Prompt 文を増やすより RAG を使うべきなのはどんなときか。
- 微調整を検討する価値があるのはどんなときか。

ある要件に対して Prompt、RAG、ファインチューニング、ツール、ルールのどれを使うべきか判断でき、構造化出力の Prompt を設計できれば、LLM アプリケーションと RAG の段階に進んで大丈夫です。
