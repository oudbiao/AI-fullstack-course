# 8.2.1 学習ガイド：この章で学ぶモデルデプロイとは何か

この章で解決したいことは、次のとおりです。

> **モデルは「ある」だけでは不十分で、安定して読み込まれ、呼び出され、サービスとして提供される必要があります。**

## まずは橋渡しの線をつくろう

RAG の本編を学び終えたばかりなら、この章でまず押さえたいのは次の点です。

- これまでに、知識がどのようにシステムへ入るかを学んだ
- この章からは、モデルがどうやって実際に呼び出され、安定した能力の入口になるのかを学ぶ

つまり、デプロイ章で本当に大事なのは「サービスを動かせるか」ではなく、

> **モデルの呼び出しが、1回きりの実験から、再利用できて、置き換え可能で、保守しやすいインターフェース能力へと変わることです。**

## この章の主な流れ

![モデル配置章の学習順序図](/img/course/ch08-deployment-chapter-flow-ja.png)

## この章は初心者にとって、こういう順番で学ぶのがおすすめ

1. まずはローカルでのモデル実行を見る  
   まず、モデルがどう読み込まれ、どう推論されるのかを理解します。

2. 次に推論サービスを見る  
   「ローカル推論」を「システムから呼び出せるサービス」へと発展させます。

3. 最後に統一 API を見る  
   ここまで来ると、なぜ複数モデルのシステムに統一された入口が自然に必要になるのかが、理解しやすくなります。

## この章で最初に押さえるべきこと

- デプロイは最後に足す外側の殻ではなく、モデルがシステムに入る正式な入口
- ローカル実行、サービス化、統一インターフェースは段階的な関係にある
- この章は、後のアプリ開発や工程化の安定性に直接影響する

## モデルエンジニアリングの詳説：いつも最強モデルを呼べばいいわけではない

現代の LLM アプリでは、性能、レイテンシ、コスト、プライバシー、デプロイの複雑さのバランスを取る必要があります。実際のシステムでは、小さなモデルで分類、言い換え、整形などの簡単なタスクを処理し、強いモデルで複雑な推論を行い、ローカルモデルでプライバシー性の高いデータを処理し、画像モデルで画像を扱い、そのうえで統一 API やモデルルーティングを使って、これらの能力を組み合わせることがあります。

![モデルサービス選定の意思決定図](/img/course/ch08-model-serving-decision-map-ja.png)

| 方向 | 解決する問題 | この章での注目点 |
|---|---|---|
| Small Language Models | 大規模モデルはコストが高く、遅延も大きい | どのタスクを小さなモデルに任せられるかを見極める |
| モデルルーティング | タスクによって必要なモデルが異なる | タスクの難しさ、コスト、プライバシーでモデルを選ぶ |
| 量子化 | ローカルデプロイではリソースが限られる | 精度、速度、VRAM のトレードオフを理解する |
| LoRA / QLoRA | 低コストでドメインタスクに適応したい | 微調整と RAG、Prompt の境界を知る |
| 蒸留 | 大規模モデルの能力を小さなモデルへ移したい | 教師モデル、生徒モデル、評価セットを理解する |
| 推論最適化 | リクエストが増えると遅延とコストが制御できなくなる | キャッシュ、バッチ処理、並列化、ストリーミング出力、レート制限を学ぶ |
| ハイブリッドデプロイ | クラウド、ローカル、エッジ環境のすべてに対応したい | 置き換え可能で監視しやすいモデルサービス入口を設計する |

この章のプロジェクト出口は、単に「モデルが動く」だけではありません。なぜそのモデルを選んだのか、1回の呼び出しにどれくらい費用がかかるのか、レイテンシは許容できるか、失敗時にどうフェイルオーバーするか、将来モデルを変えるときにアプリ層の大幅な変更が必要かどうかまで説明できることがゴールです。
## 初心者と上級学習者はどう読むべきか

初心者がこの章を初めて学ぶときは、まずローカル実行、推論サービス、統一 API という主線をつかみましょう。デプロイの細かい点を一度に全部理解する必要はありません。モデルがどう読み込まれ、どうサービス化され、アプリがどうやってインターフェースを通じて呼び出すのかを説明できれば、先へ進めます。

経験のある学習者は、この章を抜け漏れの確認と工程化の練習として使えます。境界条件、失敗ケース、評価方法、コードの再現性、そして前後のステージとのつながりに注目してください。読み終えたら、この章の内容を自分の作品の README や実験記録に残せると理想的です。

## 本章の小プロジェクト出口

この章を学び終えたら、最小の練習として、章の中で最も重要な概念またはツールを1つ選び、実行できて、スクリーンショットに残せて、README に書ける小さな成果を作るのがおすすめです。複雑である必要はありませんが、入力が何か、処理の流れが何か、出力結果が何かを説明できるようにしましょう。

## 合格基準

この章の終わりには、自分の言葉でこの章が何を解決するのか、前後の学習ステップとどうつながるのかを説明でき、かつ本章の小プロジェクト出口の最小版を完成させられるようになっているはずです。

さらに、よくあるエラーを1回、デバッグの流れを1回、または結果改善の記録を1回残せるなら、あなたはもう単に「読んだだけ」ではなく、この章を自分のプロジェクト経験として身につけ始めているということです。
