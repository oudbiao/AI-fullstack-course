# 8.1.1 予習ガイド：この章で学ぶ RAG は何か

この章で解決するのは、モデルの知識が古い、足りない、あるいは自分の業務資料に合っていないときに、外部知識をどう安定して回答の流れに組み込むか、という問題です。

RAG は単に「ベクトルデータベースをつなぐ」だけではありません。本当に学ぶのは、大規模言語モデルアプリを作るための考え方です。どの知識をモデルのパラメータに入れるべきか、どの知識を外部文書に置くべきか、ユーザーが質問したときにどう関連資料を検索するか、そしてどうやってモデルに“資料を根拠にして”答えさせるか、を考えます。

## この章がコース全体のどこにあるか

あなたはすでに第 7 章で大規模言語モデルの原理、Prompt、微調整、アラインメントを学びました。第 8 章では、コースは「モデルの能力を理解する」段階から「アプリケーションシステムを組み立てる」段階へ進みます。RAG は、そのアプリケーション主軸の最初の重要な土台です。

Prompt がモデルにタスクをよりよく理解させるためのものだとすると、微調整はモデルの振る舞い方を変えるものです。それに対して RAG は、モデルに対して更新可能で、追跡可能で、管理しやすい知識ベースを接続するもの、と考えるとわかりやすいです。

![RAG 大規模モデルアプリケーションにおける位置づけの橋渡し図](/img/course/ch08-rag-position-bridge-ja.png)

## この章で本当に解決したい問題

この章では、次の 5 つの問いに答えます。文書をなぜそのまま丸ごとモデルに入れてはいけないのか。分割、クリーニング、メタデータがなぜ検索品質に影響するのか。Embedding とベクトルデータベースは実際に何をしているのか。検索、再ランキング、フィルタリング、コンテキスト結合が最終回答にどう影響するのか。RAG システムの良し悪しをどう判断するのか。

初心者が RAG で一番誤解しやすいのは、「検索できれば、もう正しい回答ができる」と思ってしまうことです。実際には、RAG の結果は文書品質、分割粒度、検索の網羅性、再ランキングの精度、プロンプトの組み立て方、回答評価の仕方で大きく変わります。

## 初心者におすすめの学習順序

まずは RAG の最小閉ループを理解するのがおすすめです。ユーザーが質問すると、システムがまず資料を検索し、その資料と質問を一緒にモデルへ渡して回答させます。次に文書処理と分割を学び、知識がシステムに入る前にクリーニング、分割、メタデータ付与が必要だと理解します。そのあとで Embedding とベクトルデータベースを学び、類似度検索で関連部分が見つかる仕組みを押さえます。最後に検索最適化、再ランキング、評価を見るとよいです。これらが、Demo を実用プロダクトに変えられるかどうかを決めます。

![RAG コア章の学習順序図](/img/course/ch08-rag-core-chapter-flow-ja.png)

## この章でつかむべき主軸

この章の主軸は、ひとことで言うと「資料」を「検索できる知識」に変え、さらに「検索結果」を「引用できる回答」に変えることです。

![RAG 資料から回答へのパイプライン図](/img/course/ch08-rag-data-to-answer-pipeline-ja.png)

前半では、元の資料を検索可能な知識インデックスにします。後半では、ユーザーの質問に対して検索、再ランキング、回答、引用を行います。

この流れが理解できると、RAG プロジェクトの改善は Prompt だけ直せばよいわけではないとわかります。検索できないなら、文書解析、分割、検索の段階を確認します。検索はできるのに答えがずれるなら、再ランキング、コンテキストの組み立て、生成用 Prompt を確認します。答えはそれらしく見えるのに検証できないなら、出典引用と評価サンプルを追加します。

## この章と後続章の関係

RAG は、後の章で学ぶ Q&A アシスタント、個人知識ベース、企業知識ベース、Agent のツール呼び出しの土台です。モデルのデプロイ章では、モデルを安定して呼び出す方法を学び、アプリ開発章では RAG を対話、ファイルアップロード、API に組み込みます。さらに工程化の章では、ログ、エラー処理、評価、デプロイを補っていきます。

この章が不安定なままだと、後でよくある問題が起きます。知識ベース Demo は動くのに、少し複雑な質問をすると話がずれる。引用元が安定しない。ユーザーが答えのない質問をしたときに、モデルが無理に作り話をする。どこで失敗したのかが、検索なのか、モデルなのか、文書品質なのか分からない。こうした問題です。

## 初心者と上級学習者の読み方

初心者がこの章を初めて学ぶときは、まず主軸と最小の動く例に集中してください。すべての細部を一度で理解する必要はありません。この章が何を解決するのか、入力と出力は何か、最小プロジェクトはどう動かすのかを説明できれば、先に進んで大丈夫です。

経験のある学習者は、この章を知識の穴埋めと工程化の練習として使えます。境界条件、失敗例、評価方法、コードの再現性、前後の章とのつながりに注目してください。読み終えたら、本章の内容を自分の作品 README や実験記録に残すとよいです。

## 本章の小プロジェクトの出口

この章を学び終えたら、最小のコース知識ベース Q&A を作るのがおすすめです。3～5 本のコース文書を用意し、文書読み込み、分割、ベクトル化、検索、コンテキスト結合、モデル回答、出典表示までを完成させます。UI は複雑でなくて構いませんが、検索された断片と回答の出典を見せられる必要があります。

最小の提出物には、`docs/` 内の 3～5 本の Markdown 文書、`rag_demo.py` または notebook、少なくとも 10 個の固定テスト質問、検索 top-k 断片の出力結果、そして「質問 → ヒットした文書 → 回答が正しいか」をまとめた評価表を入れるのがおすすめです。

```python
query = "RAG になぜ出典引用が必要なのか？"
for i, chunk in enumerate(top_k_chunks, start=1):
    print(i, chunk["source"], chunk["text"][:80])
```

さらに発展させたいなら、小さな評価セットを追加できます。たとえば 10 個の質問を用意し、それぞれに期待するヒット文書と理想回答を付けておきます。そうすると、分割や検索戦略を調整する前後で何が変わったかを観察できます。

## 合格基準

この章の終わりには、RAG が文書から回答に至るまでの全体の流れを説明でき、chunk、embedding、ベクトルデータベース、検索、再ランキング、コンテキスト、引用がそれぞれ何をしているかを説明でき、RAG の性能が悪いときに、どの層で詰まっているかをある程度判断できるようになっているはずです。

出典の引用、検索断片の表示、簡単な評価サンプルを備えた最小の知識ベースアシスタントを作れれば、LLM アプリ開発と Agent の段階に進むための基礎条件は満たしています。
