# 段階学習タスクシート：LLM アプリ開発と RAG

この段階の目標は、大規模モデルを実際のアプリに接続できるようになることです。チャット欄で質問するだけでなく、API 呼び出し、プロンプト、文書処理、ベクトル検索、回答の引用、評価、エンジニアリングによるデプロイまで身につける必要があります。

## この段階で必ず完了するタスク

| タスク | 成果物 | 通過基準 |
| --- | --- | --- |
| LLM API 呼び出しを動かす | 最小の呼び出しスクリプト | key、リクエスト、レスポンス、例外、再試行を扱える |
| 文書処理を完成させる | 分割とクリーニングの記録 | chunk のサイズ、重なり、metadata、出典を説明できる |
| ベクトル検索を構築する | 最小の検索 Demo | 質問から関連文書の断片を検索できる |
| RAG QA を完成させる | コース Q&A プロトタイプ | 回答に出典の引用が含まれ、検索の根拠を説明できる |
| RAG 評価を完成させる | テストセットと評価表 | ヒット率、回答品質、引用品質、失敗サンプルを記録できる |
| 実践ワークショップを完了する | `rag_app_workshop.py` の出力と追加評価ケース 1 つ | 期待出力を再現し、各段階を説明し、安全に拡張できる |

## おすすめの学習順序

まずモデル API を動かし、そのあと文書と embedding を処理し、次にベクトル検索を作り、最後に検索結果と生成モデルをつなげます。最初から複雑なフレームワークを入れず、最小コードで RAG の入力と出力の境界を理解しましょう。

RAG の重要な点は「ベクトルデータベースをつないだこと」ではなく、「なぜこの文書が検索されたのか」「回答が本当に出典で支えられているのか」「どの質問は検索できないのか」「どの回答は hallucination の可能性があるのか」を説明できることです。

## AI 学習アシスタントプロジェクトとの関係

この段階は、AI 学習アシスタントの v0.8 コース Q&A アシスタントに対応します。コースの Markdown を読み込み、索引を作成し、学習者の質問に答え、引用元を示せる必要があります。このバージョンは、プロジェクトを「学習記録ツール」から「AI アシスタント」へと進化させる重要な節目です。

最小機能としては、コース文書の取り込み、見出しと本文での分割、metadata の保存、関連断片の検索、回答の生成、引用パスの表示、質問と回答の記録がおすすめです。標準版ではさらに、評価セット、失敗サンプル分析、設定可能なパラメータを追加します。

## よくあるつまずき

よくある問題には、文書の分割が細かすぎる、または大きすぎる、metadata が失われる、embedding モデルと言語が合わない、検索ではヒットするのに回答が出典を使わない、見た目は正しいが引用が裏付けていない、コンテキストが長すぎてコストと遅延が高くなる、などがあります。調査するときは、検索結果と生成された回答を分けて確認しましょう。


## やさしい版 / 標準版 / チャレンジ版のタスク

| 難易度 | やること | 向いている人 |
|---|---|---|
| やさしい版 | 出典付き回答を 5 個完成させる | 初めて学ぶ人、時間が少ない人、入門したばかりの学習者 |
| 標準版 | 10 個の評価問題と citation_ok の確認を完了する | この段階をポートフォリオに入れたい学習者 |
| チャレンジ版 | chunk、top-k、rerank 戦略ごとの失敗タイプを比較する | 基礎があり、より強いプロジェクト実績を作りたい学習者 |

## この段階のバッジと Boss 戦

| 種類 | 内容 |
|---|---|
| Boss 戦 | 引用 hallucination ドラゴン |
| 解放できるバッジ | RAG 引用警察、検索考古学者 |
| 最低通過スローガン | まず動かす、次に説明する、次に失敗を記録する |
| 証拠保存のおすすめ | スクリーンショット、ログ、失敗サンプル、評価表を `reports/`、`evals/`、`logs/` に保存する |

やさしい版を終えれば次へ進めます。標準版まで終えたら、ポートフォリオに書くのがおすすめです。チャレンジ版は余力があるときだけ取り組みましょう。

## 段階ポートフォリオの提出物

この段階の成果をポートフォリオに残したい場合は、少なくとも以下のファイル、または同等の資料を保存するのがおすすめです。

| 提出物 | 説明 |
| --- | --- |
| `chunks.jsonl` | 文書の切り分け結果。text、source、section、page、content_type などのフィールドを含む |
| `retrieval_logs.jsonl` | 各 query の top-k、score、source、命中テキストの要約 |
| `eval_questions.csv` | 固定の評価問題、標準回答、ヒットすべき文書、重要な引用 |
| `failure_cases.md` | 検索失敗、生成失敗、引用失敗、metadata 失敗などのサンプル |
| `rag_config.md` | chunk_size、overlap、top-k、rerank、prompt_version などの設定記録 |
| `rag_app_workshop_output.txt` | [5.6 実践：第 8 章 RAG アプリ完全ワークショップ](./ch05-projects/05-stage-hands-on-workshop.md) の実行出力と、自分が加えた小さな変更のメモ |
| `README.md` | 実行コマンド、入力出力例、評価結果、改善計画 |

これらのファイルは最初から完全でなくてもかまいませんが、RAG プロジェクトを「答えられる」ものから「説明できる、評価できる、振り返れる」ものへ変えてくれます。

## 段階通過の確認問題

学習を終えたら、次の問いに答えられるようになっているはずです。RAG は大規模モデルのどの制限を解決するのか、chunk サイズは何に影響するのか、embedding とキーワード検索の違いは何か、なぜ回答に引用が必要なのか、ある RAG の失敗が検索の問題か生成の問題かをどう見分けるのか。
