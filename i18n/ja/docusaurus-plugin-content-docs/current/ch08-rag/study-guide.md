# 学習ガイドとタスクシート：LLM アプリ開発と RAG をどう学ぶと一番混乱しにくいか

`8 LLM 应用开发与 RAG` に進んで、フレームワーク、データベース、デプロイ、アプリのロジックがごちゃ混ぜに感じたら、まずはシステムを階層で分けて考えましょう。LLM アプリは、ただ1つの API を呼ぶだけではなく、知識、モデル、アプリ、エンジニアリングをつなげるものです。

## この段階の基本原則

LLM アプリを最初に学ぶときは、4つの層をつかむことが大事です。知識層は資料をシステムに入れる役割、モデル層は生成と理解を担当、アプリ層は機能をまとめる役割、エンジニアリング層は安定して動かす役割です。

![LLM アプリ四層学習マップ](/img/course/ch08-study-guide-four-layer-map-ja.png)

## この段階で必ず完了するタスク

このタスクで RAG を実作業に落とし込みます。RAG プロジェクトは「答えが出る」だけでは完成ではありません。検索された根拠を確認でき、出典を引用でき、失敗を評価でき、安定して再実行できる状態を目指します。

| タスク | 成果物 | 通過基準 |
|---|---|---|
| LLM API 呼び出しを動かす | 最小の呼び出しスクリプト | key、リクエスト、レスポンス、例外、再試行を扱える |
| 文書処理を完成させる | 分割とクリーニングの記録 | chunk のサイズ、重なり、metadata、出典を説明できる |
| ベクトル検索を構築する | 最小の検索 Demo | 質問から関連文書の断片を検索できる |
| RAG QA を完成させる | コース Q&A プロトタイプ | 回答に出典の引用が含まれ、検索の根拠を説明できる |
| RAG 評価を完成させる | テストセットと評価表 | ヒット率、回答品質、引用品質、失敗サンプルを記録できる |
| 実践ワークショップを完了する | `rag_app_workshop.py` の出力と追加評価ケース 1 つ | 期待出力を再現し、各段階を説明し、安全に拡張できる |

## おすすめの学習順序

1回目はまず RAG を学びましょう。文書がどのように解析され、分割され、ベクトル化され、検索され、再ランキングされて、モデルのコンテキストに入るのかを理解します。

2回目はモデルデプロイと統一インターフェースを学びます。クラウド API、ローカルモデル、推論サービス、統一呼び出し層の違いを理解しましょう。

3回目はアプリ開発を学びます。LLM API、Function Calling、対話システム、文書解析、テンプレート生成、AI 支援コーディングなどを含みます。

4回目はエンジニアリングを学びます。非同期処理、API 設計、ログ、監視、エラー処理、Docker、デプロイは、最小限のアプリを動かせるようになってから少しずつ追加していきましょう。

5回目は総合プロジェクトに取り組み、ナレッジベース、モデル呼び出し、アプリ機能、エンジニアリングをつなげます。

## 選定の分岐: まず Prompt、RAG、微調整、それとも Agent？

初心者は 1 つの手法で何でも解決しようとしがちです。より安定した考え方は、まず「どのギャップを埋めたいのか」を見極めることです。

| 問題の種類 | 最初の選択肢 | 理由 |
|---|---|---|
| タスクを明確に伝えたい、または出力形式を制約したい | Prompt | モデルはすでにその仕事を知っており、よりよい指示が必要なだけだから |
| 最新の情報、引用可能な情報、組織固有の知識が必要 | RAG | 答えはモデルの記憶ではなく外部文書から取るべきだから |
| 多くの例で安定した振る舞い、文体、形式を保ちたい | 微調整 | パターンをモデル自身に学習させるべきだから |
| 継続的な目標駆動の行動、ツール呼び出し、状態更新が必要 | Agent | システムが各観測のあとで次の行動を決める必要があるから |

簡単な RAG の失敗診断の順番は次の通りです。

1. 文書は正しく解析され、分割されたか？
2. 検索は本当に適切な断片を見つけたか？
3. 再ランキングやコンテキストの詰め込みで証拠を落としていないか？
4. モデルは回答時に証拠を無視していないか？
5. 評価データは本当のユーザー課題を反映しているか？

この 5 つを見てもまだ曖昧なら、たいていは「RAG が壊れている」のではなく、まず Prompt 設計、データの整理、あるいは課題範囲の絞り込みで解くべき問題です。

![Prompt、RAG、微調整、Agent の選び方](/img/course/ch08-study-guide-method-choice-map-ja.png)

いちばん堅実なのは、まず最小の穴を埋める最小手段を使い、それで足りないときだけ段階を上げることです。

## おすすめの学習ペース

| 内容の種類 | 推奨時間 | 学習目標 |
|---|---|---|
| RAG 基礎 | 8～16 時間 | 文書から回答までの流れを動かせるようにする |
| モデルデプロイ | 4～8 時間 | モデルがどのように安定して呼び出されるかを理解する |
| アプリ開発 | 8～16 時間 | 対話、ツール、構造化出力をまとめられるようにする |
| エンジニアリング | 8～16 時間 | ログ、エラー処理、デプロイ説明を追加できるようにする |
| 総合プロジェクト | 16～32 時間 | 1つの見せられる LLM アプリを完成させる |

## 段階ごとのプロジェクトの進め方

最初のプロジェクトは、最小構成の RAG を作るのがおすすめです。いくつかの文書を用意して、分割、ベクトル化、検索を行い、モデルが資料に基づいて回答するようにします。

プロジェクトを広げる前に、[8.5 実践：第 8 章 RAG アプリ完全ワークショップ](./ch05-projects/05-stage-hands-on-workshop.md) を完了してください。概念の読解からポートフォリオ用ナレッジベースアシスタントへ進むための、推奨される実践橋渡しです。

2つ目のプロジェクトは、講座Q&Aアシスタントか個人ナレッジベースがおすすめです。出典の引用、検索結果の表示、フィードバック記録を追加します。

3つ目のプロジェクトは、企業ナレッジベースの Demo にするとよいです。複数ターンの対話、権限の想定、ログ、評価用サンプルを追加します。

## よくあるつまずき

もっともよくあるつまずきは、ベクトルデータベースをつなげれば RAG は完成したと思ってしまうことです。本当に効果を左右するのは、文書の品質、分割戦略、検索の再現率、再ランキング、Prompt の組み立て方、評価データです。

2つ目のつまずきは、最初からフレームワークを追いかけることです。まずは最小構成の RAG を手で書いてから、LangChain、LlamaIndex などのフレームワークを学ぶのがおすすめです。

3つ目のつまずきは、成功する Demo だけ作って、失敗を扱わないことです。実際のアプリでは、回答なし、検索できない、モデルのタイムアウト、出力形式の誤り、コストが高すぎる、といったケースを必ず考える必要があります。

## 段階ポートフォリオ成果物

![RAG 評価ループ図](/img/course/ch08-rag-evaluation-loop-map-v2-ja.png)

この段階の成果をポートフォリオに残したい場合は、少なくとも以下のファイル、または同等の資料を保存するのがおすすめです。

| 成果物 | 説明 |
|---|---|
| `chunks.jsonl` | 文書の切り分け結果。text、source、section、page、content_type などのフィールドを含む |
| `retrieval_logs.jsonl` | 各 query の top-k、score、source、命中テキストの要約 |
| `eval_questions.csv` | 固定の評価問題、標準回答、ヒットすべき文書、重要な引用 |
| `failure_cases.md` | 検索失敗、生成失敗、引用失敗、metadata 失敗などのサンプル |
| `rag_config.md` | chunk_size、overlap、top-k、rerank、prompt_version などの設定記録 |
| `rag_app_workshop_output.txt` | [8.5 実践：第 8 章 RAG アプリ完全ワークショップ](./ch05-projects/05-stage-hands-on-workshop.md) の実行出力と、自分が加えた小さな変更のメモ |
| `README.md` | 実行コマンド、入力出力例、評価結果、改善計画 |

これらのファイルは、RAG プロジェクトを「答えられる」ものから「説明できる、評価できる、振り返れる」ものへ変えてくれます。

## 段階通過質問

この段階を学び終えたら、1つの RAG アプリを自力で完成でき、データが文書から回答に至るまでの全体の流れを説明でき、効果が悪いときにどの層で問題が起きたかを切り分けられるようになっているはずです。

第 9 章へ進む前に、次の質問に答えられるか確認してください。

- RAG は大規模モデルのどの制限を解決するのか。
- chunk サイズは何に影響するのか。
- embedding 検索とキーワード検索の違いは何か。
- なぜ回答に引用が必要なのか。
- ある RAG の失敗が検索の問題か生成の問題かをどう見分けるのか。

実践ワークショップを動かし、文書と評価ケースを 1 つ追加し、そのうえで出典の引用、基本的なログ、簡単な評価サンプル付きのナレッジベースアシスタントを作れたら、AI Agent 段階に進んでよいでしょう。
