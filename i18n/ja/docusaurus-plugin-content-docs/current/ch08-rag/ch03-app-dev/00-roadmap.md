# 学習ガイド：アプリ開発のこの章では何を学ぶのか

この章で解決したいのは、「モデルの能力を、どうやって実際に使えるプロダクト機能として組み立てるか」です。

ここまでで、RAG がどのように知識を接続するか、モデルのデプロイがどうやって安定した呼び出し方法を提供するかを学びました。アプリ開発の章では、さらに一歩進んで、これらの能力をどうやってユーザーが使え、開発者が保守でき、システムが継続的に動く機能としてまとめるのかを考えます。

## この章がコース全体の中でどこにあるのか

第 8 章の主軸は、大規模モデルを「答えられる」状態から「アプリにできる」状態へ進めることです。RAG は知識のつながりを担当し、デプロイはモデルサービスを担当し、アプリ開発はモデル、知識、ツール、UI、業務フローをまとめる役割を担います。

ここでの重要な変化は、単に 1 回だけモデルを呼び出すのではなく、ひとつの完全な対話プロセスを設計することです。ユーザー入力はどこから来るのか、システムはどう意図を理解するのか、ツールを呼ぶ必要があるのか、複数ターンの文脈をどう保存するのか、出力をフロントエンドやバックエンドでどう再利用するのか。こうした点が、アプリ層で扱うべき問題です。

![LLM アプリ開発章の関係図](/img/course/ch08-app-dev-chapter-flow-ja.png)

## この章で本当に解決したい問題

この章では、次の 5 つの問いに答えます。LLM API をどう信頼性高く呼び出し、タイムアウト、再試行、コスト、エラーをどう扱うか。アプリが複雑になったとき、なぜフレームワークや抽象層が必要になるのか。Function Calling はどうやってモデル出力をシステムの動作につなげるのか。複数ターンの対話でどう文脈と状態を維持するのか。文書解析、テンプレート生成、コードアシスタントのような複雑な場面を、どう保守しやすいモジュールに分解するのか。

初心者が最も誤解しやすいのは、「LLM アプリ開発とは、入力欄にモデル API をつなぐだけ」と考えてしまうことです。実際のプロダクトでは、モデルはあくまで一層にすぎません。入力検証、文脈管理、権限、ログ、例外処理、構造化出力、ユーザーフィードバック、効果測定も必要になります。

## 初心者におすすめの学習順序

まずは API 呼び出しの実践を見て、最小の呼び出し経路、パラメータ、エラー処理、コスト意識を身につけましょう。次にフレームワーク抽象を学び、機能が複雑になったときに、prompt、モデル、ツール、検索、記憶、出力解析をなぜコンポーネントに分けるのかを理解します。その後で Function Calling を学びます。これは「テキストを生成する」段階から「動作を起こす」段階へ進むための重要な鍵です。最後に対話システムと複雑なアプリ場面を学び、複数ターンの状態、文書処理、テンプレート生成をつなげていきます。

![LLM アプリ開発の学習順序図](/img/course/ch08-app-dev-learning-order-map-ja.png)

## この章を学ぶときに押さえるべき主線

この章の主線は、ひとつのモデル呼び出しを、保守しやすいアプリの閉ループへと発展させることだとまとめられます。

![LLM アプリ能力の閉ループ図](/img/course/ch08-llm-app-capability-loop-ja.png)

前半ではまずモデルの能力をインターフェースや機能モジュールとしてまとめ、後半で対話状態、ツール呼び出し、文書処理、エンジニアリングとしての納品に進みます。

この流れを押さえると、アプリ開発で何が重要かを判断しやすくなります。すべての場面で複雑なフレームワークが必要なわけではありませんが、使えるプロダクトには必ず、入力、処理、出力、フィードバックの閉ループが必要です。

## この章と次の章との関係

アプリ開発は、RAG と Agent のあいだをつなぐ橋です。RAG によってシステムは資料を調べられるようになり、Function Calling によってシステムは動作を起こせるようになり、複数ターンの対話によってシステムは継続的にやり取りできるようになります。これらの能力が組み合わさると、自然に Agent の目標、計画、ツール、記憶、実行の閉ループへつながっていきます。

この章をしっかり学ばないと、後でよくある問題として、Agent を作る前にアプリ層がすでにぐちゃぐちゃになっている、ツール呼び出しの結果を検証していない、対話状態が積み重なって乱れる、モデル出力の形式が不安定でバックエンドの解析に失敗する、Demo の成功だけを見ていて異常系をどう処理するか分からない、といったことが起きます。

## 初心者と上級学習者はどう読むか

初心者がこの章を初めて学ぶときは、まず主線と最小の実行例をつかみましょう。すべての細部を一度で理解する必要はありません。この章が何を解決するのか、入力と出力は何か、最小プロジェクトをどう動かすのかを説明できれば、先へ進んで大丈夫です。

経験のある学習者は、この章を「抜け漏れの確認」と「エンジニアリング練習」として使えます。境界条件、失敗ケース、評価方法、コードの再現性、そして前後の段階とのつながりに注目しましょう。読み終えたら、本章の内容を自分の作品 README や実験記録にまとめておくとよいです。

## 学習時間と難易度の目安

| 学習方法 | 目安の時間 | 目標 |
|---|---|---|
| ざっと読む | 20～30 分 | この章が何を解決するのかを理解し、後でどこで使うかを知る |
| 最小通過 | 1～2 時間 | 最小例を動かして、本章の小さなゴールまで完了する |
| 深く練習 | 半日～1 日 | エラー分析、比較実験、またはプロジェクト README の記録を追加する |

## 本章の自己チェック問題

| 自己チェック問題 | 合格基準 |

## 本章の小プロジェクト出口

この章を学び終えたら、「コース Q&A と学習プランニングアシスタント」を作るのがおすすめです。ユーザーの質問を受け取り、それが概念の説明なのか、学習ルートの提案なのか、プロジェクトの相談なのか、資料検索なのかを判断します。必要なら知識ベース検索を呼び出し、最後に構造化された提案を出し、ユーザーフィードバックを記録します。

このプロジェクトは小さくてもかまいませんが、アプリの閉ループを示す必要があります。つまり、API 呼び出し、Prompt の組み立て、必要に応じたツール呼び出し、複数ターンの文脈、構造化出力、ログ記録、そして簡単なエラー処理です。

## 合格ライン

この章の終わりには、独立して LLM API 呼び出しモジュールをまとめられ、Function Calling がモデルとツールをどうつなぐかを説明でき、基本的な複数ターン対話の状態構造を設計でき、RAG、Prompt、ツール呼び出しを組み合わせて小さなアプリを構成できるようになっているはずです。

モデル呼び出しの失敗、出力形式の誤り、検索結果が空、ツールエラーといった異常系まで考えられるなら、あなたはもう大規模モデルアプリのエンジニアリング思考を身につけ始めていると言えます。
