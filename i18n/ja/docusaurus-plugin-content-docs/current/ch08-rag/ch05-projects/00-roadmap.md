# 学習ガイド：総合プロジェクトのこの章はどう学べばいいのか

この章は、ただコンポーネントを足し続けるのではなく、これまで学んだ知識層、モデル層、アプリ層、エンジニアリング層を本当にシステムとして形にする章です。

第 8 章の LLM アプリ開発と RAG のプロジェクトは、第 7 章の大規模モデルの原理、Prompt、ファインチューニングとは違います。第 7 章はモデルの能力、Prompt、ファインチューニング、そして方式の選び方により重点があります。一方で第 8 章は、LLM アプリがどう資料に接続し、モデルを呼び出し、機能を組み立て、エラーを処理し、ログを残し、デモできて改善可能なプロトタイプにしていくかに重点があります。

## この章がコース全体の中でどこにあるのか

あなたはすでに RAG、モデルデプロイ、アプリ開発、エンジニアリング化を学んできました。総合プロジェクトはこの段階の出口であり、単にモデルを呼び出せるだけではなく、文書、検索、対話、ツール、構造化出力、引用、評価、デプロイ説明をひとつの流れとして組み立てられることを証明する場です。

![LLM アプリ総合プロジェクトのロードマップ](/img/course/ch08-projects-route-map.png)

## この章で本当に解決したい問題

この章では、次の 5 つの問いに答えます。資料をどう検索可能なナレッジベースに変えるのか。モデルが根拠に基づいて答え、思いつきで話さないようにするにはどうするのか。RAG、Function Calling、マルチターン対話、構造化出力をどうアプリ機能としてまとめるのか。検索結果が空、モデルのタイムアウト、形式エラー、回答不能の問題をどう処理するのか。ログ、引用、評価サンプルを使って、システムが信頼できるとどう示すのか。

初心者が最もやりがちなミスは、プロジェクトを「ベクターデータベースをつなげれば完成」と考えてしまうことです。本当の LLM アプリプロジェクトでは、各層を説明できなければなりません。資料はどこから来るのか、どう分割するのか、どう検索するのか、どう prompt に入れるのか、モデルはどう出力するのか、システムはどう検証するのか、ユーザーはどうソースを見るのか、開発者はどう効果を評価するのか。そこまで理解してはじめて、ひとつのアプリになります。

## 初心者におすすめの学習順序

まずは企業ナレッジベース QA か講義ナレッジベース QA から始めるのがおすすめです。これは RAG のメインパスを鍛えるのに最も向いています。次にスマート QA アシスタントを作り、検索、会話状態、ツール呼び出しを製品機能としてつなげます。続いて RAG + ファインチューニングの総合システムを作って、知識強化と挙動適応がどう組み合わさるのかを理解します。最後に、ナレッジベース駆動の教材生成アシスタントを作り、文書解析、例題抽出、構造化出力、テンプレート化生成を、より完成度の高いアプリシーンにまとめます。

![LLM アプリプロジェクトの学習順序図](/img/course/ch08-project-learning-order-map.png)

## この章を学ぶときに押さえるべき主線

この章の主線は、次のようにまとめられます。LLM アプリプロジェクトは、一度生成して終わりではなく、知識、モデル、機能、エンジニアリングの閉ループです。

![LLM アプリプロジェクトの納品クローズドループ図](/img/course/ch08-project-delivery-loop.png)

この流れが分かると、プロジェクトの発表では最終回答だけを見せればよいわけではないと分かります。検索された断片、ソースの引用、失敗時の処理、ログのサンプル、評価用の問い、今後の改善案も見せるべきです。

## 各プロジェクトで何を練習するのか

| プロジェクト | 本当に鍛えること |
|---|---|
| 企業ナレッジベース QA | 検索、権限、引用、追跡可能な回答 |
| RAG + ファインチューニングの総合システム | 知識強化と挙動適応の組み合わせ方 |
| スマート QA アシスタント | 検索、会話状態、ツール呼び出しを一連の製品フローにすること |
| ナレッジベース駆動の教材生成アシスタント | 文書解析、例題抽出、構造化出力、テンプレート化された文書生成 |

## この章と次の段階の関係

この章は AI Agent 段階への前段出口です。安定した LLM アプリには、すでに知識、モデル、ツール、状態、エンジニアリングの原型が含まれています。Agent はその上に、目標駆動、複数ステップの計画、ツールの観察、記憶、安全性評価をさらに加えていきます。

この章が不安定なままだと、よくある問題は次のようになります。Agent を始める前に、RAG とアプリ層がまだ不安定。ツール呼び出しに検証がない。回答不能の問題を処理できない。システムにログと評価がない。デモは答えられるように見えても、答えの出どころを説明できない。

## 初心者と上級学習者はどう読むか

初心者がこの章を初めて学ぶときは、まず主線と最小実行例をつかみましょう。最初から細部を全部理解する必要はありません。この章が何を解決するのか、入力と出力は何か、最小プロジェクトはどう動くのかを説明できれば、先へ進めます。

経験者は、この章を弱点補強とエンジニアリング練習として使えます。境界条件、失敗ケース、評価方法、コードの再現性、そして前後の段階とのつながりに注目しましょう。読み終えたら、本章の内容を自分の作品 README や実験記録に残すのがおすすめです。

## 学習時間と難易度の目安

| 学び方 | おすすめ投入時間 | 目標 |
|---|---|---|
| ざっと読む | 20～30 分 | この章が何を解決するのかを理解し、後でどこで使うかを知る |
| 最小突破 | 1～2 時間 | 最小例を動かし、この章の小プロジェクト出口を終える |
| しっかり練習 | 半日～1 日 | エラー分析、比較実験、またはプロジェクト README の記録を補う |

## 本章の自己チェック問題

| 自己チェック問題 | 合格基準 |

## 本章の小プロジェクト出口

この章を学び終えたら、少なくとも 1 つ「ソース引用付きのナレッジベースアシスタント」を完成させるのがおすすめです。プロジェクトには、文書取り込み、分割、ベクトル化、検索、コンテキスト結合、モデル回答、ソース表示、回答不能処理、簡単な評価セットを含める必要があります。

上級版では、マルチターン対話、ユーザーフィードバック、Function Calling、文書解析、テンプレート出力を追加できます。作品集版では、アーキテクチャ図、重要なコードの説明、評価サンプル、失敗ケース、デプロイ説明を補足するのがよいでしょう。

## プロジェクトの納品物テンプレート

この章のプロジェクトは、「質問に答えられる Demo」を 1 つ出すだけにしないでください。作品を本物の LLM アプリらしく見せるために、少なくとも次の納品物を含めるのがおすすめです。

| 納品物 | 説明 |
|---|---|
| プロジェクト説明 | 対象ユーザー、知識範囲、コア課題、やらないことを説明する |
| アーキテクチャ図 | 文書解析、切り分け、検索、モデル呼び出し、引用、ログの流れを示す |
| ナレッジベースのサンプル | 元資料、切り分け結果、metadata、ソースフィールドを示す |
| 検索ログ | 典型質問に対して、どの断片がヒットしたか、スコア、順位を示す |
| 回答の引用 | 最終回答には参考ソースを表示し、ただの生成文だけにしない |
| 失敗ケース | 少なくとも 3 つの誤例を挙げ、切り分け、検索、コンテキスト、生成のどれが原因かを判断する |
| 評価サンプル | 最適化前後の効果比較に使う固定テスト問題を用意する |
| デプロイ説明 | どう実行するか、どの環境変数が必要か、どう Demo を再現するかを説明する |

このテンプレートの目的は、あなたのプロジェクトを他人が理解、再現、評価できるようにすることです。あなたの PC で偶然 1 回動いた、という状態で終わらせないためです。

## Debug 事件簿

| 事件 | 内容 |
|---|---|
| 事件名 | RAG が証拠を見つけられない |
| 現場 | 問題は明らかに文書にあるのに、システムは的外れな答えを返すか、引用が答えを支えられていない。 |
| 捜査手順 | まず生成を切って、検索結果だけを出力する。chunk、metadata、top-k、query を確認する。 |
| 事件の証拠 | retrieval_logs.jsonl、citation_check.csv、失敗タイプ統計。 |

プロジェクト練習では、成功したスクリーンショットだけを残さないでください。少なくとも 1 つは本当の失敗サンプルを選び、「現象、手がかり、容疑理由、捜査手順、修復アクション、回帰確認」の形で `reports/failure_cases.md` に書いておくと、プロジェクトがより本物のエンジニアリング作品に近づきます。

## プロジェクト納品物の基準

各総合プロジェクトは、コードを動かすだけではなく、同じ作品集基準で納品するのがおすすめです。最小納品物には、README、再現可能な実行コマンド、入力出力の例、重要フロー図、1 つの失敗サンプル分析、そして次の改善計画を含めるべきです。

| 納品物 | 最低要件 | 上級要件 |
|---|---|---|
| README | プロジェクト目標、実行方法、依存関係、例を明記する | アーキテクチャ図、設計上の取捨選択、振り返りを追加する |
| 入出力例 | 少なくとも 1 つの完全なケースを残す | 成功、失敗、境界ケースを残す |
| 評価記録 | どの指標で効果を判断するかを明記する | baseline、比較実験、エラー分析を加える |
| エンジニアリング記録 | 環境または API の問題を 1 回記録する | ログ、コスト、時間、トラブルシュート過程を記録する |
| 発表資料 | 動くことを示すスクリーンショットまたは短い GIF | 説明可能な作品集ページにする |

プロジェクトで最も大事なのは、機能をどれだけ詰め込むかではなく、何を解決したのか、システムはどう動くのか、効果をどう判断するのか、失敗したときどう切り分けるのか、次の版でどう改善するのかを、きちんと説明できることです。

## 合格基準

この章の終わりには、あなたは 1 つの RAG または LLM アプリプロジェクトを独立して完成できるはずです。資料が文書から回答になるまでの完全な流れを説明でき、検索が空、モデル出力の形式エラー、ソースなし回答などのよくある失敗を処理でき、ログと評価サンプルでシステム効果を説明できる必要があります。

ソース引用、基本ログ、エラー処理、評価サンプル、デプロイ説明を備えたナレッジベースアシスタントを作れれば、LLM アプリ開発段階の作品集出口基準に達しています。

## バージョンの進め方

| バージョン | 目標 | 納品の重点 |
|---|---|---|
| 基礎版 | 最小閉ループを動かす | 入力できる、処理できる、出力できる、そして一組の例を残す |
| 標準版 | 見せられるプロジェクトにする | 設定、ログ、エラー処理、README、スクリーンショットを追加する |
| 挑戦版 | 作品集品質に近づける | 評価、比較実験、失敗サンプル分析、次のロードマップを追加する |

まずは基礎版を完成させるのがおすすめです。最初から大きくて何でもあるものを目指さないでください。バージョンを 1 段上げるたびに、「何を新しくできるようになったか、どう検証したか、まだどんな問題があるか」を README に書き足しましょう。
