# 8 LLM アプリ開発と RAG

![LLM アプリと RAG のメインビジュアル](/img/course/ch08-rag-engineering-ja.png)

このフェーズで解決するのは、「大規模モデルを実際のシステムにどうつなぐか」です。モデル呼び出しから始まり、ドキュメント処理、ナレッジベース、RAG、ツール呼び出し、対話システム、デプロイ、ログ、そして工程化へと進みます。

## ストーリーで導入：大規模モデルに自分の資料庫をつなぐ

汎用の大規模モデルと会話するだけなら、そのモデルが知っているのは学習時に覚えた公開知識です。会社の文書、講座資料、製品マニュアル、個人ノートに答えてほしいなら、外部資料を接続する必要があります。RAG はモデルに資料係をつけるようなものです。まず資料を探し、その資料に基づいて答えさせ、最後に引用とログを残して、どこから答えが来たのか確認できるようにします。

## 体験練習：まずは「うまく答えられない質問」から改善する

RAG を作るときは、わざとシステムがうまく答えられない質問を 1 つ選び、原因を追ってみましょう。元の文書に答えがないのか、分割のせいで重要情報がバラバラになったのか、検索で取り戻せなかったのか、Prompt が引用を求めていないのか、それともモデルが文脈に従わなかったのか。原因を 1 つ見つけるたびに、あなたは実際の LLM アプリ開発のデバッグ方法を学んでいます。

## 初心者向けの最小クリア経路

初心者はまず、最小の RAG を手作業で実装しましょう。文書を用意し、テキストを分割し、ベクトルを生成し、片を検索し、Prompt を組み立て、出典付きの回答を生成します。RAG の精度が悪い原因が、文書・分割・検索・指示・モデルのどれにあるのかを切り分けられたら、最小クリア達成です。

## 上級者向けの深掘り経路

経験のある学習者は、リランキング、複数経路の召回、クエリ書き換え、評価セット構築、ログ監視、権限制御、デプロイを深掘りできます。さらに RAG を API や小型アプリとしてまとめ、フィードバック収集とコスト集計も追加してみましょう。

## 初学者は何を先にやるべきか、上級者は何を進めるべきか

このフェーズを初めて学ぶ人は、最初から複雑なフレームワークに飛びつかないでください。より安定した順番は、まず数十行のコードで「文書 → 分割 → ベクトル → 検索 → 回答 → 引用」の最小ループを動かし、その後で、より信頼できるコンポーネントに少しずつ置き換えることです。各ステップで何が起きているかを自分の目で見られるようになって初めて、その後になぜベクトルデータベース、リランキング、ログ、評価が必要なのかが分かります。

経験者は、工程品質に重点を置けます。検索に失敗したときどう劣化運転に切り替えるか、引用の信頼性をどう担保するか、ログでどう振り返るか、評価セットをどう作るか、インターフェースをどうデプロイするか、コストをどう記録するか、です。目標は「答えられる」Demo を作ることではなく、長く使われ、問題が起きたときに原因を特定できるナレッジベースシステムを作ることです。

## なぜ RAG とアプリ工程が必要なのか

大規模モデルは、あなたのプライベート文書に自動ではアクセスできませんし、知識が常に最新とも限りません。RAG は外部知識の検索結果をモデルのコンテキストに入れ、資料に基づいて答えさせます。アプリ工程は、モデルの能力を製品に接続する役割です。ユーザー入力の処理、モデル呼び出し、状態管理、ログ記録、コスト制御、効果測定を担います。

## 現代 RAG の詳説：答えられるだけでなく、保守できるようにする

2025～2026 年の RAG は、もはや「ベクトルデータベース + Prompt」だけではありません。実際のナレッジベースでは、キーワードの取りこぼし、断片の並び順の不安定さ、文書更新の古さ、引用の信頼性不足、コストや遅延の制御不能といった問題に直面します。そのため、このフェーズでは RAG を長期運用する工程システムとして学びます。

| 技術方向 | 解決する問題 | 学習の重点 |
|---|---|---|
| Hybrid Search | 純粋なベクトル検索では、正確なキーワード、番号、用語、人名を取りこぼしやすい | キーワード検索とベクトル検索を組み合わせ、結果を統合して並べ替える |
| Reranking | 最初の召回は多いが、順位が安定しない | リランキングモデルやルールで、最も関連性の高い断片を前に持ってくる |
| Query Rewrite | ユーザーの質問が短すぎる、口語的すぎる、または文脈が欠けている | 検索しやすいクエリに書き換える |
| Multi-query Retrieval | 1 つのクエリだけではカバーしきれない | 複数の観点からクエリを生成し、召回率を上げる |
| GraphRAG | 答えが複数文書にまたがるエンティティや関係に依存する | エンティティ関係を抽出し、グラフ構造を軸にコンテキストを組み立てる |
| Agentic RAG | 1 回の検索では足りず、調べながら判断する必要がある | システムが、追加検索・クエリ変更・停止を自分で判断する |
| Multimodal RAG | 知識ソースに PDF、スクリーンショット、表、画像が含まれる | 文書解析、視覚理解、テキスト検索を組み合わせる |

これらの技術を学ぶときは、全部を積み上げる必要がある機能だと思わないでください。各技術は、「なぜ普通の RAG では足りないのか」という失敗問題に答えるべきです。改善しているのは、召回、順位付け、コンテキスト構成、引用の信頼性、それとも運用更新でしょうか。

## RAGOps の詳説：ナレッジベースを公開したあと、どう継続改善するか

RAGOps は、RAG システムを公開したあとの品質維持に注目します。合格ラインの RAG プロジェクトなら、少なくとも次の情報を見られる必要があります。文書ソース、分割方法、インデックスのバージョン、召回された断片、リランキングのスコア、回答の引用、無回答時の処理、ユーザーフィードバック、Token コスト、応答時間、失敗ログです。

![RAGOps 継続改善ループ図](/img/course/ch08-ragops-improvement-loop-ja.png)

最小の RAGOps は最初から複雑である必要はありませんが、固定の評価セットは必須です。たとえば 20～50 個の講座質問を用意し、期待する命中文書、理想回答、そして「ここでは捏造してはいけない」という境界をラベル付けします。Prompt、分割戦略、Embedding モデル、リランキング方法を変えるたびに、同じ問題セットで比較し、感覚ではなくデータで判断しましょう。

## このフェーズの学習ルート

第 1 章では RAG を学びます。文書解析、分割、Embedding、ベクトルデータベース、検索戦略、RAG 最適化、評価を理解します。

第 2 章では LLM のローカルデプロイと統一インターフェースを学びます。ローカルモデル、推論サービス、統一 API の意味を理解します。

第 3 章では LLM API、LangChain の基礎、Function Calling、対話システム、文書解析、AI 支援コーディング、テンプレート文書生成などの大規模モデルアプリ開発を学びます。

第 4 章では、非同期プログラミング、API 設計、ログ監視、Docker デプロイなどの工程化実践を学びます。

第 5 章では総合プロジェクトに進み、ナレッジベース、モデル呼び出し、アプリインターフェース、工程化を組み合わせます。

## 学び終えたらできるようになること

- 基本的な RAG フローを設計できる
- 文書解析、分割、Embedding、ベクトル検索を実装できる
- RAG の精度が悪いとき、それが文書・分割・検索・指示・モデルのどこに問題があるか判断できる
- LLM API を呼び出し、アプリインターフェースとしてまとめられる

- LLM アプリにログ、エラー処理、基本的なデプロイ説明を追加できる

## よくある誤解

「ベクトルデータベースをつなげれば RAG 完了」と思わないでください。実際に効果を左右するのは、文書品質、分割戦略、検索召回、リランキング、指示の組み立て、引用の信頼性、評価方法です。

最初から複雑なフレームワークに依存するのも避けましょう。フレームワークは効率を上げますが、前提として土台の流れを理解している必要があります。まずは最小の RAG を手作業で実装し、そのあとで LangChain や LlamaIndex などのフレームワークを学ぶのがおすすめです。

## RAG エラー劇場：答えが悪いとき、たいていどこで詰まるのか

RAG の答えが正確でないときは、すぐにモデルを変えないでください。まず元の文書に答えがあるかを確認し、次に分割で重要情報が切り離されていないかを見ます。その後、検索結果が関連断片を拾えているか、Prompt がソースに基づいて答えるよう求めているかを確認し、最後にモデルが文脈に従っていないのかを判断します。

## RAG 実行可能な小実験：フレームワークなしでも検索の流れが分かる

このフェーズを学ぶときは、まず最小で動く実験を作るのがおすすめです。最初から LangChain や複雑なベクトルストアに接続しなくて構いません。5～10 個の講座テキストを用意し、キーワードの重なりや簡単な Embedding で検索を模擬しながら、各ステップの結果を出力しましょう。つまり、ユーザー質問、書き換え後のクエリ、召回された断片、並び順のスコア、最終 Prompt、回答の出典です。

```python
import re

questions = ["RAG プロジェクトにはなぜ評価セットが必要なの？"]
docs = [
    {"id": "ragops", "text": "RAGOps は文書ソース、検索断片、引用、コスト、失敗ログを記録する必要がある。"},
    {"id": "agentops", "text": "AgentOps は実行トレース、ツール権限、失敗復旧、人的確認に注目する。"},
]

def tokenize(text):
    return re.findall(r"[\w\u4e00-\u9fff\u3040-\u30ff]+", text.lower())

query = questions[0]
hits = sorted(
    docs,
    key=lambda d: len(set(tokenize(query)) & set(tokenize(d["text"]))),
    reverse=True,
)

for hit in hits[:2]:
    print(hit["id"], hit["text"])
```

この実験のポイントは、アルゴリズムの強さではありません。学習者が初めて「検索結果は確認できる」と実感できることです。最小ループが動いたら、そこからベクトルモデル、Hybrid Search、Reranking、Query Rewrite、評価セットに置き換えていきます。

## RAG 失敗ケース集：現象から問題を特定する

| 現象 | よくある原因 | 特定方法 | 修正の方向 |
|---|---|---|---|
| 答えは流暢だがソースがない | Prompt で引用を強制していない、またはコンテキストにソース ID が残っていない | 最終 Prompt と召回断片を出力する | 断片内に文書名、ページ番号、段落 ID を残す |
| 文書に答えがあるのに検索できない | 分割しすぎ、キーワード不足、またはベクトル召回がずれている | 元文書をキーワード検索し、その後で切片内容を見る | chunk を調整し、Hybrid Search か Query Rewrite を追加する |
| 召回断片は多いが並び順が不正確 | 最初の召回は「候補を見つける」だけだから | top-k スコアと手動ラベルの関連性を出力する | Reranking やルールベースのフィルタを追加する |
| 文書更新後も答えが古い | インデックスのバージョンが更新されていない | 文書バージョンとインデックス時刻を記録する | インデックス再構築、期限切れフラグ、回帰評価を追加する |
| 最適化後に良くなったか分からない | 固定の問題集がない | 同じ問題セットで改善前後を比較する | 評価セットと失敗サンプル表を作る |



| 振り返り質問 | あなたが答えられるべきこと |
|---|---|
| 資料ソース | システムの回答はどの文書、Web ページ、データベースに依存している？ |
| 知識処理 | 文書はどのように解析、クリーニング、分割され、インデックスに書き込まれる？ |
| 検索品質 | ユーザーの質問はどの断片に命中し、順位とスコアは妥当か？ |
| 答えの引用 | ユーザーは答えがどのソースに基づくか確認できるか？ |
| エラー処理 | 検索結果が空、モデルがタイムアウト、形式エラーのとき、システムはどう処理する？ |
| 評価の改善 | 最適化前後を比較するための固定問題集とログはあるか？ |

このフェーズの本当の出口は、ソース、ログ、エラー処理、評価サンプル付きのナレッジベースアシスタントを作ることです。モデル呼び出しを 1 回成功させるだけではありません。

## フェーズ評価 Rubric

| レベル | 評価基準 | 作品集の証拠 |

| 基礎合格 | 文書読み込み、分割、検索、回答、ソース表示までできる | 実行画面、サンプル質問、召回断片 |

面接や作品集で見せるときは、「RAG を作りました」で終わらせないでください。より良い言い方は、まず最小の RAG ループを作り、次にある種の質問で召回が不安定だと気づき、評価セットを追加し、top-k 断片を出力し、分割とリランキングを比較し、最後にソースを示して失敗理由を記録できるようにした、という流れです。

## フェーズプロジェクト

基本版は、ローカル文書に基づいて質問に答え、ソースも出せる個人ナレッジベース Q&A アシスタントを作ることです。標準版では、文書前処理、ベクトルインデックス、検索評価、ログ記録、簡単な Web API を追加します。チャレンジ版では、企業ナレッジベース Demo を作り、権限、フィードバック、リランキング、マルチターン対話、オンラインデプロイの説明を加えます。

もっと細かな学習リズムを見たいなら、[学習ガイド：大規模モデルアプリと工程化をどう学ぶと迷いにくいか](./study-guide.md) を読んでみてください。

## AI 学習アシスタントの一貫プロジェクトとの関係

このフェーズは、AI 学習アシスタント v0.8 に対応できます。つまり、講座 Markdown を読み込み、検索、回答、ソース引用、評価問題集に対応することです。一貫プロジェクトの流れで学んでいるなら、このフェーズの最後に少なくとも 1 回はバージョン記録を残すのがおすすめです。今回追加した機能、実行方法、入力と出力の例、起きた問題、次にどう直すか、を記録しましょう。

## フェーズクリア基準

| クリア段階 | 必要なこと |
|---|---|
| 最低クリア | 講座 Q&A またはナレッジベースアシスタントを構築し、検索、引用、評価ができる。 |
| 推奨クリア | このフェーズの少なくとも 1 つの実行可能な小プロジェクトを完成させ、README に実行方法、入出力例、問題点を記録する。 |
| 作品集クリア | このフェーズの成果を「AI 学習アシスタント」一貫プロジェクトに接続し、スクリーンショット、ログ、評価サンプル、次の計画を残す。 |

このフェーズを学び終えたら、細部を全部暗記する必要はありません。より大切なのは、このフェーズが何の問題を解決するのか、前のフェーズとの関係は何か、そして今後の学習をどう支えるのかを説明できることです。次のフェーズでは、システムが Q&A から、ツールを呼び出せる Agent へと進化します。
