# 12 AIGC とマルチモーダル

![AIGC とマルチモーダルのメインビジュアル](/img/course/ch12-multimodal-aigc.png)

このフェーズで扱うのは、「入力と出力が文字だけではなくなったとき、AI システムはどう広がるのか」です。これは方向性の拡張にあたり、LLM アプリケーション、RAG、Agent を理解したあとに、画像、音声、動画、そして異なるモダリティをまたぐ生成へ進むのに適しています。

## ストーリーで導入：AI を文字の世界から現実世界へ

AI が文字しか扱えないとき、それはとても文章が上手で推論も得意なアシスタントのようです。AI が画像、音声、動画、テキストを同時に理解できるようになると、現実世界の複雑な入出力に近づいていきます。AIGC とマルチモーダルは、単に「見栄えのよいコンテンツを作る」だけではなく、理解、生成、編集、ワークフローをつなげて、新しい創作システムを作ることです。

## 学習チャレンジマップ

![マルチモーダルと AIGC の学習チャレンジマップ](/img/course/ch12-learning-quest-map.png)

## 体験練習：ひとつのアイデアをマルチモーダルのパイプラインに分解する

たとえば「AI コースの宣伝短編動画を作る」をテーマにしてみましょう。まず文章を作り、次にイメージ画像を生成し、ナレーションを設計し、動画の絵コンテを考え、最後にレビューと出力を追加します。マルチモーダルのプロジェクトで大事なのは、どのモデルを使うかだけではなく、複数の生成ステップを制御可能な流れとして組み立てることだと分かります。

## プロジェクトのおまけ

このフェーズのボーナス作品としては、「AI クリエイティブワーク台」を作ることができます。テーマを入力すると、タイトル、本文、画像プロンプト、絵コンテ、音声原稿を自動生成し、さらに人手での修正とレビューの工程も用意します。コース全体の締めくくりとして、とてもよい発表プロジェクトになります。

## 初心者向けの最小クリアルート

初心者はまず、テキスト、画像、音声、動画がどのように同じ AI ワークフローに入るのか、そして画像生成とマルチモーダル QA の基本フローを理解しましょう。簡単な画像生成かマルチモーダル QA の Demo を 1 つ完成させ、プロンプト、モデル、後処理がそれぞれどんな役割を持つのか説明できれば、最小クリアです。

## 発展ルート

経験のある学習者は、拡散モデル、画像と言葉のアライメント、動画生成、音声合成、デジタルヒューマン、著作権コンプライアンス、コンテンツ安全性を深く学べます。さらに、複数の生成ステップを組み合わせて、レビュー可能、編集可能、出力可能なクリエイティブプロダクトのワークフローにしてみましょう。

## マルチモーダルが重要な理由

現実世界は純粋なテキストだけではありません。人間は文字、画像、音声、動画、動作を通して世界を理解します。マルチモーダルモデルは、こうした異なる形式の情報をひとつの統一システムにまとめ、AI が画像を見て、文書を読み、音声を聞き、画像を生成し、創作を支援できるようにすることを目指します。

![マルチモーダルシステムの主幹図](/img/course/ch12-multimodal-system-backbone.png)

前半では、まずマルチモーダル入力と生成能力を理解し、後半でコンテンツ制作、レビュー、出力、プロダクトプロトタイプへとつなげます。

## 初学者は何を先にやるか、上級者は何をするか

初めてこのフェーズを学ぶ人は、まずマルチモーダルを「異なる入出力形式が同じ製品フローに入るもの」と捉えましょう。画像理解、プロンプト生成、コンテンツパッケージ出力のような小さなプロジェクトから始めて、最初から長い動画生成の流れを追いかけないようにします。

経験のある学習者は、プロダクト化とリスク制御に重点を置けます。素材の出どころ、バージョン管理、人による編集、著作権・肖像権、レビュー項目、出力フォーマットなどです。目標は、生成能力を実際に納品できる創作ツールへ組み立てることです。

## マルチモーダル応用の詳説：画像・テキスト能力から実際のワークフローへ

2025～2026 年のマルチモーダル応用は、単なる「画像を見て答える」や「画像を生成する」だけではありません。よりよくある実務上の課題は、スクリーンショット、PDF、表、グラフをどう読み取るか、視覚理解を RAG にどう接続するか、Agent に画像や文書を見て行動させるにはどうするか、文章、画像、音声、動画の生成をどう組み合わせて編集可能、レビュー可能、出力可能な流れにするか、という点です。

| 詳説テーマ | 解決する問題 | 学習重点 |
|---|---|---|
| Vision-Language Models | モデルが画像と文字を同時に理解する | 画像 QA、グラフ説明、スクリーンショット理解、視覚推論 |
| Document AI | PDF、表、スキャン、スライドをそのままテキストフローに入れにくい | レイアウト解析、OCR、表抽出、構造化出力 |
| Multimodal RAG | ナレッジベースが Markdown と純テキストだけではない | 画像、PDF ページ、グラフ、テキスト断片を統一して検索・参照する |
| Multimodal Agent | Agent がスクリーンショット、ページ、文書に基づいて行動する必要がある | 視覚入力を観察し、ツールを選び、追跡可能な軌跡を生成する |
| Video Generation Workflow | 動画生成は 1 回の Prompt では終わらない | 文章、絵コンテ、キャラクター、ショット、ナレーション、字幕、レビューのパイプライン |
| Realtime Voice | 音声対話では遅延、割り込み、連続文脈が重要 | 音声入力、リアルタイム応答、文字起こし、合成、対話状態 |
| Content Safety | 生成コンテンツには著作権、肖像、事実、センシティブなリスクがある | 素材の許諾、人手レビュー、リスクラベル、出力制限 |

マルチモーダルを学ぶときは、「モデルの能力」を「製品の流れ」に変換して考えましょう。たとえば、コース資料アシスタントは Markdown だけを読むのではなく、スライドのスクリーンショット、PDF の表、図解も読む必要があります。クリエイティブワーク台は、1 枚の画像を生成するだけでなく、ユーザーが文章を修正し、バージョンを選び、リスクを確認し、最終ファイルを出力できるようにする必要があります。

## マルチモーダル RAG とマルチモーダル Agent を主線につなぐには

マルチモーダルは、コースの主線の外にある孤立した島ではありません。第 8 章の RAG はマルチモーダル RAG に広がり、第 9 章の Agent はスクリーンショットを見たり、文書を読んだり、グラフを扱えたりするマルチモーダル Agent に広がり、第 12 章ではそれらの能力をひとつの創作または理解のワークフローにまとめます。

![マルチモーダル RAG と Agent のブリッジ図](/img/course/ch12-multimodal-rag-agent-bridge.png)

締めくくりにふさわしいプロジェクトは「マルチモーダル学習アシスタント」です。これは、コースのスクリーンショットや PDF ページを読み取り、重要な概念を抽出し、コース文書と組み合わせて RAG で回答し、さらに振り返りカードや学習計画を生成できます。これにより、RAG、Agent、マルチモーダル、そして実装力を総合的に示せます。

## このフェーズの学習ルート

## 学び終えたらできるようになること

- テキスト、画像、音声、動画がマルチモーダルシステムでどう役割分担するか説明できる
- 拡散モデルと Stable Diffusion の基本ワークフローを理解できる
- 簡単な画像生成またはマルチモーダル QA Demo を構築できる

## よくある誤解

AIGC を「楽しい画像ツール」だとだけ考えないでください。本当のマルチモーダル製品には、アセット管理、プロンプトエンジニアリング、モデル選定、後処理、ユーザーワークフロー、コンプライアンスの境界が関わります。

また、毎回新しいモデルを追いかける必要もありません。変化は速いですが、より大事なのは、安定した主線である「表現、アライメント、生成、編集、評価、ワークフロー」をつかむことです。

## AIGC エラー劇場：見た目が良いことと納品できることは別

生成結果が要件に合わない場合は、まず入力制約、スタイル参照、ネガティブ要件がはっきりしているかを確認しましょう。何度も生成して比較しにくい場合は、バージョン記録が足りないということです。作品を外部公開するなら、著作権、肖像権、センシティブ内容、出典説明、出力形式も確認する必要があります。

## マルチモーダルの実行可能な小実験：まずは画像や PDF を構造化レコードにする

マルチモーダルプロジェクトの最小実験は、動画生成から始めなくてもかまいません。まずは「1 枚のスクリーンショットを構造化情報に変える」ことから始めましょう。学習者は、コースのスクリーンショット、エラー画面のスクリーンショット、または PDF ページを 1 つ選び、モデル出力を手で模擬して、データ構造を先に設計します。画像の中に何があるか、重要な情報は何か、不確かな点はどこか、RAG や Agent の次のステップに入れられるか、という観点です。

```python
visual_record = {
    "source": "course-slide-01.png",
    "content_type": "コースのスクリーンショット",
    "visible_text": ["RAGOps", "評価セット", "Trace", "コスト監視"],
    "objects": ["フローチャート", "表"],
    "uncertainty": ["右下の小さな文字が不鮮明"],
    "next_step": "マルチモーダル RAG インデックスに書き込み、コース Q&A アシスタントが参照できるようにする",
}

print(visual_record)
```

この実験で大事なのは、マルチモーダルエンジニアリングの意識を作ることです。モデルが画像を見たあと、その結果は確認でき、参照でき、編集でき、後続フローに入れられる必要があります。構造化レコードの設計ができたら、実際の Vision モデル、OCR、PDF 解析、またはマルチモーダル API に置き換えましょう。

## マルチモーダル失敗事例集：生成の前に入力と納品を管理する

| 現象 | よくある原因 | 切り分け方法 | 修正方向 |
|---|---|---|---|
| 画像理解で重要情報を見落とす | スクリーンショットの解像度が低い、文字が小さい、レイアウトが複雑 | 元画像の鮮明さと OCR 結果を確認する | 入力品質を上げる、範囲を切り出す、不確かな部分を残す |
| PDF の表の解析が乱れる | レイアウトが複雑、またはページをまたいでいる | 元の PDF、解析テキスト、構造化結果を比較する | ページごとに処理し、表を個別に抽出し、ページ番号の出典を残す |
| 生成画像はきれいだが要件に合わない | スタイル、サイズ、用途、ネガティブ要件が不明確 | 要件チェックリストで出力を確認する | Prompt テンプレート、参考画像、人手による選別を追加する |
| 動画生成がつながらない | 絵コンテとキャラクターの一貫性設計がない | 各ショットの文章、画面、切り替えを確認する | 先に絵コンテを書き、それから区切って生成・レビューする |
| 作品を外部公開できない | 著作権、肖像権、センシティブ内容、事実リスクを確認していない | レビューリストで項目ごとに確認する | 素材の出どころ、人による確認、出力制限を追加する |
