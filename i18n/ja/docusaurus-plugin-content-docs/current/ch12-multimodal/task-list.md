# ステージ学習タスク一覧：AIGC とマルチモーダル

このステージの目標は、テキスト、画像、音声、動画、ドキュメントなどのさまざまな入出力をまとめて、提出できる AI アプリケーションにすることです。大事なのは、1回の生成結果がどれだけすごいかではなく、入力の質、バージョン管理、手動レビュー、著作権の境界、最終的な提出フローが明確かどうかです。

## このステージで必ず完了するタスク

| タスク | 成果物 | 通過基準 |
| --- | --- | --- |
| マルチモーダルの入出力を理解する | マルチモーダルの処理フロー図 1 枚 | テキスト、画像、音声、動画、ドキュメントがどのようにシステムへ入るか説明できる |
| 画像理解または画像生成を動かす | 最小 Demo 1 つ | 入力、Prompt、出力、手動選別の結果を記録できる |
| ドキュメントまたはスクリーンショット理解を完成する | マルチモーダル解析サンプル 1 つ | 元のページ番号、領域、またはスクリーンショットの証拠を残せる |
| レビューと提出フローを追加する | レビュー用チェックリスト 1 つ | 著作権、肖像、センシティブ内容、事実リスクを確認できる |
| ステージプロジェクトを完成する | マルチモーダル作品またはクリエイティブ作業台 1 つ | 入出力、バージョン管理、レビュー、エクスポート、README がある |

## おすすめの学習順序

まずマルチモーダルシステムの入出力の境界を理解し、そのあとで画像理解、画像生成、動画/音声生成、マルチモーダルアプリケーションを学びましょう。生成結果を見た目の良し悪しだけで考えず、要件、素材、Prompt、バージョン、手動選別、提出形式を記録することが大切です。

マルチモーダルプロジェクトでは、出典と権限に特に注意が必要です。外部の画像、人物の肖像、音楽、動画クリップ、PDF の内容は、著作権、プライバシー、安全の境界に関わることがあります。作品集には、使用範囲とレビュー方法を書いておきましょう。

## AI 学習アシスタントプロジェクトとの関係

このステージは、AI 学習アシスタント v1.0 に対応できます。講義資料のスクリーンショット、PDF、音声メモ、図表資料を理解し、復習カード、図表つき要約、またはプレゼン素材を生成します。卒業プロジェクトの発表レイヤーとして使うこともできます。

最小機能としておすすめなのは、授業のスクリーンショットまたは PDF の 1 ページを入力し、重要情報を抽出して構造化された要約を生成し、さらに出典引用と手動確認の記録を残すことです。

## よくあるつまずき

よくある問題には、画像がぼやけていて誤って理解してしまう、PDF の表の解析が崩れる、生成した画像が用途に合わない、動画の絵コンテがつながらない、素材の出典が不明、生成結果をエクスポートまたは再利用できない、などがあります。調査するときは、まず元の入力品質、Prompt のバージョン、生成パラメータ、手動選別、レビュー記録を確認しましょう。


## ライト版 / 標準版 / チャレンジ版タスク

| 難易度 | やること | 向いている人 |
|---|---|---|
| ライト版 | 素材から出力までの 1 例を完成させる | 初回学習、時間が少ない人、または学び始めたばかりの学習者 |
| 標準版 | 素材の出典、生成過程、手動レビューを記録する | このステージを作品集に入れたい学習者 |
| チャレンジ版 | 成功例、失敗例、境界例を比較して、出力制限を書く | 基礎があり、より強いプロジェクト証拠を作りたい学習者 |

## このステージのバッジと Boss 戦

| 種類 | 内容 |
|---|---|
| Boss 戦 | マルチモーダル混沌体 |
| 解放できるバッジ | マルチモーダルレビュー担当、素材管理者 |
| 最小クリアの合言葉 | まず動かす、次に説明する、最後に失敗を記録する |
| 証拠保存のおすすめ | スクリーンショット、ログ、失敗サンプル、または評価表を `reports/`、`evals/`、または `logs/` に保存する |

ライト版を終えたら次へ進んでよいです。標準版を終えたら、作品集に載せることをおすすめします。チャレンジ版は、余力があるときだけ取り組みましょう。

## ステージ作品集の提出物

このステージの成果を作品集に残したい場合は、少なくとも次のファイル、または同等の資料を残すことをおすすめします。

| 提出物 | 説明 |
| --- | --- |
| `multimodal_pipeline.md` | 入力、解析、生成、レビュー、エクスポートのシステムフロー |
| `prompts/` | マルチモーダル Prompt、参考画像、ネガティブプロンプト、バージョン記録 |
| `outputs/` | 生成結果、候補バージョン、手動選別の理由、最終提出ファイル |
| `safety_review.md` | 著作権、肖像、センシティブ内容、事実性、使用範囲の確認 |
| `README.md` | プロジェクト目標、実行方法、素材の出典、サンプル出力、制限の説明 |

これらの資料があると、マルチモーダルプロジェクトは「見た目の良い結果を 1 つ作る」から、「要件、出典、バージョン、レビュー、提出の境界を説明できる完成度の高い作品」へとレベルアップします。

## ステージクリア問題

学習後、次の問いに答えられるようになっているはずです。マルチモーダル入力でなぜ出典を残す必要があるのか、画像生成でなぜ Prompt とバージョンを記録するのか、PDF/スクリーンショット理解でなぜ証拠を確認するのか、コンテンツ生成でなぜ手動レビューが必要なのか、どの場面でマルチモーダル機能を RAG や Agent に接続するのが適しているのか。
