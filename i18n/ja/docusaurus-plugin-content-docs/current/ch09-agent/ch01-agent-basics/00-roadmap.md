# 学習ガイド：Agent 基礎のこの章では何を学ぶのか

この章で解決したいのは、まず Agent の境界、能力、そしてシステム構造をはっきりさせることです。

多くの初学者は、初めて Agent を学ぶときに、チャットボット、ワークフロー、RAG、ツール呼び出し、マルチ Agent を全部まとめて考えてしまいます。これだと、フレームワークや Demo に引っ張られて学びやすくなってしまいます。Agent 基礎章の役割は、まず最小の閉ループを作ることです。つまり、目標、計画、行動、観察、修正を経て、最後に結果を出す流れをつかむことです。

## この章がコース全体の中で占める位置

ここまでで、あなたは LLM アプリ開発と RAG を学び、大規模モデルが文書、API、対話、コンテンツ生成につながることを知っているはずです。Agent の段階では、コースは「モデルを使うアプリ」から「目標に向かって継続的に行動できるシステム」へと進みます。

この段階での大きな変化は、普通の LLM アプリがたいてい「ユーザーが1回聞いて、システムが1回答える」形なのに対して、Agent は目標駆動、状態管理、ツール呼び出し、結果の観察、複数ステップの実行をより重視することです。

![Agent 基礎位置ブリッジ図](/img/course/ch09-basics-position-bridge.png)

## この章で本当に解決する問題

この章では、次の5つの疑問に答えます。Agent と普通のチャットボットの違いは何か。Agent と固定ワークフローの違いは何か。目標、状態、ツール、記憶、計画はそれぞれ何を担当するのか。なぜ単一 Agent はまず安定して作るべきで、そのあとでマルチ Agent を考えるのか。ひとつの Agent システムはどうやって過程を記録し、失敗を処理し、無限ループを避けるのか。

初学者が最もやりがちなミスは、いきなり LangGraph、CrewAI、AutoGen、あるいはマルチ Agent 協調を追いかけてしまい、単一 Agent の実行閉ループを先に理解しないことです。フレームワークは開発効率を上げてくれますが、システムの境界を代わりに理解してはくれません。

## 初学者におすすめの学習順序

まず「Agent とは何か」を読み、Agent、チャットボット、RAG アプリ、ツール呼び出しシステム、固定ワークフローの境界をはっきり分けましょう。次に発展の流れを見て、なぜ Agent が LLM の後であらためて注目されるようになったのかを理解します。続いて能力レベルを見て、「答えられる」「検索できる」「ツールを呼び出せる」「計画できる」「記憶を使える」「協調できる」を同じ能力の軸に並べてみます。そのあとシステム構造を見て、目標、状態、ツール、記憶、プランナー、エグゼキュータがどう組み合わさるのかを理解します。最後に「TD-Gammon から AlphaGo へ」を読み、強化学習における行動、フィードバック、計画と、現代の Agent をつなげましょう。

![Agent 基礎章の学習順序図](/img/course/ch09-basics-chapter-flow.png)

## この章で押さえるべき主線

この章の主線はこうまとめられます。Agent はモデル名ではなく、目標を中心にモデル、ツール、状態、フィードバックを組み立てるシステムの作り方です。

![単一 Agent 実行閉ループ図](/img/course/ch09-basics-execution-loop.png)

この閉ループは、そのシステムが本当に Agent かどうかを見分ける助けになります。もしシステムがモデルを1回固定で呼び出すだけなら、それは普通の LLM アプリに近いです。もしシステムが目標に応じてステップを分解し、ツールを呼び出し、結果を観察し、計画を修正し、必要なら実行を続けられるなら、それは Agent に近づいています。

## この章と後続章の関係

この章は、第9章「AI Agent と智能体システム」の入口です。後ろの推理と計画では、Agent が次の一手をどう決めるかを展開します。ツール呼び出しでは外部能力とどうつながるかを説明し、記憶システムでは文脈と経験をどう保存するかを説明します。MCP ではツールのエコシステムとプロトコルを扱い、マルチ Agent では複数の役割がどう協調するかを見ます。評価、安全性、デプロイでは、Agent をどう信頼できる形で運用するかを学びます。

この章をしっかり理解していないと、後でよくある問題にぶつかります。たとえば、マルチ Agent は単一 Agent より必ず強いと思ってしまうこと。成功した経路だけを見せて、ツール失敗やループ実行を処理しないこと。記憶を「多ければ多いほど賢い」と考えてしまい、本当は記憶がタスクに奉仕すべきだと気づかないこと。たくさんフレームワークを学んだのに、なぜそのような設計なのかがわからないことです。

## 初学者と発展学習者はどう読むか

初めてこの章を学ぶ人は、まず主線と最小実行例をつかんでください。すべての細部を一度に理解する必要はありません。この章が何を解決するのか、入力と出力は何か、最小プロジェクトをどう動かすのかを説明できれば、先へ進んで大丈夫です。

経験のある学習者は、この章を穴埋めと実装練習として読むとよいです。境界条件、失敗事例、評価方法、コードの再現性、そして前後の段階とのつながりに注目しましょう。読み終えたら、本章の内容を自分の作品の README や実験記録にまとめておくのがおすすめです。

## 学習時間と難易度の目安

| 学習方法 | 推奨時間 | 目標 |
|---|---|---|
| ざっと読む | 20～30 分 | この章が何を解決するのかを理解し、後でどこで使うかを知る |
| 最小通過 | 1～2 時間 | 最小例を動かし、本章の小プロジェクトの出口まで到達する |
| 深掘り練習 | 半日～1 日 | エラー分析、比較実験、またはプロジェクト README の記録を追加する |

## 本章のセルフチェック問題

| セルフチェック問題 | 合格基準 |
|---|---|
| この章は何を解決するのか？ | コース全体の中での位置を一言で説明できる |
| 最小の入力と出力は何か？ | 例に何が必要で、どんな結果が出るかを説明できる |
| よくある失敗点はどこか？ | エラー、精度の低さ、理解のずれの原因を少なくとも1つ挙げられる |
| 学んだあと何を残せるか？ | 本章の成果をプロジェクト README、実験記録、または作品集に書ける |

## 本章の小プロジェクトの出口

この章を学び終えたら、最小のリサーチアシスタント Agent を作るのがおすすめです。ユーザーがテーマを入力すると、Agent がまず問題を分解し、次に資料検索やツール呼び出しをするかどうかを選び、そのあと観察結果をまとめ、最後にステップ記録付きの短い要約を出力します。

このプロジェクトで大事なのは、検索結果をどれだけ豊富にするかではありません。実行過程をきちんと記録することです。つまり、目標は何か、計画は何か、どのツールを呼び出したか、何を観察したか、いつ続行を決めたか、いつ終了を決めたかを明確に残すことです。

## 合格基準

この章の終わりには、Agent とチャットボット、RAG アプリ、固定ワークフローの違いを説明でき、目標、状態、ツール、記憶、計画、観察が Agent の中で果たす役割を説明でき、単一 Agent の実行閉ループを描けるようになっているはずです。

単一 Agent の Demo を作り、各ステップのツール呼び出し、観察結果、失敗処理、最終出力を記録できれば、次の推理、ツール、記憶、MCP、マルチ Agent の各章に進んで大丈夫です。
