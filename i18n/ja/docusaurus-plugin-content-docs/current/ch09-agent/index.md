# 9 AI Agent と知能体システム

![AI Agent システムのメインビジュアル](/img/course/ch09-agent-systems-ja.png)

この段階で解決するのは、「どうすれば AI にただ答えさせるだけでなく、目標に向かってタスクを実行できるようにするか」です。Agent は、大規模モデル、ツール、記憶、計画、評価、システムエンジニアリングを組み合わせて、継続的に行動できる AI システムを作ります。

## 物語で導入：チャット相手からタスクの相棒へ

普通のチャットボットは、机の前に座って質問に答える人のようなものです。一方で Agent は、道具を使い、資料を確認し、タスクを分解し、手順を実行し、最後に結果を見直せる相棒のような存在です。単に「話す」だけでなく、「行動する」。1 回答えるだけでなく、目標に向かって継続的に前進します。

## 体験練習：まず「Agent が必要か」を判断する

AI アプリのアイデアに出会ったら、まず 3 つの質問をします。それは、複数ステップのタスクか、中間結果に応じて進路を調整する必要があるか、外部ツールや長期記憶を呼び出す必要があるか、です。3 つとも「いいえ」なら、普通のワークフローや RAG のほうが安定するかもしれません。多くが「はい」なら、Agent アーキテクチャを検討します。

## プロジェクト特典

この段階の特典作品は、「リサーチアシスタント Agent」です。あるテーマを質問に分解し、検索やナレッジベースのツールを呼び出し、証拠を整理してレポートを作成し、どの手順が成功し、どの手順が失敗したかを記録します。このプロジェクトでは、これまで学んだ Prompt、RAG、ツール呼び出し、ログ、評価を 1 つの完成したシステムにつなげます。

## 初心者向けの最小クリア路线

初心者はまず、Agent の目標、状態、計画、ツール、観察、記憶を理解します。いきなり複雑なフレームワークを積み上げる必要はありません。タスクを分解し、1～2 個のツールを呼び出し、実行過程を記録して結果を出せる最小限の Agent が作れれば、最小クリアです。

## 上級者向けの深掘り路线

経験のある学習者は、ReAct、Plan-and-Execute、記憶エンジニアリング、MCP、マルチ Agent 協調、評価、安全性、そして本番デプロイを深く学べます。さらに、固定ワークフロー、RAG、Agent を同じタスクで比較し、信頼性、コスト、失敗パターンを比べてみましょう。

## この段階の 2 通りの読み方

初心者はこの段階を読むとき、フレームワーク名や難しい論文を追いすぎないようにしましょう。1 回目は「目標 → 計画 → ツール → 観察 → 調整 → 記録」という実行の閉ループだけをつかみ、まずは小さなタスクを完了できて、各ステップの軌跡が見え、失敗したら止まれる最小 Agent を作ります。

経験のある学習者は、境界と信頼性に重点を置けます。いつ Agent を使わないのか、ツール権限をどう制限するのか、複数ステップのタスクをどう評価するのか、記憶が汚染されないようにするにはどうするのか、失敗時にどう復旧するのか、です。同じタスクを固定ワークフロー、RAG、Agent でそれぞれ実装し、安定性、コスト、説明しやすさ、保守の難しさを比較できます。

## Agent と普通の LLM アプリは何が違うのか

普通の LLM アプリは、一般的に固定フローです。ユーザーが入力し、システムがコンテキストを組み立て、モデルが答えを出します。Agent はそれよりも、目標、状態、行動を強く重視します。次に何をするかを判断し、ツールを選び、結果を読み、コンテキストを更新し、必要なら再計画します。

![Agent と普通のアプリの実行メインライン図](/img/course/ch09-agent-vs-workflow-backbone-ja.png)

前半のポイントは、「何をしたいか」を「次にどうするか」に変えることです。目標、計画、ツール選択が曖昧だと、Agent は後半で簡単に方向を外してしまいます。

## AgentOps 重点解説：Agent を追跡可能・制御可能・復旧可能にする

2025～2026 年の Agent 学習の重点は、すでに「モデルにツールを呼ばせる」ことから「知能体システムを追跡可能・制御可能・評価可能にする」ことへ移っています。信頼できる Agent は、成功結果だけを見せるべきではありません。目標は何か、なぜそのツールを選んだのか、どのパラメータを渡したのか、何を観察したのか、いくらコストがかかったのか、失敗したときにどう復旧するのか、いつ人間の確認が必要なのか、まで説明できる必要があります。

| 重点テーマ | 解決する問題 | 学習ポイント |
|---|---|---|
| MCP | ツール、ファイル、データベース、業務システムの接続方法が統一されていない | モデルと外部ツールをつなぐ標準的な方法を理解する |
| Tool Schema | モデルがツールのパラメータ、制約、エラーの意味を知らない | 明確なパラメータ、戻り値、エラーメッセージ、検証ルールを設計する |
| Agentic Workflow | 完全に自由な Agent は不安定で、固定フローは柔軟性が足りない | 確定した手順とモデルの判断を組み合わせる |
| Human-in-the-loop | 高リスク操作をデフォルトで自動実行してはいけない | 削除、送信、注文、DB 変更、公開などは人間確認が必要 |
| Agent Observability | 複数ステップの実行が失敗した後に振り返れない | 計画、アクション、観察、コスト、エラー、最終結果を記録する |
| Agent Evaluation | 1 回の成功デモだけでは不十分 | 固定タスクセットで完了率、ステップ数、ツールエラー、権限逸脱を評価する |
| Multi-agent Orchestration | 複数ロールの協働は、待ち合いや重複作業が起きやすい | 役割、通信方法、停止条件、最終責任者を明確にする |

## 制御可能な Agent の最小エンジニアリング閉ループ

Agent の核心は、「モデルを自由に発揮させる」ことではなく、自由度を境界の中に閉じ込めることです。最小限の制御可能な Agent は、まず次のように制限できます。ホワイトリストのツールだけを呼び出せる、指定したディレクトリや資料だけを読める、草案だけを出力して直接公開はできない、高リスク操作はユーザー確認が必要、各ステップは実行トレースに記録される、という形です。

![AgentOps の制御可能実行閉ループ図](/img/course/ch09-agentops-control-loop-ja.png)

プロジェクトは、「リサーチアシスタント Agent」から始めるとよいでしょう。問題を分解し、資料を検索し、要約を作り、トレースを記録できますが、メール送信、ファイル削除、データベース変更は自動で行わないようにします。こうすることで、Agent の能力を示しつつ、エンジニアリング上の境界も表現できます。

## この段階の学習ルート

第 1 章では Agent の基礎概念を学び、Agent とチャットボットの違い、発展の歴史、能力の段階、システムアーキテクチャを理解します。

第 2 章では推論と計画を学びます。Chain-of-Thought、ReAct、Plan-and-Execute、推論評価を含みます。

第 3 章ではツール利用と Function Calling を学びます。ツール説明、パラメータ設計、呼び出し戦略、安全境界、コード実行型 Agent を理解します。

第 4 章では記憶システムを学びます。短期記憶、長期記憶、エピソード記憶、手続き記憶、記憶エンジニアリングを扱います。

第 5 章では MCP を学び、モデルと外部ツールのエコシステムがどのようにプロトコルで接続されるかを理解します。

第 6 章と第 7 章では、LangGraph、LlamaIndex、CrewAI、AutoGen などの Agent フレームワークとマルチ Agent システムを学びます。

第 8 章から第 10 章では、評価、安全性、デプロイ、統合プロジェクトを学びます。

## 学習後にできるようになること

- Agent の目標、状態、ツール、記憶、計画の構造を説明できる
- ReAct または Plan-and-Execute 風の実行フローを設計できる
- ツール呼び出しのために、明確なパラメータと安全境界を設計できる
- そのタスクが本当に Agent を必要とするのか、それとも普通のワークフローや RAG で十分か判断できる
- 最小限で使えるリサーチアシスタントやデータ分析 Agent を構築できる
- Agent の評価、コスト、権限、失敗復旧を考慮できる

## よくある誤解

Agent を「モデルにツールを付けるだけ」と考えないでください。ツールはその一部にすぎません。本当に難しいのは、タスク境界、コンテキスト管理、エラー復旧、権限制御、結果評価です。

また、すべてのタスクに Agent を使う必要もありません。固定フローで、ルールが明確で、リスクが高いタスクは、従来のワークフローのほうが向いていることがあります。Agent は、オープンな問題、複数ステップの探索、動的にツールを呼び出す必要がある場面に向いています。

## Agent エラー劇場：行動できることと信頼できることは別

Agent がループで止まるなら、まず停止条件が明確かを確認します。ツール呼び出しが間違うなら、まず schema、パラメータ検証、権限境界を確認します。結果が成功して見えても振り返れないなら、実行トレースとログが不足しています。タスクのリスクが高いなら、Agent にすべて自動でやらせるのではなく、人間確認を追加すべきです。

## Agent の実行可能ミニ実験：まずトレースを出力する

Agent 学習の最小実験には、複雑なフレームワークは必ずしも必要ありません。まずは 2 つのツールだけを持つスクリプトを書けます。1 つは講座資料を検索するツール、もう 1 つは ToDo リストを作るツールです。ツール呼び出しごとに `thought`、`action`、`input`、`observation`、`cost_estimate` を記録します。これで、Agent は魔法ではなく、確認できる手順の連なりだと学習者に見えます。

```python
trace = []

def call_tool(name, tool_input):
    if name == "search_docs":
        return "RAGOps、AgentOps、評価セットに関する資料が見つかりました"
    if name == "make_todo":
        return "3 つの復習タスクを生成しました"
    return "ツールが存在しません"

step = {
    "thought": "ユーザーは RAG の復習準備をしたいので、まず講座資料を確認する",
    "action": "search_docs",
    "input": {"query": "RAGOps 評価 ログ"},
}
step["observation"] = call_tool(step["action"], step["input"])
trace.append(step)

for item in trace:
    print(item)
```

この実験の合格基準は、結果が完璧でなくても、なぜ各ステップが起きたかを振り返れることです。その後で、ツール schema、権限確認、失敗リトライ、MCP、マルチ Agent を追加していきます。

## Agent 失敗ケース集：まず境界を制御し、それから賢さを追う

| 現象 | よくある原因 | 切り分け方法 | 修正方針 |
|---|---|---|---|
| Agent がずっとループする | 目標と停止条件が不明確 | 各ステップの thought と observation が繰り返されていないか確認する | 最大ステップ数を制限し、完了条件と停止条件を追加する |
| ツールパラメータがよく間違う | schema が曖昧で、パラメータ検証がない | モデルが生成した生のパラメータとツールのエラーを記録する | フィールド型、例、デフォルト値、エラーメッセージを明確にする |
| 完了したように見えるが振り返れない | 実行トレースがない | 各ステップに計画、アクション、観察、結果の記録を求める | trace ログとタスク要約を追加する |
| 呼ぶべきでないツールを呼んだ | 権限境界が不明確 | ツールのホワイトリストとリスクレベルを確認する | 高リスクツールは人間確認、機密ツールはデフォルトで無効化する |
| 複数 Agent が互いに待ってしまう | 役割責任と最終責任者が不明確 | メッセージの流れとタスク引き継ぎ点を見る | 役割、成果物、タイムアウト、仲裁者を明確にする |



| 振り返り質問 | あなたが答えられるべきこと |
|---|---|
| 目標境界 | Agent は何を完了し、何を担当せず、いつ止まるのか？ |
| 計画実行 | どのようにタスクを分解し、次の一手を選び、結果に応じて進路を調整するのか？ |
| ツール呼び出し | ツール schema、パラメータ検証、失敗リトライ、権限境界は明確か？ |
| 記憶状態 | 現在状態、長期的な好み、過去の経験は階層的に管理されているか？ |
| 失敗復旧 | ツール失敗、検索失敗、結果が信用できないとき、システムはどう段階的に落とすのか？ |
| 評価と安全 | 固定タスクセット、コスト見積もり、安全境界、人間による引き継ぎ方法はあるか？ |

この段階の本当のゴールは、1 回だけ成功する会話を見せることではなく、追跡可能で、振り返り可能で、評価可能な Agent プロジェクトを作ることです。

面接や作品集の展示では、1 回の成功会話だけを見せないほうがよいです。より良い説明はこうです。私はタスクを目標、計画、ツール、観察、停止条件に分解し、ツール権限を制限し、各ステップの trace を記録し、固定タスクセットで Agent がループ、越権、誤ったツール呼び出しをしないか確認しました。

## 段階プロジェクト

基礎版は、テーマを質問に分解し、資料ツールを呼び出して構造化要約を生成できるリサーチアシスタントの実装です。標準版では、実行ログ、失敗リトライ、結果の自己チェック、簡単な記憶を追加する必要があります。チャレンジ版では、データ分析 Agent またはマルチ Agent 開発チームを作り、権限境界、コスト管理、評価サンプル、復旧メカニズムを組み込みます。

大きなプロジェクトを選ぶ前に、まず具体的な baseline を動かしたい場合は、[実践：追跡可能な単一 Agent アシスタントを作る](./ch10-projects/04-stage-hands-on-workshop.md) から始めてください。単一 Agent ループ、ツール schema 検証、権限ブロック、JSONL trace ログ、評価ケースを順番に確認できます。

## AI 学習アシスタントを通した継続プロジェクトとの関係

この段階は、AI 学習アシスタント v0.9 に対応できます。つまり、Q&A から学習計画 Agent へとアップグレードし、タスク分解、資料確認、ツール呼び出し、実行トレースの記録ができるようになります。継続プロジェクトの学習ルートに沿っているなら、この段階の終了時に少なくとも 1 回はバージョン記録を出すことをおすすめします。新しく追加した能力、実行方法、入出力例、遭遇した問題、次にどう直す予定か、をまとめましょう。

## 段階クリア基準

| クリアレベル | できるようになること |
|---|---|
| 最低クリア | ツール、記憶、計画、MCP、フレームワーク、評価、安全境界を設計できる。 |
| 推奨クリア | この段階で少なくとも 1 つの実行可能なミニプロジェクトを完成させ、README に実行方法、入出力例、遭遇した問題を記録する。 |
| 作品集クリア | この段階の成果を「AI 学習アシスタント」継続プロジェクトにつなげ、スクリーンショット、ログ、評価サンプル、次の計画を残す。 |

この段階を学び終えたら、すべての細部を暗記する必要はありません。もっと大事なのは、この段階が何の問題を解決するのか、前の段階とどうつながるのか、そして次の学習をどう支えるのかを、はっきり説明できることです。次は、マルチモーダル Agent、デプロイ、卒業作品へ進めます。
