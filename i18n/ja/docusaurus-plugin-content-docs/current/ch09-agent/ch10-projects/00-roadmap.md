# 学習ガイド：この統合プロジェクト章はどう学べばいいのか

この章は、これまで学んだ推論、ツール、記憶、MCP、評価、安全性、デプロイを、実際にプロジェクトへ落とし込む章です。

Agent プロジェクトの核心は「モデルが答えられるか」ではなく、システムが目標に向かって継続的に行動できるかです。つまり、タスクを分解し、ツールを選び、結果を観察し、状態を更新し、失敗を処理し、過程を記録し、適切なタイミングで停止して成果物を出せるかが重要です。

## この章がコース全体の中で占める位置

第 9 章「AI Agent とエージェントシステム」の統合プロジェクトは、Agent 学習の出口です。前の章では、Agent の目標、計画、ツール、記憶、MCP、マルチ Agent、評価・安全性、デプロイ運用をそれぞれ学びました。このプロジェクト章では、それらを統合して、デモ可能で、振り返り可能で、拡張できるエージェントシステムにします。

![Agent 統合プロジェクトのロードマップ](/img/course/ch09-projects-route-map-ja.png)

## この章が本当に解決したい問題

この章は、次の 5 つの問いに答えます。どうやって Agent プロジェクトの明確な目標と完了条件を定義するか。どうやってリサーチアシスタント、データ分析 Agent、マルチ Agent 開発チームのようなプロジェクト形態を選ぶか。どうやって各ステップの計画、ツール呼び出し、観察結果を記録するか。どうやってツール失敗、検索失敗、信頼できない出力、コスト高を処理するか。どうやって Agent プロジェクトを、単なる「一度うまく動いたデモ」ではなく作品集としてまとめるか。

初心者が最もやりがちな間違いは、Agent がタスクを成功した最終結果だけを見せることです。本当に価値のある Agent プロジェクトは、プロセスを見せられる必要があります。なぜそのように計画したのか、どのツールを呼んだのか、ツールが何を返したのか、失敗したときどう修正したのか、最後にどのようにタスク完了と判断したのか、まで示すことが大切です。

## 初心者におすすめの学習順序

まずはスマートリサーチアシスタントから始めるのがおすすめです。検索、引用、要約、信頼できる出力の練習に最適だからです。次にデータ分析 Agent を作り、Python ツール呼び出し、表の分析、グラフ生成、結果の説明を練習します。最後にマルチ Agent 開発チームに進み、役割分担、タスク調整、コード生成、レビュー、ドキュメント協働を学びます。

![Agent プロジェクトの学習順序図](/img/course/ch09-project-learning-order-map-ja.png)

## この章で押さえるべき主線

この章の主線は、次のようにまとめられます。Agent プロジェクトの作品集が示すのは、「追跡可能な実行の閉ループ」であって、一回のモデル出力ではありません。

![Agent プロジェクトの納品閉ループ図](/img/course/ch09-project-delivery-loop-ja.png)

この線を理解すると、Agent プロジェクトの成果物には何が必要かが分かります。たとえば、システム構成図、タスクフロー図、ツール一覧、呼び出しログの例、失敗ケース、安全ルール、評価例、最終出力などです。

## 3 つのプロジェクトでそれぞれ何を練習するのか

| プロジェクト | 本当に練習すること |
|---|---|
| スマートリサーチアシスタント | 検索、引用、要約、信頼できる出力 |
| データ分析 Agent | ツール呼び出し、表分析、結果の説明 |
| マルチ Agent 開発チーム | 複数役割の協働、タスク分解、レビューの閉ループ |

## この章とコース全体の関係

Agent 統合プロジェクトは、これまでのほぼすべての段階をつなげることができます。Python とデータ分析はツール能力を提供し、機械学習と深層学習はモデル理解を支え、大規模言語モデルと RAG は言語と知識の能力を支えます。さらに、エンジニアリングはデプロイと可観測性を提供し、マルチモーダルは入力・出力形式をさらに広げられます。

この章が固まっていないと、よくある問題は次の通りです。Agent に明確な停止条件がない。ツール呼び出しが記録されていない。失敗時にモデルが適当に作り話をする。記憶と状態が混乱する。マルチ Agent が単に複数の役割で雑談しているだけになる。評価と安全の境界がない。プロジェクトを再現できない。

## 初心者と上級学習者はどう読むべきか

初心者がこの章を初めて学ぶときは、まず主線と最小実行例に集中してください。すべての細部を一度に理解する必要はありません。この章が何を解決するのか、入出力は何か、最小プロジェクトはどう動かすのかが説明できれば、先へ進んで大丈夫です。

経験者は、この章を抜け漏れチェックとエンジニアリング練習として読むとよいでしょう。境界条件、失敗ケース、評価方法、コードの再現性、そして前後の章とのつながりに注目してください。読了後は、本章の内容を自分の作品 README や実験記録にまとめておくのがおすすめです。

## 本章の自己チェック問題

## 本章の小プロジェクト出口

この章を学び終えたら、少なくとも 1 つの「追跡可能な単一 Agent プロジェクト」を完成させるのがおすすめです。そこには、目標入力、計画生成、ツール呼び出し、観察記録、失敗処理、最終出力、評価の振り返りを含めてください。

上位版では、長期記憶、MCP ツール、マルチ Agent 協働、デプロイ API、可視化された実行軌跡も追加できます。作品集版では、構成図、実行スクリーンショット、ログ抜粋、テストタスク集、失敗ケース、コスト見積もりを補足するとよいでしょう。

## プロジェクト成果物テンプレート

Agent プロジェクトで最も避けたいのは、「最後の答え」だけを見せることです。プロジェクトを本物のシステムに近づけるために、少なくとも次の成果物を含めることをおすすめします。

| 成果物 | 説明 |
|---|---|
| 目標定義 | Agent が何を達成するのか、何を担当しないのか、いつ完了とするのかを明確にする |
| システム構成図 | モデル、ツール、記憶、状態、評価、安全境界を示す |
| ツール一覧 | 呼び出し可能なツール、入出力形式、失敗時の状況を列挙する |
| 実行軌跡 | 計画、行動、観察、再計画の一連の流れを示す |
| ツール呼び出しログ | 各呼び出しのパラメータ、結果、エラー、再試行を示す |
| 失敗ケース | 少なくとも 3 つの失敗タスクを挙げ、計画、ツール、記憶、安全のどれが原因かを分析する |
| 評価タスク集 | Agent が安定して目標を達成できるか確認するための固定タスク集を用意する |
| コストと安全の説明 | 呼び出しコスト、権限の境界、停止条件、人的引き継ぎ方法を説明する |

このテンプレートがあれば、あなたの Agent が「会話できる」だけではなく、追跡可能で、振り返り可能な実行ループを持っていることを他の人に見せられます。

## Debug 探偵案件

| 案件 | 内容 |
|---|---|
| 案件名 | Agent その場でぐるぐる回る事件 |
| 発生現場 | Agent が計画を繰り返し、同じツールを何度も呼び出したり、実行してはいけない動作をしたりする。 |
| 捜査手順 | 最大ステップ数を制限し、thought/action/observation を保存し、リスクの高いツールには手動確認を追加する。 |
| 結了証拠 | agent_traces.jsonl、tool_calls.jsonl、権限境界の説明。 |

プロジェクト練習では、成功スクリーンショットだけを残さないでください。少なくとも 1 つは本当の失敗サンプルを選び、「現象、手がかり、容疑原因、捜査手順、修正アクション、回帰確認」の形で `reports/failure_cases.md` に書き込みましょう。そうすると、プロジェクトがより本物のエンジニアリング作品らしくなります。

## プロジェクト成果物の基準

各統合プロジェクトは、コードが動けばよいのではなく、同じ作品集基準で提出するのがおすすめです。最小の成果物には、README、再現可能な実行コマンド、サンプル入出力、重要フロー図、1 件の失敗サンプル分析、今後の改善計画を含めてください。

| 成果物 | 最低要件 | 上位要件 |
|---|---|---|
| README | プロジェクト目標、実行方法、依存関係、サンプルを明記する | 構成図、設計上のトレードオフ、振り返りを追加する |
| サンプル入出力 | 少なくとも 1 つの完全なケースを残す | 成功、失敗、境界ケースを残す |
| 評価記録 | どの指標で効果を判断するか明記する | baseline、比較実験、エラー分析を入れる |
| エンジニアリング記録 | 1 回の環境問題または API 問題を記録する | ログ、コスト、時間、障害対応の流れを記録する |
| 発表素材 | スクリーンショットや短い GIF で実行可能性を示す | 説明可能な作品集ページにまとめる |

プロジェクトで最も大切なのは、機能をたくさん詰め込むことではなく、何を解決したのか、システムがどう動くのか、効果をどう判断するのか、失敗時にどう特定するのか、次の版でどう改善するのかを、はっきり説明できることです。

## 合格基準

この章が終わるころには、Agent プロジェクトを 1 つ作れ、その目標、状態、ツール、記憶、計画、評価、デプロイ方法を明確に説明できるようになっているはずです。最後の答えだけではなく、各ステップのツール呼び出し、観察結果、失敗処理、最終出力を示せる必要があります。

あなたのプロジェクト文書を見れば、他の人が Agent の実行過程を 1 回再現でき、そのシステムがなぜそのように判断したのか理解できるなら、AI Agent 段階の作品集出口基準を満たしています。

## バージョン進化のおすすめ

| バージョン | 目標 | 成果物の重点 |
|---|---|---|
| ベーシック版 | 最小の閉ループを動かす | 入力できる、処理できる、出力できる、そしてサンプルを残す |
| スタンダード版 | 見せられるプロジェクトにする | 設定、ログ、エラー処理、README、スクリーンショットを追加する |
| チャレンジ版 | 作品集品質に近づける | 評価、比較実験、失敗サンプル分析、次のロードマップを追加する |

まずはベーシック版を完成させることをおすすめします。最初から大きく全部を作ろうとしないでください。バージョンを 1 つ上げるたびに、「何の能力を追加したか、どう検証したか、まだ何が問題か」を README に書き足していきましょう。
