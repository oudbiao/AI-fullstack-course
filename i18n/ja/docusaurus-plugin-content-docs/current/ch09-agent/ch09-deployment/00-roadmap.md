# 学習ガイド：デプロイと運用保守のこの章では何を学ぶのか

この章で解決したいのは、Agent のプロトタイプをどうやって、本当に長く動き続け、復旧でき、コストを抑えられ、保守できるシステムに変えるか、です。

多くの Agent はローカルの Demo ではうまく見えますが、実際の環境に置くと新しい問題に出会います。たとえば、リクエストの並列実行、モデルのタイムアウト、ツールの失敗、状態の消失、タスクの中断、コストの急増、ログ不足、権限設定の複雑さ、ユーザーのフィードバックを戻せないことなどです。デプロイと運用保守の章で鍛えるのは、こうした本番システムの考え方です。

## この章がコース全体の中でどこにあるか

ここまでで、あなたは Agent のコア機能と、評価・安全性について学んできました。デプロイと運用保守の段階に入ると、コースは「この Agent は 1 回のタスクを完了できるか」から、「ユーザーに継続してサービスを提供できるか」へと移ります。

デプロイは、コードをサーバーに置けば終わり、ではありません。Agent にとってのデプロイには、モデル呼び出し層、ツールサービス層、タスクキュー、状態ストレージ、ログ追跡、エラー復旧、コスト監視、権限管理、バージョン更新まで含まれます。

![Agent 本番運用アーキテクチャ図](/img/course/ch09-production-runtime-map-ja.png)

前半ではまずサービス、設定、実行環境を整え、後半で監視、コスト、障害復旧、本番運用を補っていきます。

## この章で本当に解決したい問題

この章では、次の 5 つの問いに答えます。Agent サービスはどう分割して設計するのか。長時間タスク、非同期タスク、中断からの復旧はどう扱うのか。状態、記憶、ツール結果、実行軌跡はどう永続化するのか。モデル呼び出し、ツール呼び出し、複数 Agent の協調で増えるコストをどう抑えるのか。リリース後は、ログ、監視、アラート、ユーザーフィードバックを使ってどう改善を続けるのか、です。

初心者がもっとも誤解しやすいのは、「Agent のロジックを書けば、そのまま公開できる」と思ってしまうことです。実際の本番環境では、1 回うまく動かすことよりも、失敗、タイムアウト、並列実行、再試行、権限変更、モデルの揺れがあってもなお、制御可能であることのほうがずっと難しいのです。

## 初心者におすすめの学習順序

まずデプロイアーキテクチャを学び、フロントエンド、バックエンド、モデルサービス、ツールサービス、ストレージの関係を理解しましょう。次にランタイム管理を見て、同期タスク、非同期タスク、長時間タスク、キューを重点的に理解します。続いて永続化と復旧を学び、タスク状態、記憶、ログ、中間結果をなぜメモリだけに置いてはいけないのかを知ります。その後でコスト最適化を学び、token、モデル選択、キャッシュ、バッチ処理、ツール呼び出し回数がコストにどう影響するかを理解します。最後に本番ベストプラクティスを学び、監視、アラート、権限、段階的リリース、ロールバックを補いましょう。

## この章を学ぶときに押さえるべき主線

この章の主線は、次のようにまとめられます。Agent をデプロイするとは、モデル、ツール、状態、評価を、実行可能で、観測できて、復旧できるエンジニアリングシステムに載せることです。

![Agent デプロイの可観測性と復旧の閉ループ図](/img/course/ch09-deployment-observability-loop-ja.png)

前半ではまずサービス、設定、実行環境を整え、後半で監視、コスト、障害復旧、本番運用を補っていきます。

この線が見えるようになると、本番 Agent で大事なのは「賢そうに見えること」ではなく、各ステップが記録され、復旧でき、制限でき、最適化できることだと分かります。

## この章と後続章の関係

デプロイと運用保守は、第 9 章で AI Agent とエージェントシステムが実際のプロジェクトに進むための出口です。ここでは最終的な総合プロジェクトにもつながり、さらに前の評価、安全性、ツール、記憶設計にもつながります。公開できる Agent プロジェクトには、機能の閉ループ、エラー処理、ログ追跡、安全境界、コスト意識、デプロイ手順の説明がすべて必要です。

この章が安定して身についていないと、後でよく起きるのは、ローカルでは動くのに本番ではタイムアウトする、タスク中断後に復旧できない、マルチターンの状態が消える、ログが最終回答しか記録せず失敗箇所を特定できない、モデル呼び出しコストが制御できない、段階的リリースやロールバックの戦略がない、といった問題です。

## 初心者と上級学習者はどう読むか

初心者が初めてこの章を学ぶときは、まず主線と最小の実行例をつかみましょう。細部を一度に全部理解する必要はありません。この章が何を解決するのか、入力と出力は何か、最小プロジェクトをどう起動するのかが説明できれば、先へ進んで大丈夫です。

経験のある学習者は、この章を「抜け漏れの確認」と「エンジニアリング練習」として読むとよいでしょう。境界条件、失敗例、評価方法、コードの再現性、そして前後の章とのつながりに注目してください。読み終えたら、この章の内容を自分の作品 README や実験記録に整理して残すのがおすすめです。

## 学習時間と難易度の目安

| 学習方法 | 推奨時間 | 目標 |
|---|---|---|
| ざっと読む | 20～30 分 | この章が何を解決するのかを理解し、後でどこで使うかを知る |
| 最小限の合格 | 1～2 時間 | 最小例を動かして、この章の小プロジェクトの出口まで到達する |
| じっくり練習 | 半日～1 日 | エラー分析、比較実験、またはプロジェクト README の記録を追加する |

## 本章の自己チェック問題

| 自己チェック問題 | 合格基準 |
|---|---|
| この章は何を解決するのか？ | このコース全体の中での位置を 1 文で説明できる |
| 最小の入力と出力は何か？ | 例に必要な入力と、得られる結果を説明できる |
| よくある失敗ポイントはどこか？ | 少なくとも 1 つ、エラー・効果の低さ・理解のズレの原因を挙げられる |
| 学んだ後に何を残せるか？ | 本章の成果をプロジェクト README、実験記録、作品集に書ける |

## 本章の小プロジェクト出口

この章を学び終えたら、前章までで作った学習アシスタントまたはリサーチアシスタント Agent を、最小限のデプロイ可能バージョンにしてみましょう。API 入口があり、タスク作成、モデル呼び出しとツール呼び出しの実行、タスク状態の保存、呼び出し軌跡の記録ができ、失敗時には理解しやすいエラーメッセージを返せることが理想です。

プロジェクトは複雑な UI を追わなくて構いません。ただし、デプロイ手順、環境変数の設定、ログ記録、基本的なコスト見積もり、少なくとも 1 つの失敗復旧戦略は必ず含めましょう。

## 合格基準

この章の終わりには、Agent アプリケーションのデプロイアーキテクチャを描けること、ランタイム管理、状態の永続化、障害復旧、ログ監視、コスト最適化がそれぞれ何を解決するのか説明できること、そしてローカル Demo を、再起動可能で、観測できて、設定可能な小さなサービスに変えられることが目標です。

さらに、Agent プロジェクトに API 入口、タスク状態、呼び出しログ、エラー処理、コスト記録、デプロイ文書を持たせられれば、Agent 段階のエンジニアリング出口基準に到達したことになります。
