# 学習ガイドとタスクシート：AI Agent とインテリジェントエージェントシステムはどう学ぶと一番混乱しにくいか

`第 9 章 AI Agent 与智能体系统` に入って、ツール、記憶、フレームワーク、マルチAgent、MCP が全部ごちゃ混ぜに感じたら、まずは Agent の最小ループに戻りましょう。目標、計画、行動、観察、修正です。

## この段階の総則

Agent の最初の段階では、急いでマルチAgent に進んだり、いきなりフレームワークを積み重ねたりしないでください。まずは単体の Agent を安定させましょう。つまり、目標を理解し、次の一手を決め、ツールを呼び出し、結果を観察し、続けるかどうかを判断できるようにします。

![最小 Agent 学習ループ図](/img/course/ch09-study-guide-minimal-agent-loop-ja.png)

## この段階で必ず完了するタスク

このタスクで Agent 学習を観察可能にします。Agent プロジェクトは「ツールを 1 回呼べた」だけでは完成ではありません。trace を再生でき、停止条件を説明でき、どこで人間の確認が必要かを示せる状態にします。

| タスク | 成果物 | 合格基準 |
|---|---|---|
| Agent の基本構造を理解する | 1 枚のアーキテクチャ図またはノート | モデル、ツール、記憶、プランナー、実行器の関係を説明できる |
| ツール呼び出しを動かす | 最小の tool calling サンプル | ツールの引数を定義し、戻り値とエラーを処理できる |
| 計画・実行フローを完成させる | 1 つの複数ステップタスク Demo | 各ステップの入力、出力、状態、失敗理由を記録できる |
| 評価と安全性を追加する | Agent テスト記録 1 つ | タスク成功率、ツール精度、権限リスク、停止条件を評価できる |
| 単体 Agent のガイド付きワークショップを実行する | `agent_workshop.py` の出力と trace ログ | `logs/agent_traces.jsonl`、小さな評価出力、権限チェックを説明できる |
| 段階プロジェクトを 1 つ完成させる | 追跡可能な Agent プロジェクト | trace、ログ、再生サンプル、手動確認の境界がある |

## 推奨学習順序

1 回目はまず Agent の基礎を学び、Agent、普通のチャットボット、固定ワークフロー、RAG アプリの境界をはっきりさせます。

2 回目は推論と計画を学びます。ReAct、Chain-of-Thought、Plan-and-Execute、推論評価を含みます。

3 回目はツール呼び出しを学びます。ツールの説明、パラメータ schema、エラー処理、権限の境界、安全対策は、Agent が信頼して実行できるかどうかの鍵です。

4 回目は記憶システムを学びます。短期記憶、長期記憶、エピソード記憶、手続き記憶は、すべてタスクに役立つためにあります。「賢そうに見せる」ためではありません。

5 回目に MCP、フレームワーク、マルチAgent、評価、安全、デプロイを学びます。フレームワークは、システムの階層を理解してから使うべきです。

## 選定の分岐: まず Agent を使わないほうがよい場面

初心者は Agent を早く使いすぎがちです。より安全な考え方は、その問題が本当に「目標駆動の実行ループ」を必要としているかを先に見極めることです。

| 問題の種類 | 最初の選択肢 | 理由 |
|---|---|---|
| 固定されたプロセスで、手順が完全に決まっている | ワークフロー | プログラムがあらかじめ各ステップを決められるから |
| 最新または引用可能な知識が必要 | RAG | まず外部知識をつなぎ、モデルに推測させないほうがよいから |
| 分類や抽出のような 1 回だけの構造化操作 | Prompt / Function Calling | 完全な Agent ループは不要だから |
| 継続的な目標駆動の行動、ツール呼び出し、状態更新が必要 | Agent | ここが Agent の最も強い領域だから |

本当に Agent を作るなら、まず次の 3 つの安全門を通します。

1. 権限は最小化されているか
2. 最大ステップ数の制限があるか
3. 各ステップをログ化して手動で引き継げるか

## アップグレード順序: 単体 Agent を先に、マルチAgent はあとで

1. まず単体 Agent を安定させる
2. 次にツールと記憶を接続する
3. 単体 Agent が安定してから MCP やフレームワークを追加する
4. 最後にマルチAgent 協調を考える

## おすすめの学習ペース

| 内容タイプ | 推奨時間 | 学習目標 |
|---|---|---|
| Agent 基礎 | 4～8 時間 | Agent と普通のアプリを区別できるようになる |
| 推論とツール | 8～16 時間 | 単体 Agent の実行ループを一通り動かせるようになる |
| 記憶と MCP | 8～16 時間 | 外部コンテキストとツールエコシステムを理解する |
| フレームワークとマルチAgent | 8～20 時間 | いつ複雑なオーケストレーションが必要か判断できるようになる |
| 評価・デプロイ・プロジェクト | 16～32 時間 | 観察可能で、復旧可能な Agent Demo を作る |

## 段階的なプロジェクトの進め方

最初のプロジェクトはリサーチアシスタントがおすすめです。1 つのテーマを入力すると、Agent が問題を分解し、資料を検索し、要約を整理して、構造化レポートを出力します。

大きなプロジェクトを選ぶ前に、まず [追跡可能な単一 Agent の実践ワークショップ](./ch10-projects/04-stage-hands-on-workshop.md) を動かしてください。本ステージの baseline として、1 つの Agent、いくつかのツール、権限チェック、trace ログ、固定評価ケースを確認できます。

2 つ目のプロジェクトはデータ分析 Agent がおすすめです。データを読み込み、分析計画を立て、Python ツールを呼び出し、グラフと結論を生成します。

3 つ目のプロジェクトでは、マルチAgent の開発チーム Demo を作れます。異なる役割の Agent が、要件分析、コード生成、テスト、ドキュメント作成を協力して進めます。

## よくあるつまずきポイント

最もよくあるつまずきは、マルチAgent を早くやりすぎることです。単体 Agent のツール呼び出し、エラー復旧、コンテキスト管理がまだできていないと、マルチAgent は混乱をさらに大きくするだけです。

2 つ目のつまずきは、成功パスだけを見せることです。実際の Agent では、ツール失敗、パラメータエラー、権限不足、ループ実行、コスト暴走、出力の信頼性不足を必ず考える必要があります。

3 つ目のつまずきは、システムより先にフレームワークを学びすぎることです。まず Agent のアーキテクチャを理解してから、LangGraph、CrewAI、AutoGen、または他のフレームワークを選ぶべきです。

## 段階ポートフォリオ成果物

![Agent trace 再生図](/img/course/ch09-workshop-trace-jsonl-replay-map-ja.png)

この段階の成果を作品集として残したいなら、少なくとも次のファイル、または同等の資料を残すことをおすすめします。

| 成果物 | 説明 |
|---|---|
| `tools_schema.md` | 各ツールの名前、用途、パラメータ、戻り値、使用範囲 |
| `agent_traces.jsonl` | 各実行の goal、step、action、arguments、observation、next_decision |
| `safety_boundary.md` | ツールのリスク分類、手動確認ルール、最大ステップ数、権限制限 |
| `human_approval_examples.md` | 高リスク操作の確認文の例。操作、対象、リスク、ロールバック方法を含む |
| `failure_cases.md` | ツール選択ミス、パラメータエラー、ループ呼び出し、未対応の参照、安全ブロックなどの失敗サンプル |
| `README.md` | プロジェクトの目的、ツール一覧、実行方法、trace 例、評価結果、制限説明 |

これらの資料があると、Agent プロジェクトは「ツールを呼び出せる」から「追跡可能で、制御できて、振り返りもできる」システムへと進化します。

## 段階通過質問

この段階を学び終えたら、1 つの Agent の目標、状態、ツール、記憶、計画、評価の方法を説明できるはずです。

先へ進む前に、次の質問に答えられるか確認してください。

- Agent と普通の LLM アプリの違いは何か。
- なぜツール説明が重要なのか。
- どのタスクが Agent に向いているのか。
- どうやってループ呼び出しを防ぐのか。
- なぜ trace、replay、手動確認が必要なのか。

単体 Agent のプロジェクトを 1 つ作り、各ステップのツール呼び出し、観察結果、失敗時の処理、最終出力を記録できれば、マルチAgent、デプロイ、またはマルチモーダルの方向へ進んで大丈夫です。
