# 学習ガイド：評価と安全、この章で何を学ぶのか

この章で扱うのは、Agent は「動けばよい」だけではなく、「うまく動いているか」「安全か」「問題が起きたときに見えるか」まで確認する必要がある、ということです。

多くの Agent デモは成功パスだけを見せます。たとえば、目標を 1 つ入力すると、システムがツールを呼び出し、最後にそれらしい結果を返します。しかし実際のシステムでより重要なのは、なぜそのように動いたのか、途中のプロセスは信頼できるのか、ツール呼び出しが権限を越えていないか、答えは検証できるのか、失敗したときに追跡できるのか、コストは制御できるのか、ユーザーが理解して介入できるのか、という点です。

## この章がコース全体の中でどこにあるのか

ここまでに、あなたは Agent の目標、計画、ツール、記憶、MCP、そして複数 Agent を学んできました。評価と安全の章に入ると、コースは「実装できる」から「信頼できる」へと移ります。

Agent のリスクは、普通のチャットシステムより高くなります。なぜなら、内容を生成するだけでなく、ツールを呼び出し、データを読み、ファイルを変更し、コードを実行し、外部の処理を起動することもあるからです。そのため、評価と安全は最後に付け足すものではなく、Agent システム設計の一部として最初から考える必要があります。

前半では Agent のタスクリスク、失敗パターン、評価観点を見分け、後半ではテストセット、安全境界、人的な引き継ぎ、本番前チェックを設計します。

## この章で本当に解決したい問題

この章では、次の 5 つの疑問に答えます。どうすれば Agent がタスクを完了したと判断できるのか。最終回答だけでなく、計画、ツール呼び出し、中間観察をどう評価すべきか。ベンチマークと独自の評価セットにはそれぞれどんな役割があるのか。Guardrails、権限管理、入出力の検証、人的確認は、どうやってリスクを下げるのか。ログ、軌跡、コスト、エラー情報は、デバッグと運用にどう役立つのか。

初心者が最も見落としやすいのは、Agent の間違いは最終回答にだけ出るとは限らない、という点です。タスク理解の段階ですでにずれていることもあれば、ツール選択を間違えることもある、パラメータの作り方を誤ることもある、観察結果の要約で重要な事実を落とすこともあります。見た目はうまくいっているように見えても、実際には途中で失敗しているのです。だからこそ、Agent の評価ではプロセスを見る必要があります。

## 初心者におすすめの学習順序

まずは評価方法を学び、結果評価、プロセス評価、人的評価、自動評価を区別できるようにしましょう。次に benchmarks を見て、公開ベンチマークは参考になる一方で、実際のプロジェクトには自分専用のタスクセットが必要だと理解します。続いて安全とアラインメントを学び、権限越え、プロンプトインジェクション、ツールの誤用、データ漏えい、幻覚のリスクを理解します。その後で Guardrails を学び、入力フィルタ、出力検証、権限の境界、人的確認を身につけます。最後に可観測性を学び、ログ、呼び出し軌跡、エラー、遅延、コストを記録できるようにします。

## この章で押さえるべき主線

この章の主線は、次のようにまとめられます。評価はシステムが有効かどうかを教えてくれる。安全はシステムが何をできるかを制御する。可観測性は問題がどこで起きたかを教えてくれる。

![Agent リスク原因究明のループ図](/img/course/ch09-agent-risk-debug-loop.png)

前半では Agent のタスクリスク、失敗パターン、評価観点を見分け、後半ではテストセット、安全境界、人的な引き継ぎ、本番前チェックを設計します。

この流れが分かると、評価は「本番前に一度点数をつける作業」ではなく、継続的に改善する仕組みだと分かります。失敗するたびに、その原因を切り分けられるようにする必要があります。モデルの理解ミスなのか、計画ミスなのか、ツールのミスなのか、権限のミスなのか、資料の問題なのか、それとも最終表現の問題なのか、を見極めます。

## この章と後続章の関係

評価と安全は、デプロイと運用の前提です。評価がなければ、システムを本番に出す価値があるか分かりません。安全境界がなければ、Agent のツール呼び出しは制御不能なリスクになります。可観測性がなければ、本番で問題が起きても原因を特定できません。後半のデプロイ章では、これらの要件をアーキテクチャ、ログ、復旧、コスト、本番運用の実践に落とし込んでいきます。

この章をしっかり学ばないままだと、次のような問題が起きやすくなります。Demo は成功して見えるのに再現可能な評価がない。ツール権限が広すぎる。ユーザー入力でシステムをだましたり、誤操作させたりできる。問題が起きても最終回答しか見えず、途中の失敗箇所が分からない。コストと遅延が制御不能なのに記録がない。

## 初心者と上級学習者はどう読むか

初心者が初めてこの章を学ぶときは、まず主線と最小限の実行例をつかみましょう。すべての細部を一度で理解する必要はありません。この章が何を解決するのか、入出力は何か、最小プロジェクトをどう動かすのかを説明できれば、先へ進んで大丈夫です。

経験のある学習者は、この章を「抜け漏れの確認」と「エンジニアリング練習」として読むとよいです。境界条件、失敗ケース、評価方法、コードの再現性、そして前後の段階とのつながりに注目しましょう。読み終えたら、この章の内容を自分の作品の README や実験記録にまとめておくのがおすすめです。

## 学習時間と難易度の目安

| 学習方法 | 目安時間 | 目標 |
|---|---|---|
| ざっと読む | 20～30 分 | この章が何を解決するのかを理解し、後でどこで使うかを知る |
| 最小通過 | 1～2 時間 | 最小例を動かし、章内の小プロジェクトを完了する |
| 深く練習する | 半日～1 日 | エラー分析、比較実験、プロジェクト README への記録を追加する |

## この章の小プロジェクト出口

この章を学び終えたら、前に作った研究アシスタントや学習アシスタントの Agent に、評価層と安全層を追加するのがおすすめです。10～20 個のテストタスクを用意し、毎回の実行で計画、ツール呼び出し、観察結果、最終出力、人的評価を記録します。さらに、少なくとも 3 つの安全ルールを加えましょう。たとえば、機密操作は確認が必要、ツールのパラメータは必ず検証する、出典のない情報は断定しない、などです。

このプロジェクトの重点は、Agent の振る舞いを追跡できること、評価できること、振り返れることです。見た目のよい出力を 1 回出すことだけが目的ではありません。



## Agent 評価指標の総覧

Agent の評価では、タスク結果と実行プロセスの両方を見る必要があります。答えが正しそうに見えても、実行経路が安全とは限らないし、コストが制御できるとも、再現できるとも限りません。

| 観点 | 指標 | どんな問いに答えるか |
|---|---|---|
| タスク効果 | タスク成功率、人的評価、完了度 | ユーザーの目的は達成されたか |
| ツール使用 | ツール選択の正確率、パラメータエラー率、ツール失敗率 | Agent は外部機能を正しく呼び出せたか |
| プロセス品質 | ステップ数、再試行回数、ループ率、人的介入率 | 実行は安定して制御できたか |
| 安全境界 | 権限越え動作率、確認カバー率、拒否正確率 | 高リスク動作を制約できているか |
| コスト性能 | token コスト、遅延、並行処理の安定性 | システムは長期運用に耐えられるか |

これから Agent プロジェクトを作るときは、少なくとも 10～20 件の再生可能なタスクサンプルを残しましょう。各サンプルには、ユーザーの目的、計画、ツール呼び出し、結果、失敗原因、改善提案が見えるようにしておくべきです。

## 合格基準

この章が終わるころには、結果評価とプロセス評価の違いを説明でき、小規模な Agent テストセットを設計でき、Guardrails、権限管理、入出力検証、可観測性の役割を説明でき、呼び出し軌跡から Agent の失敗箇所を特定できるようになっているはずです。

ひとつの Agent デモを、ログ、評価サンプル、安全ルール、失敗の振り返り機構を備えたシステムに変えられれば、デプロイと運用の段階に進むための基礎条件は満たしています。
