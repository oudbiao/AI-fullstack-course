# 9.7.1 事前ガイド：この章で学ぶ多 Agent とは何か

この章で解決したいのは、いつ 1 つの Agent では足りないのか、なぜタスクを複数の Agent に分けるのか、そして分けたあとにどうすればシステムがさらに混乱しないのか、ということです。

多 Agent は、とてもかっこよく見えます。計画 Agent、実行 Agent、レビュー Agent、プロダクトマネージャー Agent、エンジニア Agent、テスト Agent など、それぞれが違う役割を演じます。ですが、このコースで強調したいのは、多 Agent は数が多ければ多いほどよいわけではない、ということです。タスクに本当に役割分担、並列処理、相互チェック、または複雑な協働が必要なときだけ、多 Agent に意味があります。

## この章がコース全体の中でどこにあるか

ここまでで、単一 Agent の目標、計画、ツール、記憶、そして MCP を学んできました。多 Agent は、その土台の上にある、より複雑なオーケストレーションです。もし単一 Agent のツール呼び出し、状態管理、失敗からの復旧がまだ安定していないなら、多 Agent はむしろ混乱を大きくするだけになりがちです。

この章の大事なポイントは「たくさんの役割を作ること」ではなく、協働システムの境界を理解することです。誰が何を担当するのか、どう通信するのか、誰が最終決定をするのか、どの状態を共有するのか、重複作業や誤った誘導、無限ループをどう防ぐのか、を考えます。

## この章で本当に解決したい問題

この章では、次の 5 つの問いに答えます。どんな場合に多 Agent が必要で、どんな場合は単一 Agent のほうがよいのか。多 Agent によくあるアーキテクチャパターンには何があるのか。Agent 同士はどう通信し、状態を共有するのか。タスクの調整、衝突処理、最終決定はどう設計するのか。多 Agent システムのコスト、品質、安全性、安定性をどう評価するのか。

初心者がもっとも誤解しやすいのは、「複数の Agent を使えば、自動的に賢くなる」という考えです。実際には、多 Agent は追加の複雑さを生みます。たとえば、コンテキストの重複、役割の境界のあいまいさ、メッセージのノイズ、コスト増加、責任の不明確さ、エラーの連鎖などです。

## 初心者におすすめの学習順序

まず、多 Agent が適している範囲を学び、どんなときに使うべきでないかを明確にしましょう。次に、主管-実行者、ディベート型、パイプライン型、専門家会議型など、よくあるアーキテクチャパターンを見ます。続いて、通信と調整を学び、メッセージ形式、共有状態、タスクキュー、集約メカニズムを理解します。最後に、課題と実践を学び、コスト制御、失敗からの復旧、評価、安全性に重点を置きます。

## この章で押さえるべき主線

この章の主線は、次のようにまとめられます。多 Agent は、タスク分担と協働の仕組みであって、単にチャットボットを複製することではありません。

![多 Agent 協働調整図](/img/course/ch09-multi-agent-coordination-map-ja.png)

この線を理解できると、多 Agent の鍵は「調整コストが分業の利益より小さいかどうか」だとわかります。もし複数の Agent の通信コスト、エラーの伝播、評価コストが利益を上回るなら、単一 Agent か固定ワークフローに戻るべきです。

## この章と後続章の関係

多 Agent は、評価・安全・デプロイの章に直接つながります。評価では、複数の Agent が本当に品質を上げているのか、それとも単により多くの内容を生成しているだけなのかを見ます。安全では、権限の分離、ツールアクセス、Prompt injection、役割の権限逸脱を扱います。デプロイでは、並行処理、コスト、ログ、可観測性、失敗からの復旧を考慮します。

この章をしっかり理解していないと、後でよくある問題にぶつかります。たとえば、役割は多いのに責任が重なっている、各 Agent が同じ分析を繰り返している、最終決定者がいない、中間メッセージがどんどん長くなる、コストが急に上がる、失敗したときにどの Agent の判断が原因かわからない、などです。

## 初心者と上級学習者はどう読むか

初心者は、この章を初めて学ぶとき、主線と最小実行例をまずつかみましょう。すべての細部を一度に理解する必要はありません。この章が何を解決するのか、入力と出力は何か、最小プロジェクトをどう動かすのかが説明できれば、先へ進めます。

経験のある学習者は、この章を抜け漏れの確認と実務化の練習として使えます。境界条件、失敗例、評価方法、コードの再現性、そして前後の段階とのつながりに注目しましょう。読み終わったら、この章の内容を自分の作品の README や実験記録にまとめておくとよいです。

## 本章の小プロジェクト出口

この章を学び終えたら、「コース内容改善チーム Demo」を作るのがおすすめです。調査 Agent にコース上の問題を探してもらい、編集 Agent に内容を言い換えてもらい、レビュー Agent に初心者にとってわかりやすいか確認してもらい、統括 Agent に最終判断をまとめてもらいます。

このプロジェクトで大事なのは Agent の数ではなく、役割の境界、メッセージ形式、成果物、レビュー基準が明確かどうかです。各 Agent の入力、出力、ツール呼び出し、そして最終的に採用した理由・却下した理由を記録する必要があります。

## 合格基準

この章の終わりには、あるタスクに多 Agent が必要かどうかを判断でき、主管-実行者、パイプライン、ディベート、専門家会議などの基本パターンを説明でき、簡単な Agent 通信と集約メカニズムを設計でき、多 Agent のコスト、評価、安全リスクを説明できるようになっているはずです。

もし複雑なタスクを 2～3 個の Agent に分け、それぞれが追跡可能で、レビュー可能で、統合可能な結果を出せるなら、そして互いに同じ話を繰り返すだけでなければ、多 Agent の入門設計能力は身についていると言えます。
