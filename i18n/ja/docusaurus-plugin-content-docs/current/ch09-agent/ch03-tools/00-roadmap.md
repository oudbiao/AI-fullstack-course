# 学習ガイド：この章でツールについて何を学ぶのか

この章で解決するのは、Agent は「話せる」だけでなく「実行できる」必要がある、という点です。

もしシステムが文字を生成するだけなら、それは答えを返すだけの存在に近いです。  
もし目標に応じてツールを選び、パラメータを渡し、結果を観察しながら次の判断を続けられるなら、そのシステムはようやく行動する力を持ち始めます。  
ツール利用の章は、Agent が「言語能力」から「実行能力」へ進むための重要な章です。

## この章がコース全体でどこに位置するか

あなたはすでに Agent の基礎と推論・計画を学び、Agent が目標に向かって次の一手を考え続ける必要があることを知っています。  
このツールの章では、コースは次の問いに答え始めます。  
次の一手がテキスト生成ではなく、資料検索、ファイル読み取り、API 呼び出し、コード実行、検索の開始、データベース操作だった場合、システムはどうすれば安全かつ確実にそれを実行できるのか。

ツールは、Agent が外部世界につながるためのインターフェースです。  
ツールがなければ Agent は言語レイヤーに留まります。  
ツールがあって初めて、Agent は実際の業務フローに入ることができます。

![Agent ツール行動レイヤーマップ](/img/course/ch09-tools-action-layer-map.png)

## この章で本当に解決したい問題

この章では、次の 5 つの問いに答えます。  
Function Calling はなぜモデルの出力を実行可能な動作に変えられるのか。  
ツール説明とパラメータ schema はなぜ、モデルが正しくツールを呼び出せるかどうかを左右するのか。  
複数のツールをどう選び、どう調整するのか。  
ツール失敗、パラメータエラー、権限不足が起きたとき、どうやって復旧するのか。  
そして、なぜツール利用には安全境界の設計が欠かせないのか。

初心者が最も誤解しやすいのは、「Agent にツールをたくさんつなげれば、もっと強くなる」という考えです。  
実際には、ツールが増えるほど、選択ミス、パラメータミス、権限リスク、ループ呼び出し、コスト暴走の可能性も高くなります。  
ツールの能力は、タスク目標、呼び出しルール、エラー処理、安全戦略とセットで設計する必要があります。

## 初心者におすすめの学習順序

まずは Function Calling の最小フローを学び、モデルがどのように関数名とパラメータを出力するのかを理解しましょう。  
次にツール説明を学び、ツールの用途、入力フィールド、制約、例をどう書けばよいかを知ります。  
そのあと制御戦略を学び、いつどのツールを呼ぶべきか、無意味なループをどう避けるかを理解します。  
最後にツールの安全性と複数ツールの実践を学び、権限、検証、タイムアウト、監査、失敗復旧を補います。

## この章でつかむべき主線

この章の主線は、次のようにまとめられます。  
ツール呼び出しは「モデルが呼びたくなったら呼ぶ」のではなく、「管理された境界の中で計画を動作に変える」ことです。

![Agent ツールの制御された呼び出しループ図](/img/course/ch09-tool-control-loop.png)

前半では、ツールで何ができるのか、パラメータは何か、境界はどこかを定義します。  
後半では、呼び出し結果、エラー復旧、権限管理、実行ログを扱います。

この流れが理解できると、Agent の信頼性はモデルの賢さだけで決まるのではなく、ツール API がどれだけ明確か、パラメータを検証できるか、失敗から復旧できるか、権限が制御されているかにも左右されると分かります。

## この章と後続章の関係

ツール利用は、記憶、MCP、マルチ Agent、デプロイの基礎です。  
記憶では外部ストレージの読み書きが必要になるかもしれません。  
MCP は本質的に、より標準化されたツールエコシステムを提供します。  
マルチ Agent の協働ではツール調整の複雑さが増します。  
デプロイ段階では、ツール権限、ログ、監査、安全性を必ず考慮しなければなりません。

この章が不安定なままだと、後でよく起こる問題は次のようなものです。  
Agent は計画できているように見えても、実行時にパラメータを頻繁に間違える。  
ツール結果を正しく観察・要約できない。  
ツール失敗後にモデルが結果をでっち上げる。  
ツール権限が広すぎて安全リスクが高くなる。  
複数ツールのシステムがデバッグしづらくなる。

## 初心者と上級学習者はどう読むか

初心者がこの章を初めて学ぶときは、まず主線と最小の動く例に集中しましょう。  
すべての細部を一度に理解する必要はありません。  
この章が何を解決するのか、入力と出力は何か、最小プロジェクトをどう起動するのかを説明できれば、先へ進んで大丈夫です。

経験のある学習者は、この章を抜け漏れの確認と実装練習として読むとよいです。  
境界条件、失敗例、評価方法、コードの再現性、前後の段階とのつながりに注目しましょう。  
読み終えたら、この章の内容を自分の作品 README や実験記録にまとめておくとよいです。

## この章の小プロジェクトの到達点

この章を学び終えたら、「ツール付き学習アシスタント」を作るのがおすすめです。  
ユーザーが学習タスクを入力すると、Agent はコース検索ツールで資料を探し、計画生成ツールでタスクを分解し、ファイル生成ツールで学習計画を出力できます。  
各ツール呼び出しでは、関数名、パラメータ、戻り値、次の判断を記録する必要があります。

最小の成果物には、3 つのツール schema、1 つのツールホワイトリスト、少なくとも 5 件のツール呼び出しテストケース、1 つの失敗呼び出し記録、そして 1 本の出力可能な trace を含めることをおすすめします。  
大事なのはツールの数ではなく、各呼び出しがパラメータ制約と権限境界に合っているかどうかを確認できることです。

```python
trace.append({
    "tool": "search_course_docs",
    "args": {"query": "RAGOps 評価"},
    "observation": "2 件の文書にヒット",
    "next": "復習計画を生成する",
})
```

このプロジェクトで重要なのはツールの数ではなく、ツール呼び出しの過程が透明か、パラメータが構造化されているか、失敗時に段階的な代替策があるかです。

## 合格基準

この章の終わりには、Function Calling の基本フローを説明でき、分かりやすいツール schema を設計でき、ツール説明・パラメータ検証・エラー処理・安全境界がなぜ重要かを説明でき、最小構成の複数ツール Agent フローを実装できるようになっているはずです。

Agent のツール呼び出しログを読み、失敗が計画段階・パラメータ段階・ツール実行段階・結果観察段階のどこで起きたかを判断できるなら、あなたはすでに Agent の行動レイヤーの核心をつかんでいます。
