# 段階学習タスクシート：AI Agent とエージェントシステム

この段階の目標は、Agent が大規模モデル、ツール、記憶、計画、実行フローをどのようにつなぐのかを理解することです。Agent を「もっと会話が上手いモデル」と考えないでください。むしろ、タスクを分解し、ツールを呼び出し、状態を記録し、失敗を処理できるワークフローシステムに近いものです。

## この段階で必ず完了するタスク

| タスク | 成果物 | 合格基準 |
| --- | --- | --- |
| Agent の基本構造を理解する | 1 枚のアーキテクチャ図 | モデル、ツール、記憶、プランナー、実行器の関係を説明できる |
| ツール呼び出しを動かす | 最小の tool calling サンプル | ツールの引数を定義し、戻り値とエラーを処理できる |
| 計画・実行フローを完成させる | 1 つの複数ステップタスク Demo | 各ステップの入力、出力、状態、失敗理由を記録できる |
| 評価と安全性を追加する | Agent テスト記録 1 つ | タスク成功率、ツール精度、権限リスクを評価できる |
| 段階プロジェクトを完成させる | 追跡可能な Agent プロジェクト 1 つ | trace、ログ、再生サンプル、手動確認の境界がある |

## おすすめの学習順序

まず Agent とは何かを理解し、そのあと推論と計画、ツール呼び出し、記憶、MCP、フレームワーク、マルチAgent、評価と安全性を学びます。フレームワークは実装の助けになりますが、核心を隠してしまわないように注意してください。状態はどう流れるのか、ツールはいつ呼び出すのか、失敗はどう復旧するのか、権限はどう制限するのか、ここが大事です。

Agent を学ぶときは、特に可観測性に注意してください。ある Agent が自分で何をしたのか、なぜそうしたのか、どの段階で失敗したのかを説明できないなら、実際のアプリケーションにはなかなか使えません。

## AI 学習アシスタントプロジェクトとの関係

この段階は、AI 学習アシスタントの v0.9 学習計画 Agent に対応します。これは質問に答えるだけでなく、学習目標に合わせてタスクを分解し、章を提案し、学習計画を作成し、検索ツールを呼び出して資料を探し、各実行ステップを記録します。

最小機能のおすすめは、学習目標を入力すると段階計画を出力すること、コース検索ツールを呼び出して関連章を探すこと、今週のタスクリストを生成すること、trace を記録することです。標準版では、長期学習記録、復習リマインド、間違いノートの整理、手動確認を追加できます。

## よくあるつまずき

よくある問題には、ツールの説明が不十分で呼び出しが失敗する、Agent が無限ループになる、タスク分解が細かすぎるまたは粗すぎる、記憶が汚染される、ユーザー入力をそのままツールに渡して安全リスクが発生する、ログがなくて問題を調査できない、などがあります。Agent プロジェクトでは、権限の境界と停止条件は、機能そのものと同じくらい重要です。


## ライト版 / 標準版 / チャレンジ版のタスク

| 難易度 | やること | 向いている人 |
|---|---|---|
| ライト版 | Agent に 3 つの固定タスクを実行させ、手順を表示する | 初めて学ぶ人、時間が少ない人、入門したばかりの学習者 |
| 標準版 | agent_traces と tool_calls を保存する | この段階を作品集に入れたい学習者 |
| チャレンジ版 | 権限外テスト、手動確認、trace replay を追加する | 基礎があり、より強いプロジェクトの証拠を作りたい学習者 |

## この段階のバッジと Boss 戦

| 種類 | 内容 |
|---|---|
| Boss 戦 | 無限ループの魔王 |
| 解放できるバッジ | Trace 記録係、Agent 安全担当 |
| 最小合格スローガン | まず動かす、次に説明する、最後に失敗を記録する |
| 証拠保存のおすすめ | スクリーンショット、ログ、失敗サンプル、評価表を `reports/`、`evals/`、`logs/` に保存する |

ライト版を終えれば次に進めます。標準版を終えたら、作品集に載せることをおすすめします。チャレンジ版は、余裕があるときだけ取り組みましょう。

まず一度ガイド付きで動かしたい場合は、[実践：追跡可能な単一 Agent アシスタントを作る](./ch10-projects/04-stage-hands-on-workshop.md) を完了してください。そこで作る `agent_workshop.py`、`logs/agent_traces.jsonl`、小さな評価出力を、このステージの最小証拠として使えます。

## 段階作品集の提出物

この段階の成果を作品集として残したいなら、少なくとも次のファイル、または同等の資料を残すことをおすすめします。

| 提出物 | 説明 |
| --- | --- |
| `tools_schema.md` | 各ツールの名前、用途、パラメータ、戻り値、使用範囲 |
| `agent_traces.jsonl` | 各実行の goal、step、action、arguments、observation、next_decision |
| `safety_boundary.md` | ツールのリスク分類、手動確認ルール、最大ステップ数、権限制限 |
| `human_approval_examples.md` | 高リスク操作の確認文の例。操作、対象、リスク、ロールバック方法を含む |
| `failure_cases.md` | ツール選択ミス、パラメータエラー、ループ呼び出し、未対応の参照、安全ブロックなどの失敗サンプル |
| `README.md` | プロジェクトの目的、ツール一覧、実行方法、trace 例、評価結果、制限説明 |

これらの資料があると、あなたの Agent プロジェクトは「ツールを呼び出せる」から「追跡可能で、制御できて、振り返りもできる」システムへと進化します。

## 段階の合格問題

学習後には、次の質問に答えられるようになっているはずです。Agent と普通の LLM アプリの違いは何か、なぜツール説明が重要なのか、どのタスクが Agent に向いているのか、どうやってループ呼び出しを防ぐのか、なぜ trace、replay、手動確認が必要なのか。
