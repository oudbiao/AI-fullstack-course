# よくあるエラーとトラブルシューティング索引

AI 学習でつまずきやすいのは、概念そのものよりも、環境、依存関係、データ形式、API、モデルの入出力、そして実務上の境界です。トラブルシューティングでは、いきなりエラーメッセージをコピーして検索するのではなく、まず問題がどの層にあるかを見分けましょう。環境、コード、データ、モデル、検索、ツール呼び出し、デプロイ、キャッシュのどれかを考えます。

## 1枚でわかる：トラブルシューティングは順番を飛ばさない

```mermaid
flowchart LR
  A["完全なエラーを保存する"] --> B["ディレクトリと環境を確認する"]
  B --> C["最小入力で再現する"]
  C --> D["失敗した層を見分ける"]
  D --> E["修正して README に戻す"]
```

| まず1つだけやるなら | 何をするか |
|---|---|
| コマンドエラー | まず現在のディレクトリと実行コマンドを確認する |
| Python エラー | まず Python の実行環境と依存関係を確認する |
| RAG の回答が外れる | まず検索結果を出力して、いきなり Prompt を直さない |
| Agent が暴走する | まずステップ数を制限して trace を保存する |

## トラブルシューティングの全体フロー

エラーが出たら、次の順番で確認します。まず完全なエラー内容と実行コマンドを保存し、次に現在のディレクトリと実行環境を確認します。そのあと最小入力で問題を再現し、最後に依存関係、パス、データ、パラメータ、またはビジネスロジックのどこに原因があるかを特定します。解決方法は、学習ログやプロジェクト README に必ず書き戻しましょう。

最後の1行だけを見ないでください。Python、Node、Docusaurus、RAG、Agent のエラーでは、エラーメッセージの途中に重要な手がかりがあることが多いです。たとえば、実際に読んだパス、使われた Python 実行環境、要求したモデル名、ツールパラメータの schema、またはヒットした文書断片などです。

## 環境とツールのエラー切り分けツリー

| 見えている現象 | まず確認すること | 最小再現方法 | 参照し直す内容 |
| --- | --- | --- | --- |
| `command not found` またはコマンドが見つからない | 現在のターミナル、PATH、依存関係のインストール有無、プロジェクトのルートにいるか | `pwd`、`which コマンド名` を実行し、package.json や requirements を確認する | 開発者ツールの基礎、環境準備 |
| `ModuleNotFoundError` / `ImportError` | 正しい Python 環境が有効か、pip が同じ実行環境に入っているか | `python -c "import パッケージ名"` と `python -m pip show パッケージ名` を実行する | Python 環境、パッケージ管理 |
| `docusaurus: command not found` | `npm install` を実行したか、プロジェクトのルートで npm scripts を実行しているか | `docusaurus` を直接実行せず、`npm run start` を実行する | 環境準備、コースの保守フロー |
| Docusaurus のページが古いまま | `.docusaurus`、`build`、ブラウザキャッシュ、起動ディレクトリ | `npm run clean` のあとに再起動する | このページ、コースの保守フロー |
| Git push が失敗する | リモート URL、権限、ネットワークプロキシ、ブランチ名、ログイン状態 | `git remote -v` と `git branch --show-current` を実行する | Git とバージョン管理 |

## Python とデータのエラー切り分けツリー

| 見えている現象 | まず確認すること | 最小再現方法 | 参照し直す内容 |
| --- | --- | --- | --- |
| ファイルが読めない | 現在の作業ディレクトリ、相対パス、ファイル名の大文字小文字、エンコーディング | `pwd` を実行し、最小スクリプトで `open("パス")` を試す | Python のファイル読み書き |
| JSON の解析に失敗する | ファイルが正しい JSON か、1 行 1 JSONL か、文字コードに問題がないか | 少量のデータをコピーして `json.loads` で個別に試す | Python のファイルとデータ構造 |
| DataFrame に列が存在しない | 列名の空白、大文字小文字、header 行、読み込み時の区切り文字 | `df.columns.tolist()` を出力する | Pandas のデータ読み込みとクリーニング |
| グラフの中国語が文字化けする | フォント設定、保存形式、実行環境 | 最小の中国語タイトル付きグラフで試す | データ可視化 |
| SQL の検索結果が空 | テーブル名、フィルター条件、接続先 DB、トランザクションが commit されているか | まず最初の 5 行を取得し、そのあと少しずつ条件を追加する | データベース基礎 |

## 機械学習と深層学習のエラー切り分けツリー

| 見えている現象 | まず確認すること | 最小再現方法 | 参照し直す内容 |
| --- | --- | --- | --- |
| モデルのスコアが異常に高い | データリークがないか、訓練セットとテストセットが重複していないか、特徴量に答えが入っていないか | 少数の特徴量だけで baseline を再学習する | 特徴量エンジニアリング、モデル評価 |
| 訓練スコアは高いが検証が悪い | 過学習、データ量、正則化、データ拡張、分割方法 | ランダムシードを固定し、訓練/検証カーブを比較する | 深層学習の訓練テクニック |
| モデルが収束しない | 学習率、データの正規化、ラベル形式、loss がタスクに合っているか | 少量データで過学習できるか試す | PyTorch の訓練ループ |
| PyTorch の shape mismatch | batch 次元、channel 次元、系列長、loss の入力形式 | 各層で tensor shape を出力する | PyTorch の基礎、CNN/Transformer |
| GPU が動かない | ドライバ、CUDA、PyTorch のバージョン、device 設定 | `torch.cuda.is_available()` を実行する | 深層学習環境 |

## LLM と RAG のエラー切り分けツリー

| 見えている現象 | まず確認すること | 最小再現方法 | 参照し直す内容 |
| --- | --- | --- | --- |
| API 呼び出しに失敗する | API Key、モデル名、利用上限、ネットワーク、リクエスト形式、タイムアウト | いちばん短い prompt で単独リクエストする | LLM API 実践 |
| 出力形式が安定しない | Prompt で schema を明確にしているか、検証と再試行があるか | モデルに最小の JSON だけを返させる | 構造化出力 |
| RAG が質問に答えていない | query、chunk、embedding、top-k、rerank、文書範囲 | 検索結果だけを出力し、モデルは呼ばない | RAG 基礎と検索戦略 |
| 関連文書が見つからない | 文書が取り込まれているか、metadata が保持されているか、chunk が大きすぎないか小さすぎないか | 元文のキーワードで直接検索する | 文書処理とベクターデータベース |
| 引用の信頼性が低い | 答えが本当に一致した文書で裏付けられているか、引用パスが記録されているか | 答えの文と引用部分を比較する | RAG 評価 |
| コストが急に増えた | コンテキストが長すぎないか、再試行が多すぎないか、ループ呼び出しがないか、モデル選択は適切か | token、リクエスト数、各ステップの所要時間を記録する | LLM の工程化 |

## Agent のエラー切り分けツリー

| 見えている現象 | まず確認すること | 最小再現方法 | 参照し直す内容 |
| --- | --- | --- | --- |
| Agent が無限ループする | 停止条件、最大ステップ数、ツールの返却内容が明確か、目標が広すぎないか | 3 ステップに制限して、各ステップの計画を出力する | Agent の推論と計画 |
| ツールパラメータが間違っている | schema のフィールド名、型、必須項目、例がわかりやすいか | ツールパラメータ呼び出しを手で 1 回書いてみる | ツール呼び出しと Function Calling |
| ツール呼び出しは成功するが結果が使えない | ツールの返却形式、エラーコード、空結果の扱い | 固定入力でツールを呼び、元の返り値を出力する | Agent のツール戦略 |
| 記憶が汚染される | エラー情報や一時的な文脈を長期保存していないか | 記憶を空にして同じタスクを再実行する | Agent の記憶設計 |
| 高リスク操作が制御できない | 人間の確認、権限制限、監査ログがあるか | ツールを dry-run モードに変えて試す | Agent の安全性と Guardrails |
| 失敗を振り返れない | trace、ツールの入出力、状態、エラーを記録しているか | 1 回の失敗タスクを replay する | Agent の可観測性 |

## デプロイと工程化のエラー切り分けツリー

| 見えている現象 | まず確認すること | 最小再現方法 | 参照し直す内容 |
| --- | --- | --- | --- |
| ローカルでは動くのに、デプロイすると動かない | 環境変数、ファイルパス、依存バージョン、起動コマンド | クリーンな環境で README に従って再実行する | Docker とデプロイ |
| ポートにアクセスできない | サービスが起動しているか、ポートマッピング、ファイアウォール、host 設定 | ローカルから health check API を curl する | API 設計とデプロイ |
| ログに重要なエラーが出ない | ログレベル、例外キャッチ、request ID | わざとエラーを起こしてログを確認する | ログと監視 |
| 本番とローカルで結果が違う | モデルバージョン、設定、データインデックス、キャッシュ | 重要な設定とバージョンを出力する | 工程化のベストプラクティス |

## トラブルシューティング記録テンプレート

問題が起きるたびに、次の内容を記録するのがおすすめです。継続して積み上げると、自分だけの実務知識ベースになります。

```md
## 問題タイトル

### エラーの現象
どんなコマンドを実行して、どんなエラーが見えたか。

### 完全なエラー内容
最後の1行だけでなく、重要なエラー部分を貼る。

### 再現手順
1. どのディレクトリに入るか
2. どんなコマンドを実行するか
3. どんな入力や設定を使うか

### 調査の流れ
環境、パス、依存関係、データ、パラメータ、ログのどこを確認したか。

### 最終的な解決方法
何をどう直したか、なぜ効いたのか。

### 次回どう防ぐか
README、スクリプト、検証コマンド、テストサンプルのどこを補うべきか。
```

トラブルシューティングの目的は、「今回のエラーを早く押さえ込むこと」だけではありません。次に同じ種類のエラーが起きたとき、もっと簡単に切り分けられるようにすることです。ひとつでも本物のエラーを解決したら、それをプロジェクト文書、検証スクリプト、ログ項目、またはテストサンプルとして残しましょう。
