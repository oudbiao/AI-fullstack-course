# 教学と自学の使い方ガイド

![学習ペースカード](/img/course/intro-learning-pace-cards-ja.png)

同じコースでも、使い方によって進め方が変わります。順序はほぼ同じで、変わるのはペース、課題、振り返り方法です。

## 1. 使い方を選ぶ

| 方式 | 毎週の重点 | 証拠 |
|---|---|---|
| 自学 | 小さな結果を1つ動かし、振り返りを書く | README、コマンド、画像、失敗記録 |
| 講師主導 | 枠組みを教え、デモし、課題をレビューする | ノート、デモコード、評価基準 |
| 学習グループ | 読了報告ではなく成果物を共有する | 相互レビューした README と出力例 |

## 2. おすすめのリズム

| 手順 | 自学者 | 講師 | 学習グループ |
|---|---|---|---|
| 予習 | マップと目標を見る | 誤解されやすい点を選ぶ | 役割を決める |
| 練習 | 最小コードを動かす | 一度デモし、学習者に変更させる | 1人がデモする |
| プロジェクト | 小さな成果物を残す | 基本課題と挑戦課題を出す | 成果物をレビューする |
| 振り返り | 失敗を1つ記録する | 共通の失敗を話す | 詰まりを集める |

## 3. 課題ルール

よい課題は、動かせて、確認できて、振り返れます。「第X章を読む」だけにしないほうがよいです。

| ステージ | 課題の形 |
|---|---|
| Python | 入出力例付き CLI スクリプト |
| データ | 図表と結論がある Notebook またはレポート |
| ML/DL | 指標と失敗例がある baseline または学習実験 |
| RAG | 検索ログと引用付き Q&A デモ |
| Agent | trace と停止条件があるツール呼び出しタスク |

## 4. 受け入れルール

![ポートフォリオ受け入れ階段](/img/course/intro-portfolio-acceptance-ladder-ja.png)

プロジェクトレビューでは、動くか、他の人が再現できるか、結果を評価できるか、失敗と次の一手を説明できるかを確認します。

コースを更新するときは、マップ、プロジェクト、評価基準、よくある失敗を優先します。新しいモデル名を追うだけで主線を乱さないでください。
