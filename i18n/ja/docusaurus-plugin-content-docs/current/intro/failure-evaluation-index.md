# 失敗ケースと評価テンプレート索引

AI プロジェクトは、成功例だけを見せればよいわけではありません。作品集に近づくほど、「どの入力でシステムが失敗したのか」「失敗はどの層に属するのか」「どう再現するのか」「修正後にどう回帰テストするのか」を説明できる必要があります。このページでは、授業全体に散らばっている失敗ケースと評価テンプレートを 1 つの索引にまとめ、プロジェクト作成時にすぐ参照できるようにします。

## まず図を見よう：失敗から評価材料へ

```mermaid
flowchart LR
  A["失敗を見つける"] --> B["具体的な層に分類する"]
  B --> C["証拠ファイルを保存する"]
  C --> D["テスト例を追加する"]
  D --> E["作品集の振り返りを更新する"]
```

| 手元にあるもの | まず確認する場所 |
|---|---|
| エラーとコマンド出力 | 環境、Python、デプロイ層 |
| 誤答や幻覚 | Prompt、RAG、引用チェック |
| ツールの誤操作や停止しない挙動 | Agent、ツール権限、trace |
| 指標の異常や効果の不安定さ | データ、モデル、評価セット |

## 失敗の層ごとに確認する

| 失敗の層 | よくある現象 | 優先して見直す内容 | 残しておくべき証拠 |
|---|---|---|---|
| 環境とツール | コマンドが見つからない、依存関係の衝突、Git コミット失敗 | 1 開発者ツール基礎、環境準備、トラブルシューティング索引 | コマンド記録、環境バージョン、エラー画面 |
| Python プログラム | ファイルパスの間違い、JSON 解析失敗、関数ロジックの混乱 | 2 Python プログラミング基礎 | 入出力例、例外ログ、修正記録 |
| データ分析 | 欠損、重複、外れ値により結論が信頼できない | 3 データ分析と可視化 | データ辞書、クレンジングログ、図表の説明 |
| 数学と指標 | 類似度、確率、loss、指標の説明が曖昧 | 4 AI 数学基礎、5 機械学習 | 小さな実験、指標の説明、振り返りメモ |
| 機械学習 | データリーク、過学習、baseline が不明確 | 5 機械学習 | train/test 分割、baseline、誤サンプル |
| 深層学習 | shape mismatch、loss が下がらない、メモリ不足 | 6 深層学習と Transformer | 学習ログ、曲線、設定ファイル |
| Prompt | JSON 解析失敗、フィールド欠落、ラベルのずれ | 7 大規模言語モデルの原理と Prompt | Prompt のバージョン、固定テストサンプル、schema |
| RAG | 検索できない、引用で根拠を示せない、回答の幻覚 | 8 LLM 応用と RAG | chunks、retrieval logs、eval questions |
| Agent | ツール選択ミス、ループ、権限超過、trace 欠落 | 9 AI Agent | tool schema、agent trace、安全境界 |
| マルチモーダル | OCR の誤り、生成の制御が難しい、著作権や肖像権のリスク | 10 コンピュータビジョン、11 自然言語処理、12 AIGC とマルチモーダル | 素材の出典、手動確認、書き出し制限 |
| デプロイ運用 | ローカルでは動くが本番で失敗する、コストが制御できない | 工程化、卒業プロジェクトガイド | `.env.example`、ログ、監視とレート制限の説明 |

## プロジェクト種類ごとに評価材料を準備する

| プロジェクト種類 | 最低限の評価材料 | 作品集レベルの評価材料 |
|---|---|---|
| Python 小ツール | 3 つのコマンド入出力例 | 正常、異常、空入力、ファイル破損などのテスト例 |
| データ分析プロジェクト | データ品質チェック表 1 枚 | データ辞書、クレンジング前後比較、結論の限界 |
| 機械学習プロジェクト | train/test 指標と baseline | 交差検証、誤サンプル、特徴リークチェック |
| 深層学習プロジェクト | loss 曲線と検証指標 | 設定記録、学習ログ、混同行列、失敗画像またはテキスト |
| Prompt プロジェクト | 固定入力と出力の比較 | Prompt バージョン表、schema 検証、回帰テストセット |
| RAG プロジェクト | 10 個の固定質問と出典確認 | gold_doc、gold_answer、citation_ok、失敗タイプ統計 |
| Agent プロジェクト | 3～5 個の固定タスク | 完了率、平均ステップ数、ツールエラー率、権限超過テスト、コスト |
| マルチモーダルプロジェクト | 1 つの素材から出力までの完全な事例 | 成功、失敗、境界ケース、手動編集と確認記録 |
| 卒業プロジェクト | 20～50 個の固定テスト質問またはタスク | baseline、最適化記録、失敗の要因分析、デモスクリプト |

## 推奨ファイル名

各プロジェクトでは、少なくとも 1 つの `reports/` または `evals/` ディレクトリを残すことをおすすめします。ファイル名を統一しておくと、後から作品集を整理しやすくなります。

```text
reports/
├── baseline.md
├── failure_cases.md
├── improvement_record.md
└── demo_notes.md

evals/
├── eval_questions.csv
├── prompt_eval_cases.csv
├── agent_tasks.jsonl
└── citation_check.csv

logs/
├── llm_calls.jsonl
├── retrieval_logs.jsonl
├── agent_traces.jsonl
└── tool_calls.jsonl
```

## 失敗サンプルの最小フォーマット

```md
## 失敗サンプルのタイトル

- 入力：失敗を引き起こした実際の入力
- 期待：本来起こるべきこと
- 実際：システムが実際に出力または実行した内容
- 層：環境 / データ / モデル / Prompt / RAG / Agent / デプロイ
- 証拠：ログ、スクリーンショット、trace、検索断片、またはコード位置
- 原因：現時点で最も可能性が高い説明
- 修正：何をどう直す予定か
- 回帰：今後、再発防止に使うテスト例
```

失敗サンプルは長く書く必要はありませんが、必ず再現できることが大切です。再現できない失敗は、印象にすぎません。再現できて、原因を特定できて、回帰テストまでできる失敗こそ、作品集に載せるべき工学的な証拠です。
