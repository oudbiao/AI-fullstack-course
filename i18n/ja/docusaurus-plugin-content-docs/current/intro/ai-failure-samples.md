# AIアプリ失敗サンプル集

AIアプリ開発でいちばん多い問題は、「まったく動かない」ことではなく、動いているように見えるのに、特定の入力で不安定になることです。失敗サンプル集の役割は、こうした問題を層ごとに記録し、どの層に戻って調べればよいかを分かるようにすることです。

このページは、各章の代わりになるものではなく、索引です。問題に出会ったら、まず失敗が LLM API、Prompt、RAG、Agent、ツール、安全、デプロイのどれに当たるかを判断し、その後で該当する章に戻って詳しく確認しましょう。

## 一目で分かる：失敗サンプルの調べ方

```mermaid
flowchart LR
  A["実際の入力を保存する"] --> B["期待結果と実際の結果を比べる"]
  B --> C["失敗した層を判断する"]
  C --> D["ログまたは trace を確認する"]
  D --> E["修正して回帰テストに追加する"]
```

| 失敗した層 | 最初に見るもの |
|---|---|
| LLM API | request_id、エラーコード、token、遅延 |
| Prompt | raw output、schema、固定テストサンプル |
| RAG | top-k の検索結果、chunk、metadata、引用 |
| Agent | tool call、observation、max_steps、権限 |
| デプロイ | 環境変数、依存関係のバージョン、ログ、レート制限 |

## 失敗サンプルの記録方法

各失敗サンプルには、少なくとも次の項目を入れるのがおすすめです。

| 項目 | 説明 |
|---|---|
| ユーザー入力 | 失敗を引き起こした実際の質問やタスク |
| 期待結果 | 本来システムが出すべき出力、または実行すべき内容 |
| 実際の結果 | 実際にどう答えたか、どう動いたか |
| 失敗した層 | LLM API / Prompt / RAG / Agent / Tool / Safety / Deploy |
| 関連ログ | request_id、trace、retrieved_docs、tool_call など |
| 暫定原因 | もっとも可能性が高い問題点 |
| 修正内容 | 何を変える予定か |
| 回帰テスト | 今後同じ問題が再発しないことをどう確認するか |

ログがない失敗サンプルは振り返りが難しく、回帰テストがない失敗サンプルは次回また起きやすくなります。

## LLM API 層の失敗

| 現象 | 優先して確認するもの | 修正の方向 |
|---|---|---|
| リクエストがたまに失敗する | timeout、rate limit、ネットワークエラー、サーバーエラー | timeout を増やす、少数回の retry、エラー分類、fallback を追加する |
| コストが急に増えた | prompt_tokens、completion_tokens、コンテキスト長 | コンテキストを圧縮する、重複した履歴を減らす、usage を記録する |
| 遅延がはっきり遅くなった | モデル選択、top-k、retry 回数、ネットワーク状態 | retry を制限する、結果を cache する、同期処理と非同期処理を分ける |
| 出力が空、または形式がおかしい | raw output、エラーコード、レスポンス解析ロジック | 生のレスポンスを残す、レスポンス形式を統一する |
| どの呼び出しで問題が起きたか分からない | request_id とログ項目が足りない | model、prompt_version、latency、error を統一して記録する |

API 層の失敗は、まずログを見るべきで、いきなり Prompt を直すべきではありません。リクエストが成功したか、token をどれだけ使ったか、どれくらい時間がかかったかが分からないと、その後の原因特定が難しくなります。

## Prompt / 構造化出力の失敗

| 現象 | 優先して確認するもの | 修正の方向 |
|---|---|---|
| JSON の解析に失敗する | 余計な説明文がないか、括弧が閉じているか | JSON のみを出力するように指示する、解析失敗時の retry を追加する |
| フィールドが欠ける | schema の必須項目が明確か | Prompt に required フィールドを列挙する |
| フィールドの型が安定しない | 数値、boolean、配列が文字列になっていないか | 型と例を明確にする、validator を追加する |
| 分類ラベルがぶれる | enum の値が固定されているか | 許可する値を示し、勝手な分類を禁止する |
| Prompt を変えたら古いサンプルが悪化した | 固定のテストセットがない | Prompt の version 管理と回帰テスト用サンプルを作る |

Prompt の問題は、「もっと分かりやすく書く」だけでは解決しないことが多いです。より安定させるには、schema、例、検証、失敗サンプル、回帰テストをまとめて整えるのが効果的です。

## RAG 層の失敗

| 現象 | 優先して確認するもの | 修正の方向 |
|---|---|---|
| 知識ベースに答えがあるのにヒットしない | chunk、query、embedding、キーワード一致 | 切り分け方法を調整する、hybrid retrieval を追加する、または query rewrite を行う |
| 正しい資料が後ろの方に並ぶ | top-k の生結果、score、rerank 前後の比較 | rerank を追加する、hybrid retrieval の重みを調整する |
| 資料はヒットしたのに回答で条件を漏らす | context packing、prompt、回答形式 | 重要な条件を残す、1つずつ引用するように求める |
| 引用と回答が一致しない | source_refs、引用断片、claim | citation check を行う、根拠のない結論を禁止する |
| 版や出典が違う | metadata filter、source_origin、日付項目 | フィルタ条件と出典の優先順位を追加する |

RAG の失敗で大切なのは、「そもそも正しく見つけられていない」のか、「見つけたのにうまく使えていない」のかを分けることです。前者は検索ログ、後者は context と生成の制約を見ます。

## Agent / ツール層の失敗

| 現象 | 優先して確認するもの | 修正の方向 |
|---|---|---|
| Agent が間違った tool を選ぶ | tool 名、description、候補 tool の数 | schema を見直す、不要な tool を減らす、tool 選択ログを追加する |
| tool の引数が間違っている | arguments、引数の検証、エラー返却 | validator と構造化エラーを追加する |
| Agent がずっとループする | max_steps、停止条件、observation が重複していないか | 最大 step 数を設定する、新しい情報がないときは停止する |
| tool が失敗するとすぐ落ちる | safe_dispatch、retryable、fallback | retry 可能なエラーと不可能なエラーを分ける |
| 最終結果の理由が説明できない | trace が不足している | goal、step、action、arguments、observation、next_decision を記録する |

Agent の失敗は、最終回答だけ見ても分かりません。大事なのは、各 step でなぜその行動を選んだのか、そして observation が次の判断にどう影響したのかを見ることです。

## 安全・権限の失敗

| 現象 | 優先して確認するもの | 修正の方向 |
|---|---|---|
| Agent がやるべきでない操作を実行した | tool の権限、リスク分類、手動確認 | 高リスク tool は既定で確認を入れる、または無効化する |
| 外部ドキュメントに誘導されて権限を越えた | 外部内容を指示として扱っていないか | 外部内容を信頼しないようにマークする、tool 層で権限制限を行う |
| ログに機密情報が漏れる | ログ項目、マスキング方針 | API key、個人情報、内部資料をマスキングする |
| ユーザーがリスクを本当に確認していない | 確認文が明確か | 操作内容、対象、引数、リスク、ロールバック方法を表示する |
| 問題発生後に責任追跡できない | audit_log が不足している | request_id、tool 名、引数、確認者、結果を記録する |

安全の問題は、モデルの「自覚」に頼るだけでは足りません。システム側で、権限、ホワイトリスト、確認、監査、ロールバックを使って境界を守る必要があります。

## デプロイと本番運用の失敗

| 現象 | 優先して確認するもの | 修正の方向 |
|---|---|---|
| ローカルでは動くのに本番で失敗する | 環境変数、依存関係のバージョン、パス、権限 | デプロイ設定を明確に書く、`.env.example` を使う |
| API key が無効、または漏えいした | 秘密情報の管理、ログ、フロントエンド露出 | バックエンド経由で呼び出す、ログをマスキングする、key をローテーションする |
| 高並列時に不安定になる | rate limit、queue、timeout、retry | rate limit を追加する、非同期タスクと degrade 戦略を入れる |
| 本番コストが制御できない | token 集計、cache、ユーザーごとの上限 | コスト監視、リクエスト上限、モデルの段階分けを行う |
| ユーザーのフィードバックが改善に入らない | フィードバック項目とサンプル収集がない | thumbs、修正テキスト、失敗サンプルを保存する |

デプロイの失敗は、単なるコードの問題ではなく、設定、権限、runtime、監視が一緒に決めるものです。

## 失敗サンプルの振り返りテンプレート

```md
## 失敗サンプルのタイトル

### ユーザー入力

### 期待結果

### 実際の結果

### 失敗した層

### 関連ログ

### 暫定原因

### 修正内容

### 回帰テスト

### 解決済みかどうか
```

各フェーズのプロジェクトでは、少なくとも 3 個の失敗サンプルを残すのがおすすめです。ポートフォリオ級のプロジェクトでは、修正前がどうだったか、修正後に指標やサンプルがどう変わったか、まだ解決していない失敗は何かまで示せるとよいです。
