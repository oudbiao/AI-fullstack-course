# AI フルスタック用語集

この用語集は暗記するためのものではありません。授業を読み進める中で知らない言葉に出会ったときに、「これは何を意味するのか」「最初にどこで理解すればよいのか」「何と混同しやすいのか」をすぐに確認するためのものです。ある概念がまだよく分からなくても、まずはそれがどんな問題を解決するのかをつかみ、そのあとで該当する章に戻って細かい内容を補いましょう。

## 開発基礎

| 用語 | 簡単な説明 | 最初におすすめの理解場所 |
| --- | --- | --- |
| ターミナル | コマンドでシステムとやり取りする入口 | 開発者ツールの基礎 |
| 現在のディレクトリ | コマンドを実行している場所。多くのパスのエラーはここに関係します | ターミナルとコマンドライン |
| パッケージマネージャー | pip、npm、conda などの依存関係をインストール・管理するツール | 開発環境の設定 |
| 仮想環境 | プロジェクトごとに Python の依存関係を分ける環境 | Python 環境 |
| Git | コードのバージョン変化を記録するツール | Git とバージョン管理 |
| Commit | ひとまとまりの、あとから追跡できるコード変更記録 | Git の基本フロー |
| README | プロジェクトの説明ファイル。何のプロジェクトか、どう実行するか、どう確認するかを伝える | 段階的プロジェクトと作品集 |
| API | プログラム同士がデータをやり取りしたり機能を呼び出したりするためのインターフェース | Python プロジェクト、LLM API 呼び出し |
| JSON | よく使われる構造化データ形式。API、設定、ログに向いている | Python ファイルと API の章 |

## データと機械学習

| 用語 | 簡単な説明 | 混同しやすい点 |
| --- | --- | --- |
| データセット | 分析、学習、評価に使うデータのまとまり | データセットは訓練セットと同じではありません。訓練セットはその一部です |
| 特徴 | モデルが予測や判断に使う入力情報 | 特徴は多ければよいわけではなく、質と情報漏えいのリスクのほうが重要です |
| ラベル | 監督学習でモデルに予測してほしい目標 | 分類ラベルと回帰の目標は異なります |
| 訓練セット | モデルに学習させるためのデータ | 最終的な性能の代表としては使えません |
| テストセット | 汎化能力を評価するためのデータ | 何度も使って調整すべきではありません |
| 検証セット | モデル選択やパラメータ調整に使うデータ | テストセットとは役割が違います |
| Baseline | 比較の出発点となる、最もシンプルなモデルやルール | 低レベルなやり方ではなく、改善が有効かを判断する基準です |
| データ漏えい | モデル学習時に、本来知るべきでない情報を見てしまうこと | オフライン指標が不自然に高くなり、実際の性能は下がります |
| 過学習 | モデルが訓練データを覚えすぎて、新しいデータにうまく一般化できないこと | 訓練スコアが高いからといって良いモデルとは限りません |
| 再現率 | 本当に関連するサンプルのうち、どれだけ見つけられたかの割合 | RAG の検索や分類タスクでよく使われます |
| F1 | 適合率と再現率をまとめた指標 | クラス不均衡では、正解率より参考になります |

## 深層学習と大規模言語モデルの基礎

| 用語 | 簡単な説明 | 最初におすすめの理解場所 |
| --- | --- | --- |
| テンソル | 深層学習フレームワークの基本データ構造である多次元配列 | PyTorch の基礎 |
| Loss | モデルの予測と正解の差 | ニューラルネットワーク学習の全体像 |
| 逆伝播 | 損失に基づいて、パラメータをどう調整すべきかを計算する प्रक्रिया | ニューラルネットワークの基礎 |
| Optimizer | 勾配に基づいてパラメータを更新するアルゴリズム | PyTorch の学習ループ |
| Embedding | テキスト、画像、カテゴリをベクトル表現に写像すること | NLP、RAG、ベクトル検索 |
| Attention | 入力のどの部分がより重要かをモデルが判断する仕組み | Transformer の基礎 |
| Transformer | 現代の大規模言語モデルの重要な基盤アーキテクチャ | 深層学習と Transformer |
| Token | モデルがテキストを処理するときの基本単位 | 大規模言語モデルの仕組みと Prompt |
| Context Window | モデルが一度に見られるコンテキストの長さ | LLM アプリ開発 |
| Pretraining | 大規模データで事前に汎用能力を学ぶこと | 大規模言語モデルの事前学習 |
| Fine-tuning | 特定のデータで追加学習し、タスクに合わせること | ファインチューニングの章 |
| Alignment | モデルの出力を人間の意図、安全性、ルールにより合うようにすること | アライメントの章 |

## LLM、Prompt、RAG

| 用語 | 簡単な説明 | 混同しやすい点 |
| --- | --- | --- |
| LLM | 大規模言語モデル。テキストを理解し生成できるモデル | LLM は完成したアプリではなく、能力の中核のひとつです |
| Prompt | モデルに渡す課題説明、入力、制約、出力形式 | Prompt は呪文ではなく、タスク設計です |
| System Prompt | モデルに役割、ルール、境界を設定する優先度の高い指示 | 安全の境界を通常のユーザー指示だけに任せてはいけません |
| Structured Output | モデルに JSON、表、固定フィールドなどの形式で出力させること | 検証と再試行を組み合わせる必要があります |
| Function Calling | モデルに tool schema に従って呼び出しパラメータを生成させること | これは tool を直接実行するものではなく、実行はあくまでプログラムが制御します |
| RAG | Retrieval-Augmented Generation。外部資料を検索してからモデルに回答させる方法 | RAG はベクトルデータベースそのものではありません。ベクトルDB はよく使われる構成要素のひとつです |
| Chunk | ドキュメントを分割した断片 | 大きすぎると精度が落ち、小さすぎると文脈が失われやすくなります |
| Vector Database | ベクトルを保存・検索するデータベース | 答えが正しいかどうかを判断する役割はありません |
| Hybrid Search | キーワード検索とベクトル検索を組み合わせる方法 | 正確な語と意味の近さの両方をバランスよく扱いたいときに向いています |
| Rerank | 一次検索結果を並べ替え直すこと | 最終的なコンテキストの質を上げるためによく使います |
| Citation | 回答が引用した出典 | 引用があっても、答えを本当に支えているとは限らないので確認が必要です |
| Hallucination | 一見もっともらしいが、信頼できない内容をモデルが生成すること | RAG で減らせますが、完全になくすことはできません |

## Agent とエンジニアリング

| 用語 | 簡単な説明 | 混同しやすい点 |
| --- | --- | --- |
| Agent | 目標に向けて計画し、tool を呼び出し、状態を記録できる AI ワークフロー | Agent は「よりおしゃべりな LLM」ではなく、システム設計です |
| Tool | Agent が呼び出せる外部機能。たとえば検索、計算、ファイル操作など | tool の権限は必ず管理する必要があります |
| Memory | Agent が短期または長期のコンテキストを保存する仕組み | 記憶は多ければよいわけではなく、汚染はリスクにつながります |
| Planning | 目標を手順に分解すること | 計画は実行可能で、停止できて、巻き戻せる必要があります |
| Trace | Agent の各ステップの入力、出力、tool、状態を記録したもの | trace がない Agent は原因調査が難しくなります |
| Replay | 過去の軌跡に基づいて実行過程を再現すること | デバッグと評価に使います |
| Guardrails | 入力、出力、tool、権限に保護の境界を設けること | モデルの自律性だけに頼ってはいけません |
| Human-in-the-loop | 重要なステップに人間の確認を入れること | 高リスク操作や元に戻せない操作に向いています |
| Observability | システムのログ、指標、trace、エラーを観察できる能力 | 本番後に追加するのではなく、プロジェクト中盤から意識すべきです |
| Deployment | プロジェクトをアクセス可能な環境に配置すること | デプロイできたからといって本番で使えるとは限らず、監視とロールバックも必要です |
| Evaluation | サンプル、指標、人手の基準を使ってシステムの良し悪しを判断すること | AI アプリの評価では、自動指標と人手レビューの組み合わせが必要になることが多いです |

## 用語集の使い方

知らない言葉に出会ったら、まずどのカテゴリに属するかを見てから、対応する段階の内容に戻って読んでください。ひとつの用語が分からないからといって、学習の流れ全体を止める必要はありません。最初の学習では、それが何を解決するのか、入力と出力は何か、今のプロジェクトとどう関係するのかだけ分かれば十分です。実際にプロジェクトで使う段階になったら、そのときに詳しい原理や実装を補いましょう。
