# Debug 事件捜査ミッション集

![Debug 事件捜査ミッション集](/img/course/debug-detective-missions.png)

Debug は学習の失敗ではなく、エンジニアリング学習が本当に始まる場所です。多くの初心者が退屈だと感じるのは、チュートリアルが成功する道しか見せないからです。でも実際のプロジェクトで最も価値のある力は、たいてい「なぜ期待どおりに動かなかったのか」から生まれます。

このページでは、よくあるエラーを探偵ミッションとしてまとめています。各ミッションには、事件、手がかり、容疑、捜査手順、真相、修復記録があります。まず自分で推理してから、提案された調査方向を確認してみましょう。

## まず、どの種類の事件かを見分ける

| 現象 | まず疑うこと |
|---|---|
| コマンドが見つからない、環境が一致しない | ディレクトリ、PATH、依存関係、仮想環境 |
| ファイルは開けるがプログラムが読めない | エンコード、形式、空ファイル、壊れた JSON |
| データ列が存在しない、または結論が変 | 列名、区切り文字、欠損、重複、リーク |
| モデルスコアが異常に高い、または異常に低い | 分割方法、baseline、指標、データ品質 |
| LLM の出力形式がずれる | Prompt の制約、schema、検証、再試行 |
| RAG の答えに出典がない | 検索、chunk、metadata、引用チェック |
| Agent が止まらない、または権限を越える | ツール権限、最大ステップ数、停止条件、人間の確認 |

## 事件捜査ミッションの進め方

事件に遭遇したら、すぐ答えを見ないでください。まず次の4つの手順で進めます。現象をそのまま保存し、2～3個の容疑を立て、最小限の実験で切り分け、最後に真相を修復記録として残します。

```mermaid
flowchart LR
  A[事件現場] --> B[手がかりを集める]
  B --> C[容疑を挙げる]
  C --> D[最小実験]
  D --> E[真相を特定]
  E --> F[失敗サンプルに記録]
```

各ミッションでは、`failure_cases.md`、`debug_notes.md`、`retrieval_logs.jsonl`、`agent_traces.jsonl` のような証拠ファイル、または README 内のトラブルシューティング記録を1つ残しましょう。

## 事件 1：消えたコマンド

| 項目 | 内容 |
|---|---|
| 事件現場 | ターミナルで `python`、`pip`、`npm`、`docusaurus` を入力すると command not found と表示される |
| 手がかり | 同じPCでも時々動き、時々動かない。ターミナルを変えると症状が違う |
| 容疑 | コマンド未インストール、PATH 未反映、プロジェクトのルートにいない、仮想環境が有効化されていない |
| 捜査手順 | `pwd`、`which python`、`python --version`、`npm --version` を実行し、現在のディレクトリとツールの場所を確認する |
| 真相の方向 | コードのロジックではなく、環境とパスの問題 |
| 修復証拠 | README にインストールコマンド、実行ディレクトリ、バージョンの説明を追加する |

この事件で鍛えるのは環境意識です。初心者の最初の関門は、複雑なコードを書くことではなく、自分がどのディレクトリにいて、どのインタプリタを使い、どのコマンドを実行しているかを把握することです。

## 事件 2：開けない JSON の宝箱

| 項目 | 内容 |
|---|---|
| 事件現場 | プログラムが `tasks.json` を読み込むと `JSONDecodeError` が出る |
| 手がかり | ファイルは存在するのに、開くと括弧が足りない、余分なカンマがある、または空ファイルになっている |
| 容疑 | 手動でファイルを壊した、書き込みが途中で止まった、JSONL を JSON として読んでいる |
| 捜査手順 | 最小スクリプトで `json.loads` を使って少量の内容を読み、ファイルが空でないか、形式が JSON か JSONL かを確認する |
| 真相の方向 | データファイルの形式が壊れており、プログラムに例外処理が足りない |
| 修復証拠 | 壊れたファイルへの注意メッセージ、バックアップ処理、再生成処理を追加する |

この事件で鍛えるのは入力防御です。実際のプロジェクトでは、ファイル、API、ユーザー入力のどれも壊れる可能性があり、プログラムは理想的な入力だけで動けばよいわけではありません。

## 事件 3：DataFrame 列名失踪事件

| 項目 | 内容 |
|---|---|
| 事件現場 | Pandas が `KeyError: 'minutes'` を出す、またはある列が存在しない |
| 手がかり | CSV を見るとその列はあるように見えるのに、プログラムでは読めない |
| 容疑 | 列名に空白がある、大文字小文字が違う、区切り文字が間違っている、header 行の読み取りを間違えた、エンコードが異常 |
| 捜査手順 | `df.columns.tolist()` を出力し、`df.head()` を確認し、`read_csv` の引数を見直す |
| 真相の方向 | データ読み込みとフィールドの規約に関する問題 |
| 修復証拠 | 列名のクリーニング、データ辞書、読み取り引数の説明を追加する |

この事件で鍛えるのは、データを最初に確認する習慣です。分析の前に列名、行数、欠損、サンプルを見ることは、いきなりモデルを作るより大事です。

## 事件 4：ありえないほど高いモデルスコア

| 項目 | 内容 |
|---|---|
| 事件現場 | モデルの正解率が 99% なのに、少しデータを変えると不安定になる |
| 手がかり | 特徴量に答えが含まれているかもしれない、train と test に重複がある、分割が間違った手順の後に行われている |
| 容疑 | データリーク、重複サンプル、目的変数が特徴量に入っている、評価用データが小さすぎる |
| 捜査手順 | train/test が先に分割されているか確認し、怪しい特徴を削除し、Dummy baseline を学習し、重複サンプルを比較する |
| 真相の方向 | スコアは信頼できず、モデルが本当に強いわけではない |
| 修復証拠 | リーク検査の記録、baseline 指標、失敗サンプルの分析 |

この事件で鍛えるのは評価への警戒心です。見た目の良いスコアが良いモデルを意味するとは限りません。スコアの理由を説明できて初めて力になります。

## 事件 5：JSON ずれ事件

| 項目 | 内容 |
|---|---|
| 事件現場 | LLM が、あるときは正しい JSON を出力するが、あるときは説明文を1つ足したり、フィールドを1つ省いたりする |
| 手がかり | Prompt には「JSON を出力してください」としか書かれておらず、フィールド schema、例、検証がない |
| 容疑 | 制約が弱い、出力の検証がない、失敗時の再試行がない、テスト入力が少なすぎる |
| 捜査手順 | 10個の入力を固定し、元の出力を保存し、JSON parser と schema で検証する |
| 真相の方向 | Prompt は一回きりの文章ではなく、テストが必要なコンポーネント |
| 修復証拠 | Prompt のバージョン表、schema 検証の通過率、失敗サンプル |

この事件で鍛えるのは Prompt 工学の信頼性です。一度うまくいっただけでは安定性は分かりません。固定のテストセットがあって、はじめて問題が見えます。

## 事件 6：RAG が証拠を見つけられない

| 項目 | 内容 |
|---|---|
| 事件現場 | ユーザーの質問は確かに文書にあるのに、RAG が的外れな答えを返すか、分からないと言う |
| 手がかり | 検索結果が正しい chunk に当たっていない、または当たっていても metadata が失われている |
| 容疑 | 文書が取り込まれていない、chunk が大きすぎる/小さすぎる、query の表現が合っていない、top-k が小さすぎる、embedding が適していない |
| 捜査手順 | いったん生成を止めて検索結果だけを表示する。元の文書のキーワードで検索し、chunk の内容と metadata を確認する |
| 真相の方向 | 生成の前に、検索がすでに失敗している |
| 修復証拠 | `retrieval_logs`、chunk のサンプル、`eval_questions`、失敗タイプの集計 |

この事件で鍛えるのは、RAG を段階ごとに切り分けて調べる力です。まず検索、それから生成、最後に引用を確認します。問題を全部モデルのせいにしてはいけません。

## 事件 7：引用ハルシネーション事件

| 項目 | 内容 |
|---|---|
| 事件現場 | RAG の答えは一見正しいが、引用された抜粋では答えを支えられていない |
| 手がかり | 答えはモデルの常識から来ていて、検索資料ではないかもしれない。引用は飾りに見える |
| 容疑 | Prompt で文ごとの根拠を求めていない、citation の粒度が粗い、検索抜粋と答えが対応していない |
| 捜査手順 | 各回答文に対して根拠となる抜粋を付け、`citation_ok` フィールドを追加する |
| 真相の方向 | 引用があることと、証拠で支えられていることは同じではない |
| 修復証拠 | `citation_check.csv`、失敗質問、修正後の Prompt |

この事件で鍛えるのは事実一致性です。ポートフォリオの RAG プロジェクトでは、答えだけでなく、引用チェックを示せることが大切です。

## 事件 8：Agent がその場でぐるぐる回る事件

| 項目 | 内容 |
|---|---|
| 事件現場 | Agent が同じ計画を繰り返し、同じツールを何度も呼び、なかなか終わらない |
| 手がかり | 各ステップの thought と observation がよく似ていて、目標の完了条件がない |
| 容疑 | タスクが広すぎる、停止条件が不明確、ツールの返答があいまい、最大ステップ数がない |
| 捜査手順 | 3ステップに制限して実行し、action、input、observation、done 判定を出力する |
| 真相の方向 | Agent に境界が足りないのであって、もっと自由が必要なわけではない |
| 修復証拠 | `agent_traces.jsonl`、停止条件の説明、失敗前後の trace 比較 |

この事件で鍛えるのは、制御できる Agent 設計です。Agent が行動できるほど、境界も必要になります。

## 事件 9：こっそり権限を越えるツール

| 項目 | 内容 |
|---|---|
| 事件現場 | Agent が、呼んではいけない削除、送信、書き込み、外部操作を実行した |
| 手がかり | ツールの説明にリスク等級がない、システムに人間の確認がない、テストが成功パスしか見ていない |
| 容疑 | ツールのホワイトリストが広すぎる、権限が段階化されていない、dry-run がない、監査ログがない |
| 捜査手順 | ツールを read-only、write、send、delete に分け、高リスクツールはデフォルトで人間の確認を入れる |
| 真相の方向 | 権限設計の問題であり、単一の Prompt だけでは完全には解決できない |
| 修復証拠 | ツール権限表、権限越えテストのサンプル、人間確認のスクリーンショットまたはログ |

この事件で鍛えるのは安全境界です。ポートフォリオの Agent プロジェクトで、権限越えをどう防ぐかを示せると、単に自動化を見せるより説得力があります。

## 事件 10：自分の PC では動くのに、他の人では動かない

| 項目 | 内容 |
|---|---|
| 事件現場 | 自分の PC では動くのに、同級生の環境やデプロイ環境では失敗する |
| 手がかり | README に依存関係、環境変数、データファイル、起動ディレクトリ、バージョンの説明が足りない |
| 容疑 | 暗黙の依存、ローカルの絶対パス、サンプルデータ未提出、`.env.example` がない |
| 捜査手順 | きれいなディレクトリで再度 clone し、README に従って最初から実行する |
| 真相の方向 | プロジェクトの納品が不完全であって、使う人が分かっていないわけではない |
| 修復証拠 | README、依存ファイル、`.env.example`、起動ログを更新する |

この事件で鍛えるのは納品意識です。自分の PC で動くのは第一歩にすぎません。ほかの人が再現できて、初めて作品集のプロジェクトになります。

## 探偵メモのテンプレート

```md
## 事件：RAG が証拠を見つけられない

### 事件現場
ユーザーが「Agent と RAG の違いは何ですか」と質問したが、システムの答えは一般的すぎて、正しい章を引用していない。

### 既知の手がかり
検索結果は RAG 章にしか当たっておらず、Agent 章には当たっていない。

### 容疑
文書の取り込み範囲が狭すぎる、metadata に stage 情報が保存されていない、query が言い換えられていない。

### 捜査手順
生成を止めて top-5 の検索結果だけを表示する。「Agent のツール呼び出し」と「エージェント」というキーワードで再検索する。

### 真相
インデックスには ch08-rag しか取り込まれておらず、ch09-agent は取り込まれていなかった。

### 修復
文書の取り込み範囲を広げ、metadata に stage と title を保存する。

### 回帰チェック
この問題を eval_questions.csv に追加する。
```

Debug 事件捜査ミッションの目的は、エラーを面白く、調べやすく、振り返りやすくすることです。事件を1つ解決するたびに、あなたは本物のエンジニアリング経験を1つ増やしていきます。
