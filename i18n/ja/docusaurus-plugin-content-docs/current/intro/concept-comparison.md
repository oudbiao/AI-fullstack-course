# 概念分類ボード

![概念分類ボード](/img/course/intro-concept-sorting-board-ja.png)

新しい用語を見たら、いきなり暗記しようとしなくて大丈夫です。まず分類し、そのあと短い意味を確認しましょう。

## 1. 覚える前に分類する

| 箱 | よく入る用語 | ひとことで言うと |
|---|---|---|
| 呼び出しと接続 | API、SDK、ライブラリ、フレームワーク、CLI、Web API | 自分のコードが外部機能につながる方法 |
| モデルとサービス | モデル、推論、学習、微調整、デプロイ | モデルを使うのか、変えるのか、サービス化するのか |
| LLM の流れ | Prompt、System Prompt、Few-shot、構造化出力、関数呼び出し、ツール利用 | モデルに指示し、必要ならツールにつなぐ方法 |
| データとベクトル | Token、Embedding、Chunk、Metadata、ベクトルDB、コンテキストウィンドウ | 内容を分け、表し、検索し、モデルに渡す方法 |
| エンジニアリング | ログ、評価、遅延、コスト、セキュリティ、監視 | デモを安定したプロジェクトにできるか |

## 2. よく出る用語を一行で

| 用語 | 簡単な意味 | 混同しないもの |
|---|---|---|
| API | サービスを呼び出すための約束 | SDK。SDK は呼び出しを楽にする道具 |
| SDK | 開発用ツールキット | サービスそのもの |
| モデル | 学習済みパラメータで予測や生成をする中心部分 | その外側の API |
| 推論 | モデルを使って出力を得ること | 学習。学習はパラメータを変える |
| 微調整 | 既存モデルをさらに学習させること | Prompt の修正 |
| Prompt | 今回の入力とタスク説明 | 私有知識の保存場所 |
| RAG | 外部文書を検索し、文脈として渡して答える流れ | モデルの微調整 |
| Agent | 計画し、複数手順を進め、ツールを使えるシステム | 1 回 API を呼ぶだけのチャットボット |
| Embedding | テキストや画像をベクトルで表すこと | ベクトルを保存するデータベース |
| Chunk | 検索用に切り出した文書片 | 元の文書全体 |

## 3. 3つの判断

| 問題が... | まず使うもの | 理由 |
|---|---|---|
| 出力形式が安定しない | Prompt と構造化出力 | タスクの境界がまだ曖昧 |
| モデルが自分の資料を知らない | RAG | 足りないのは外部知識 |
| 何度も行動する必要がある | ワークフローまたは Agent | システムが判断や実行を行う必要がある |

初学者は、まず Prompt を直し、知識が足りないときに検索を足し、本当に外部操作が必要なときにツール呼び出しを足す、という順番で考えると迷いにくくなります。

## 4. ミニ練習

次の文を読んで、どの分類か考えてみましょう。

| 状況 | 正しい分類 |
|---|---|
| 「回答が違う段落を引用している。」 | データとベクトル、そのあと RAG 評価 |
| 「JSON の項目がときどき欠ける。」 | LLM の流れ |
| 「ローカルでは動くが、デプロイ後に失敗する。」 | エンジニアリング |

README を書くときも、使ったモデルサービス、API または SDK、RAG やツールの有無、評価方法、既知の限界をはっきり書きましょう。
