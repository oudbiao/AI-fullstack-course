# プロジェクトページ納品基準

コース内のプロジェクトページは、学習者に「何を作るか」だけでなく、「どんな証拠を提出するか」も伝える必要があります。実行方法、サンプル入出力、評価方法、失敗例がないと、ただの練習で終わってしまい、作品集にはなりにくいです。

## 一枚でわかる：プロジェクトページ納品の閉ループ

```mermaid
flowchart LR
  A["プロジェクト目標"] --> B["実行方法"]
  B --> C["サンプル入出力"]
  C --> D["評価方法"]
  D --> E["失敗例"]
  E --> F["次の計画"]
```

| まず 3 つだけ補うなら | 理由 |
|---|---|
| 実行方法 | 他の人が再現できるようにするため |
| サンプル入出力 | プロジェクトの実際の結果を伝えるため |
| 失敗例 | システムの限界を理解していることを示すため |

## プロジェクトページの推奨構成

| モジュール | 何に答えるか | 最低要件 |
|---|---|---|
| プロジェクト目標 | このプロジェクトは何を解決するか | ユーザー入力、システム出力、利用シーンを説明する |
| 実行方法 | 他の人はどう再現するか | 依存関係のインストール、実行コマンド、環境変数の説明を示す |
| サンプル入出力 | プロジェクトは実際にどう見えるか | 少なくとも 1 つの実際の入力と対応する出力を示す |
| プロジェクト構成 | ファイルはどう整理されているか | コアソース、データ、ログ、レポートのディレクトリを列挙する |
| 方法説明 | なぜこの方法なのか | データ、モデル、Prompt、RAG、Agent、またはツールの選択理由を説明する |
| 評価方法 | どうやって効果を判断するか | 指標、baseline、テストセット、または手動チェック基準を明確に書く |
| 失敗例 | どこがまだ不安定か | 少なくとも 1～3 件の失敗例と考えられる原因を記録する |
| 次の計画 | どう改善するか | 次の版で何を変えるか、その理由を示す |

プロジェクトがコース後半に近づくほど、工程全体の閉ループを強く意識すべきです。RAG プロジェクトでは検索ログと引用チェックが必要で、Agent プロジェクトでは trace と権限の境界が必要です。多モーダルプロジェクトでは素材の出どころと確認用チェックリストが必要です。

## README の最低合格ライン

各段階のプロジェクト README は、少なくとも次の質問に他の人が答えられる状態にしておくべきです。つまり、このプロジェクトは何をするのか、どう実行するのか、入出力は何か、現在の効果はどうか、どこで失敗するのか、次にどう改善するのか、です。

~~~md
# プロジェクト名

## プロジェクト目標

## 実行方法

## サンプル入出力

## プロジェクト構成

## 方法説明

## 評価方法

## 失敗例

## 次の計画
~~~

プロジェクトが LLM、RAG、または Agent を含む場合は、さらに Prompt のバージョン、モデル呼び出し設定、検索設定、ツール schema、ログフィールド、安全境界も説明する必要があります。

## プロジェクト種類ごとの追加要件

| プロジェクト種類 | 追加の証拠 | よくある不合格サイン |
|---|---|---|
| Python 小型ツール | コマンドライン引数、ファイル入出力の例、例外処理 | 作者の PC でしか動かず、入力例がない |
| データ分析プロジェクト | データ辞書、クレンジング記録、グラフの結論、制約 | 図だけで、説明やデータ品質の記載がない |
| 機械学習プロジェクト | baseline、指標、データ分割、誤り分析 | 最高点だけ報告し、評価方法を説明していない |
| 深層学習プロジェクト | 学習曲線、設定、checkpoint、失敗例 | loss のスクリーンショットはあるが、実験を再現できない |
| Prompt プロジェクト | Prompt バージョン、固定テストサンプル、構造化チェック | 1 回の成功出力だけを見せている |
| RAG プロジェクト | chunks、retrieval logs、eval questions、citation check | 答えは正しそうでも、引用で裏付けられていない |
| Agent プロジェクト | tools schema、agent traces、max_steps、手動確認 | Agent が何をしたか再生できない |
| 多モーダルプロジェクト | 素材の出どころ、Prompt バージョン、確認用チェックリスト、出力結果 | 生成画像だけを見せていて、出どころや利用範囲を説明していない |

## プロジェクト提出前チェックリスト

段階的なプロジェクトを提出する前に、README は単独で読めるか、実行コマンドはそのまま使えるか、サンプル入出力は本物か、評価方法は明確か、失敗例は記録されているか、プロジェクトの制約は書かれているか、次の計画は具体的かを、項目ごとに確認することをおすすめします。

もしプロジェクトがまだ完全でなくても、「最小閉ループ」の形で先に提出してかまいません。大事なのは、毎回の反復で証拠を残すことです。つまり、今回何を追加したか、何を検証したか、どこで失敗したか、そして次になぜそのように改善するのか、を記録することです。
