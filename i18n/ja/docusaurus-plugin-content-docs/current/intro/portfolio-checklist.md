# ポートフォリオ受け入れチェックリスト

![ポートフォリオ受け入れ階段](/img/course/intro-portfolio-acceptance-ladder-ja.png)

ポートフォリオに必要なのは、多すぎる機能ではありません。動くこと、他の人が再現できること、評価できること、選択理由を説明できることです。

## 1. 受け入れ階段

| レベル | 最低限の証拠 | まだ弱い状態 |
|---|---|---|
| 練習 | ローカルで動き、結果が出る | スクリーンショットしかない |
| プロジェクト | README、実行コマンド、入出力例がある | 他の人が再現できない |
| ポートフォリオ | 評価セット、ログ、失敗サンプル、デモ画像がある | 成功例しか見せていない |
| 面接 | 構成、指標、限界、コスト、安全性、代替案を説明できる | なぜその選択をしたか説明できない |

各ステージのプロジェクトは、少なくともプロジェクトレベルを目指します。卒業制作はポートフォリオレベルまで上げたいところです。

## 2. 1分チェック

共有する前に、次の8項目を確認します。

| 項目 | 合格の目安 |
|---|---|
| README | 目的、ユーザー、機能一覧が明確 |
| 実行コマンド | 初めて読む人が最小フローを動かせる |
| 依存関係 | `requirements.txt`、`pyproject.toml`、または `package.json` がある |
| 例 | 1-3個の実際の入出力がある |
| スクリーンショットやデモ | 何をするプロジェクトか見て分かる |
| 評価 | 固定の質問セット、指標、または手動採点ルールがある |
| 失敗サンプル | うまくいかない例と原因が記録されている |
| 次の一手 | 「あとで改善」ではなく、具体的な改善が書かれている |

3項目以上欠けているなら、まだ練習コードとして磨きましょう。

## 3. AIプロジェクト別の証拠

| 種類 | 残す証拠 |
|---|---|
| 機械学習 | Baseline、訓練/テスト分割、指標、エラー例 |
| 深層学習 | 学習ログ、検証曲線、過学習チェック |
| Prompt アシスタント | Prompt の版、入出力例、失敗例 |
| RAG | 文書ソース、分割方針、検索ログ、引用チェック |
| Agent | ツール schema、実行 trace、停止条件、権限境界 |
| マルチモーダル | 入力形式、処理フロー、人の確認、品質基準 |

RAG の中心は、きれいな画面ではなく、検索と引用の信頼性です。Agent の中心は、手順の多さではなく、追跡できて制御できる実行です。

## 4. 3分デモの順番

発表ではこの順に話します。

1. ユーザーは誰で、何に困っているか。
2. 最小機能は何をするか。
3. AI はどこで使われているか。
4. 出力をどう検証したか。
5. 以前どこで失敗し、何を変えたか。
6. 次に何を改善するか。

## 5. 最小 README 構成

```md
# プロジェクト名

## 目的

## 実行方法

## 入出力例

## 評価

## 既知の失敗

## 次の一手
```

長くて曖昧な説明より、短くて再現できる説明のほうが強いです。
