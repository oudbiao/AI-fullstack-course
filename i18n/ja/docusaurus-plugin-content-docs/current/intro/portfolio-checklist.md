# ポートフォリオ確認チェックリスト

このチェックリストは、あなたのステージプロジェクトや卒業プロジェクトが、まだ「練習コード」なのか、それとも「見せられる作品」になっているのかを確認するためのものです。ポートフォリオ用のプロジェクトは、機能が多い必要はありませんが、必ず動かせて、説明できて、評価できて、振り返れて、さらに改善できる必要があります。

## 一枚でわかる：ポートフォリオ確認レベル

```mermaid
flowchart LR
  A["練習レベル：ローカルで動く"] --> B["プロジェクトレベル：他人が再現できる"]
  B --> C["ポートフォリオレベル：評価と展示ができる"]
  C --> D["面接レベル：判断基準と境界を説明できる"]
```

| レベル | 重点となる証拠 |
|---|---|
| 練習レベル | コードと基本出力 |
| プロジェクトレベル | README、実行コマンド、入力と出力の例 |
| ポートフォリオレベル | 評価、ログ、失敗サンプル、スクリーンショットやデモ |
| 面接レベル | アーキテクチャ、指標、制約、コスト、安全性、代替案 |

## 1分でできる簡易チェック

時間がとても少ないなら、まず次の8項目を確認してください。プロジェクトに README があるか、README に実行コマンドがあるか、コマンドどおりに他人が実行できるか、入力と出力の例があるか、スクリーンショットやデモがあるか、評価方法があるか、失敗サンプルがあるか、次の改善計画があるか。

この8項目のうち3項目以上が欠けているなら、そのプロジェクトはまだポートフォリオに入れる段階ではないことが多いです。役に立つ練習ではあっても、まだ展示できる成果にはなっていません。

## README チェック

| チェック項目 | 合格基準 |
| --- | --- |
| プロジェクト背景 | ユーザーが誰か、問題は何か、なぜ AI が必要かを説明できる |
| 機能一覧 | すでに完成した機能を列挙できる。計画だけを書かない |
| 実行方法 | 明確なコマンド、依存関係のインストール方法、環境説明がある |
| 入力と出力の例 | 少なくとも1〜3個の実例がある |
| プロジェクト構成 | 主要なディレクトリとファイルの役割を説明できる |
| 技術選定 | なぜ現在のモデル、フレームワーク、データベース、ツールを使ったのか説明できる |
| 既知の制限 | システムが苦手なこと、まだカバーしていない場面を自分から説明できる |
| 次の計画 | 「継続改善」のような曖昧な表現ではなく、具体的な改善方向がある |

README の目的は長く書くことではなく、他の人があなたに聞かなくてもプロジェクトを理解できるようにすることです。特に、実行方法と入力と出力の例は、再現できるレベルまで具体的に書く必要があります。

## 再現性チェック

| チェック項目 | 合格基準 |
| --- | --- |
| 依存関係の記録 | Python プロジェクトには requirements または pyproject、フロントエンドプロジェクトには package.json がある |
| 設定の説明 | API Key、モデル名、パス、ポートなどの設定に説明がある |
| データの説明 | サンプルデータの出典、形式、フィールドの意味が明確 |
| 最小実行コマンド | 最小限の流れを一つのコマンドで実行できる |
| エラー処理 | 依存関係不足、key 不足、ファイル不足などのよくあるエラーに案内がある |
| 環境の分離 | 個人PCの隠しパスや一時ファイルに依存しない |

プロジェクトがあなたのPCでしか動かないなら、まだポートフォリオプロジェクトとは言えません。少なくとも README を見れば、他の人が最小版を再現できる必要があります。

## AI 能力チェック

| プロジェクト種別 | 何を確認するか |
| --- | --- |
| 機械学習プロジェクト | baseline、学習/テスト分割、指標、誤りサンプル分析があるか |
| 深層学習プロジェクト | 学習ログ、検証指標、学習曲線、過学習の分析があるか |
| Prompt プロジェクト | Prompt の版管理、入出力例、失敗例、改善プロセスを記録しているか |
| RAG プロジェクト | 文書ソース、分割戦略、検索結果、引用確認、評価問題集があるか |
| Agent プロジェクト | ツール定義、実行トレース、停止条件、権限の境界、失敗時の復旧があるか |
| マルチモーダルプロジェクト | 入力形式、生成/理解の流れ、人手レビュー、品質基準を説明しているか |

プロジェクトの種類によって、確認すべき重点は異なります。すべてを同じ基準で見ないでください。RAG プロジェクトの核心は見た目の良さではなく、検索と引用の信頼性です。Agent プロジェクトの核心は手順の多さではなく、実行の追跡可能性と権限の制御です。

## エンジニアリングチェック

| チェック項目 | 合格基準 |
| --- | --- |
| ログ | 重要なリクエスト、モデル出力、ツール呼び出し、エラー情報が見える |
| パラメータ設定 | モデル、パス、しきい値、検索件数などの主要パラメータを設定できる |
| モジュール分割 | データ処理、モデル呼び出し、業務ロジック、評価コードが全部混ざっていない |
| テスト用サンプル | 少なくとも固定サンプルがあり、回帰確認に使える |
| コスト意識 | token、遅延、呼び出し回数、リソース消費の基本記録がある |
| 安全境界 | リスクの高いツール呼び出し、ファイル書き込み、外部リクエストなどが制御されている |

エンジニアリングとは、プロジェクトを複雑にすることではありません。より安定させ、原因を追いやすくし、継続改善しやすくすることです。構造がわかりやすい小さなプロジェクトのほうが、再現しにくい大きなプロジェクトよりもポートフォリオに向いています。

## 評価と振り返りチェック

| チェック項目 | 合格基準 |
| --- | --- |
| テストセット | 固定の問題、サンプル、またはタスクリストがある |
| 指標 | 効果を測る方法が少なくとも1つある |
| 成功サンプル | どんな場面でうまくいくかを示せる |
| 失敗サンプル | どんな場面で失敗するかを示し、その理由を説明できる |
| 境界サンプル | あいまいな入力、長い入力、情報不足、異常入力での挙動を示せる |
| 改善計画 | 失敗原因に基づいて、次の具体的な改善案を出せる |

失敗サンプルを書くことを怖がらないでください。ポートフォリオでは、成功スクリーンショットよりも失敗サンプルと振り返りのほうが価値が高いことがよくあります。なぜなら、それらはあなたが本当にシステムの境界を理解していることを示せるからです。

## 3分デモチェック

プロジェクトを発表するときは、次の順番で話すとわかりやすいです。まず、このプロジェクトが何の問題を解決するのか、次にユーザーがどう使うのか、システムが入力をどう処理するのか、AI がどこで役立っているのか、どうやって効果を確認するのか、どんな失敗があったのか、そして次にどう改善するのか、です。

もし3分以内にこれらをわかりやすく説明できないなら、プロジェクトの伝え方を整理する必要があります。まず README に戻って、背景、アーキテクチャ、例、評価を一本のわかりやすいストーリーにまとめてください。

## 最終合格基準

| レベル | 基準 |
| --- | --- |
| 練習レベル | ローカルで動かせる。基本コードと簡単な出力がある |
| プロジェクトレベル | README、実行コマンド、入力と出力の例、基本的なエラー処理がある |
| ポートフォリオレベル | 評価、ログ、失敗サンプル、スクリーンショットやデモ、明確な技術的判断がある |
| 面接レベル | アーキテクチャ、指標、制約、コスト、安全性、代替案を説明できる |

コースの各ステージプロジェクトは、少なくとも「プロジェクトレベル」まで到達するべきです。卒業プロジェクトは「ポートフォリオレベル」まで仕上げることをおすすめします。面接で使う予定があるなら、できるだけ「面接レベル」まで目指してください。
