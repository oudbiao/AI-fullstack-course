# E.B.4 メタプログラミング

:::tip この節の位置づけ
メタプログラミングは、つい「派手な技術」として語られやすいテーマです。  
でも、エンジニアリングで本当に価値があるのは、たいていとても具体的な場面です。

- 重複する定型コードを減らしたい
- 登録ロジックを自動化したい
- 設定に応じてコードの構造を生成したい

この節では、メタプログラミングを魔法のように説明するのではなく、次を中心に学びます。

> **いつ本当に使う価値があるのか、いつコードをかえって分かりにくくしてしまうのか。**
:::

## 一、メタプログラミングとは何か？

### 一言でいうと

ざっくり言うと、次のように理解できます。

> **コードを使って、コードそのものの構造を生成・変更・整理すること。**

例えば:

- クラスを動的に作る
- サブクラスを自動登録する
- フィールド宣言に基づいて振る舞いを生成する

### なぜ実務で必要になるのか？

同じような定型処理が多すぎることがあるからです。  
毎回手で書くと、コードは次のようになりがちです。

- 長くなる
- 保守しにくい
- 抜け漏れが起きやすい

### たとえ話

普通のプログラミングは、家具を手作業で作るようなものです。  
メタプログラミングは、先に型を作って、似た家具をまとめて作るようなものです。

---

## 二、動的なクラス作成：メタプログラミングの最も直接的な入口

```python
def build_model_class(name, fields):
    return type(name, (), fields)


User = build_model_class("User", {"role": "student", "active": True})
user = User()

print(user.role, user.active)
print(User.__name__)
```

### このコードは何を示しているのか？

これが示しているのは、次の2点です。

- クラス自体もオブジェクトである
- クラスは実行時に作成できる

### なぜ日常的な書き方ではないのか？

`class User:` と素直に書くほうが、たいてい分かりやすいからです。  
動的なクラス作成が役立つのは、次のような場合です。

- 構造が非常に重複している
- フィールドが設定から来る
- フレームワーク層で自動生成が必要

こういう場面では、価値が出やすくなります。

---

## 三、レジストリパターン：実務でよく使われるメタプログラミング

レジストリは、とても実用的なパターンです。  
システムが次のような情報を自動で覚えられるようにします。

- どんな実装があるか
- ある名前がどのクラスに対応するか

```python
REGISTRY = {}


def register(name):
    def decorator(cls):
        REGISTRY[name] = cls
        return cls
    return decorator


@register("csv")
class CsvLoader:
    def load(self):
        return "load csv"


@register("json")
class JsonLoader:
    def load(self):
        return "load json"


loader = REGISTRY["json"]()
print(loader.load())
print(REGISTRY)
```

### なぜレジストリはこれほどよく使われるのか？

次のような用途にとても向いているからです。

- プラグインシステム
- データローダー
- モデルのバックエンド
- ツール登録

### なぜこれを「実用的なメタプログラミング」と呼ぶのか？

見た目の派手さのために動的にしているのではなく、  
次のような手作業を明確に減らしているからです。

- 対応表を手で管理すること

## 四、ディスクリプタ：フィールドの振る舞いをまとめる

ディスクリプタは、最初は少し抽象的に見えます。  
でも、実務での感覚はまず次のように覚えるとよいです。

- 属性アクセスにルールを追加する仕組み

以下は、最小限のバリデータの例です。

```python
class PositiveNumber:
    def __set_name__(self, owner, name):
        self.private_name = "_" + name

    def __get__(self, instance, owner):
        if instance is None:
            return self
        return getattr(instance, self.private_name, None)

    def __set__(self, instance, value):
        if value <= 0:
            raise ValueError("value must be positive")
        setattr(instance, self.private_name, value)


class Config:
    timeout = PositiveNumber()


cfg = Config()
cfg.timeout = 3
print(cfg.timeout)
```

### この例でまず学ぶべきことは？

細かいプロトコルそのものより、次の理解が大事です。

- 属性にも独自の読み書きロジックを付けられる

これは設定システム、ORM、バリデーションフレームワークでとてもよく使われます。

## 五、メタプログラミングはいつ本当に価値があるのか？

### 同じパターンが非常に多いとき

例えば:

- 似たような loader が大量にある
- プラグインが大量にある
- フィールドの検証ロジックが大量にある

### フレームワーク層のコード

次のようなものを書くときは、メタプログラミングの価値が大きいです。

- プラグインフレームワーク
- 設定フレームワーク
- 登録と発見の仕組み

### 向いていない場合

普通の業務モジュールであれば、  
無理にメタプログラミングを使うと、かえって次のようになりやすいです。

- 読みにくくなる
- デバッグしにくくなる

---

## 六、よくある誤解

### 誤解1：動的なら必ず高度である

動的であること自体は手段であって、価値そのものではありません。

### 誤解2：重複は全部メタプログラミングで消すべき

場合によっては、あえて明示的に数回書いたほうが分かりやすいこともあります。

### 誤解3：メタプログラミングはフレームワーク作者だけのもの

フレームワークでは特によく使われますが、  
実務でもレジストリのようなパターンはとてもよく使われます。

---

## まとめ

この節で一番大事なのは、メタプログラミングを黒魔法として覚えることではなく、  
次の判断軸を持つことです。

> **メタプログラミングが最も価値を出すのは、重複が多く構造化された場面で、定型コードを減らし振る舞いを統一できるときです。複雑さをはっきり下げられないなら、使う価値は低いです。**

この点をつかんでおけば、あとでフレームワークのソースコードを読むときも、なぜそのように設計されているのかが理解しやすくなります。
