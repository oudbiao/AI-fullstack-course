# ナイーブベイズ

![ナイーブベイズの証拠累積図](/img/course/elective-naive-bayes-evidence-ja.png)

:::tip この節の位置づけ
ナイーブベイズは、次のように誤解されやすいです。

- 古すぎる
- 仮定が強すぎる

でも、いくつかの場面では今でもとても役立ちます。特に次のようなケースです。

- テキスト分類
- ルール性が強い少量データのタスク
- できるだけ早く、低コストでベースラインを作りたいとき

この節では、「ナイーブ」という言葉の裏にある実務上の価値をしっかり見ていきます。
:::

## 学習目標

- ナイーブベイズの中心となる確率的な直感を理解する
- テキスト分類で特によく使われる理由を理解する
- 「条件付き独立の仮定」が弱点でもあり強みでもある理由を理解する
- 実行可能な例を通して、最小限のテキスト分類フローを身につける

---

## 一、ナイーブベイズは何をしているのか？

### 1.1 「どのカテゴリがこのサンプルを生成しやすいか」を比較している

たとえば、次のようなテキストがあるとします。

- 「返金はいつ反映されますか」

モデルは次のように比較します。

- これは返金カテゴリのテキストとして生成された可能性が高いか？
- それともログイン問題カテゴリのテキストとして生成された可能性が高いか？

### 1.2 なぜ「ナイーブ」と呼ぶのか？

それは、とても強いけれど便利な仮定を置くからです。

- カテゴリが与えられたとき、各特徴量は互いに独立である

現実には、この仮定は完全には成り立たないことが多いです。  
それでも、次のような利点があります。

- 学習が速い
- 必要なデータが少ない
- 実装が簡単

### 1.3 たとえ話

ナイーブベイズは、次のように考えるとイメージしやすいです。

- まず、それぞれの単語が「返金カテゴリ」にとって有利かどうかを見る
- そのあと、こうした局所的な証拠を足し合わせる

複雑なモデルのように深い相互作用は学びませんが、  
多くのタスクではこれで十分なことがあります。

---

## 二、なぜテキスト分類に特に向いているのか？

### 2.1 単語頻度それ自体が強いシグナルだから

たとえば、次のような単語が出てきたら。

- 返金
- 請求書
- 証明書
- パスワード

多くの場合、それだけでカテゴリを強く示唆できます。

### 2.2 Multinomial Naive Bayes は bag-of-words と相性がいい

テキストでは、よくある入力は次のようなものです。

- 各単語が何回出現したか

これは `MultinomialNB` ととても相性がいいです。

### 2.3 安価なベースラインとしてとても優秀なことが多い

特に次のようなときに向いています。

- コーパスが大きくない
- ラベルが少ない
- まずはタスクの実現可能性を素早く確かめたい

こういう場面では、最初に試す価値があります。

---

## 三、まずは小さなテキスト分類の例を動かしてみよう

この例では、bag-of-words + ナイーブベイズで、最小限のカスタマーサポート意図分類を行います。

```python
from sklearn.feature_extraction.text import CountVectorizer
from sklearn.naive_bayes import MultinomialNB
from sklearn.pipeline import make_pipeline

texts = [
    "返金はいつ反映されますか",
    "返金を申請するにはどうすればいいですか",
    "請求書はいつ発行できますか",
    "電子請求書はどこに届きますか",
    "パスワードを忘れたらどうすればいいですか",
    "パスワードの再設定画面はどこですか",
]

labels = [
    "refund",
    "refund",
    "invoice",
    "invoice",
    "password",
    "password",
]

clf = make_pipeline(
    CountVectorizer(),
    MultinomialNB(alpha=1.0),
)

clf.fit(texts, labels)
pred = clf.predict(["返金はどう処理されますか", "電子請求書はいつ発行されますか"])
print(pred.tolist())
```

### 3.1 この例で最初に見るべき 2 層は？

1. `CountVectorizer()`  
   まずテキストを単語頻度の特徴量に変換する
2. `MultinomialNB()`  
   そのあと、各カテゴリでの単語分布をもとに判定する

### 3.2 なぜこの流れはテキストタスクで定番なのか？

とても軽いからです。

- 学習が速い
- 解釈しやすい
- デバッグしやすい

しかも、小さなタスクでは十分よい結果が出ることも少なくありません。

---

## 四、なぜ平滑化が重要なのか？

### 4.1 学習中に一度も見ていない単語があったらどうなる？

あるカテゴリのテキストに、ある単語が一度も出てこなかったとします。  
そのまま確率を掛け算すると、カテゴリ全体の確率が 0 に近づいてしまうことがあります。

### 4.2 だから smoothing を行う

たとえば `alpha=1.0` はよく使われる平滑化の設定です。  
これはざっくり言うと、次のような意味です。

- 「一度見なかった」だけで、その単語の確率を完全にゼロ扱いしない

### 4.3 実務上の直感

平滑化は単なる数学の細部ではありません。  
新しい単語が出てきたときに、モデルがどれくらい壊れやすいかに直接影響します。

---

## 五、ナイーブベイズの長所と限界

### 5.1 長所

- 学習が非常に速い
- 少量データでも始めやすい
- テキスト分類で十分使えることが多い
- 比較的説明しやすい

### 5.2 限界

- 特徴量の独立仮定が強すぎる
- 複雑な相互作用を表現しにくい
- より複雑な意味表現では、通常より強いモデルに劣る

### 5.3 とても実用的な判断基準

もし目標が次のようなものなら。

- 軽量で、安価で、説明しやすいテキスト分類のベースラインをまず作りたい

ナイーブベイズは、最初に試す価値が十分あります。

---

## 六、よくある誤解

### 6.1 誤解1：仮定が現実的でないなら、学ぶ必要はない

多くのアルゴリズムの仮定は、現実と完全には一致しません。  
大事なのは、それが実務上価値があるかどうかです。

### 6.2 誤解2：教材の例にしか使えない

中小規模のテキスト分類タスクでは、今でも有用なベースラインとしてよく使われます。

### 6.3 誤解3：Naive Bayes を使えば特徴量を見なくてよい

特徴量の表現は今でもとても重要です。  
bag-of-words の質が悪ければ、モデルの結果も悪くなります。

---

## まとめ

この節で最も大切なのは、次の判断を持つことです。

> **ナイーブベイズは仮定を単純化しているけれど、テキスト分類や少量データのベースラインでは、今でも学習が速く、コストが低く、説明しやすい実用的なモデルである。**

この判断がはっきりしていれば、小さなタスクを解くときに、いつこれを道具箱から取り出すべきかが分かるようになります。
