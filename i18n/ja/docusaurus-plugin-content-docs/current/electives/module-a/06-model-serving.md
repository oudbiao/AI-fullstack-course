# モデルのサービス化

![モデルのサービス化アーキテクチャ図](/img/course/elective-model-serving-architecture.png)

![モデルサービスの指標とバージョンルーティング図](/img/course/elective-serving-metrics-version-routing-map.png)

:::tip 読み方のヒント
モデルを本番に出したあとで一番大事に見るべきなのは、単発の推論時間ではなく、キューの待ち時間、batch 効率、P95/P99 レイテンシー、エラー率、バージョンルーティング、そしてロールバックできるかどうかです。図は「長く動き続けるシステム」として読むようにしてください。
:::

:::tip この節の位置づけ
「モデルを動かすこと」と「モデルをサービス化すること」は別物です。

前者は、たとえば：

- スクリプトの中で1回モデルを呼び出す

後者は、たとえば：

- リクエストを受ける
- 並べる
- 振り分ける
- 結果を返す
- 監視と更新を行う

この節で解決したいのは、「モデルを呼べる」状態から「モデルをサービスとして提供できる」状態へ進むことです。
:::

## 学習目標

- モデルサービス化の中心となる構成要素を理解する
- リクエストキュー、バッチ処理、バージョンルーティングがサービス内で果たす役割を理解する
- 動くサンプルを通して、最小限の serving の流れを作る
- モデルを本番公開する前に何の指標を見るべきか意識できるようにする

---

## 一、なぜモデルサービス化は「API を包めば終わり」ではないのか？

### 1.1 実際のリクエストは、1件ずつ均等に来るわけではないから

サービスでは次のようなことが起こります。

- 突発的な高負荷
- 大きさの違うリクエスト
- 遅いリクエストと速いリクエストの混在

そのため、必要になるのは：

- キュー
- スケジューリング
- タイムアウト
- 指標

### 1.2 モデルは更新されるから

本番公開後も、次のようなことが発生します。

- 新バージョンの段階的リリース
- モデルのロールバック
- 複数バージョンの共存

つまり、サービス化は「今のモデルを外に見せる」だけではなく、  
今後どう保守するかまで含みます。

### 1.3 たとえ話

単発の推論は、あなたが自分でキッチンに立って1食作るようなものです。  
モデルのサービス化は、レストランを開くようなものです。

- お客さんは並ぶ
- 厨房は調理を調整する
- 料理は安定して提供する必要がある

---

## 二、サービス化の最も重要なコンポーネント

### 2.1 リクエスト入口

役割：

- リクエストを受け取る
- パラメータを検証する
- 認証を行う

### 2.2 キュー

役割：

- リクエストをバッファする
- トラフィックを平滑化する

### 2.3 バッチ処理器

複数の小さなリクエストをまとめて、スループットを高めます。

### 2.4 モデル実行器

実際に次を行います。

- 前処理
- 推論
- 後処理

### 2.5 バージョンルーティングとヘルスチェック

役割：

- どのリクエストをどのモデルバージョンに送るか決める
- どのインスタンスがトラフィックを受けられるか確認する

## 三、まずは最小限の serving の流れを動かしてみる

この例では、次の流れを再現します。

1. リクエストがキューに入る
2. バッチ処理器が batch ごとに取り出す
3. モデル実行器がまとめて処理する
4. モデルバージョンごとに結果を返す

```python
from collections import deque


request_queue = deque(
    [
        {"id": "req1", "text": "返金規則", "model_version": "v1"},
        {"id": "req2", "text": "請求書の規則", "model_version": "v1"},
        {"id": "req3", "text": "住所変更", "model_version": "v2"},
        {"id": "req4", "text": "証明書の説明", "model_version": "v2"},
    ]
)


def batch_pop(queue, batch_size):
    batch = []
    while queue and len(batch) < batch_size:
        batch.append(queue.popleft())
    return batch


def model_executor(batch):
    outputs = []
    for item in batch:
        outputs.append(
            {
                "id": item["id"],
                "model_version": item["model_version"],
                "answer": f"[{item['model_version']}] processed:{item['text']}",
            }
        )
    return outputs


all_outputs = []
while request_queue:
    batch = batch_pop(request_queue, batch_size=2)
    print("batch:", batch)
    outputs = model_executor(batch)
    all_outputs.extend(outputs)

print("\noutputs:")
for item in all_outputs:
    print(item)
```

### 3.1 このコードから何を学ぶべきか？

モデルサービス化の基本的な動きが分かります。

- リクエストは、来たらすぐ1件ずつ実行するわけではない
- いったんキューに入り、戦略に従って振り分けられる

### 3.2 なぜバッチ処理はサービスで重要なのか？

多くのモデルは、まとめて処理したほうがスループットが高くなります。  
もし1件ずつ別々に実行すると：

- リソースの利用効率が悪くなることがある

### 3.3 なぜここでは `model_version` を明示的に残すのか？

本番サービスでは、複数バージョンの共存はよくあります。  
バージョン情報がないと、段階的リリースやロールバックがとても大変になります。

## 四、本番公開後に見るべき指標は何か？

### 4.1 レイテンシー

少なくとも次を見ます。

- 平均レイテンシー
- P95 レイテンシー

### 4.2 スループット

たとえば：

- 1秒あたりのリクエスト数
- 1秒あたりの batch 数

### 4.3 エラー率

含まれるもの：

- リクエスト失敗
- タイムアウト
- モデル内部の例外

### 4.4 バッチ処理効率

たとえば：

- 平均 batch size

これが長期間ずっと小さいなら、  
バッチ処理戦略が本当に機能していない可能性があります。

---

## 五、モデルサービス化で起こりやすい落とし穴

### 5.1 誤解1：モデルの推論時間だけ見ればよい

実際のレイテンシーには、通常次も含まれます。

- キュー待ち
- 前処理
- 後処理
- ネットワークのオーバーヘッド

### 5.2 誤解2：batch は大きいほどよい

batch を大きくするとスループットは上がることがあります。  
しかし、1件あたりのレイテンシーが悪化することもあります。

### 5.3 誤解3：バージョンルーティングなしで、そのまま本番モデルを置き換える

こうすると、問題が起きたときのロールバックが非常につらくなります。

---

## まとめ

この節で最も大事なのは、サービスの視点を持つことです。

> **モデルのサービス化とは「API を作ってモデルを呼ぶこと」ではなく、キュー、バッチ処理、バージョン、ヘルスチェック、指標を組み合わせて、モデルを継続運用できるサービスにすることです。**

この理解ができていれば、  
次に学ぶエッジデプロイや総合プロジェクトでも、全体のつながりをつかみやすくなります。
