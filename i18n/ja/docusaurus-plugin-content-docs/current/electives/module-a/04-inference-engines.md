# E.A.4 推論エンジン

![推論エンジンとハードウェア適合図](/img/course/elective-inference-engine-hardware-ja.png)

![推論エンジン選定マトリクス図](/img/course/elective-inference-engine-selection-matrix-ja.png)

:::tip 図の読み方
推論エンジンは、速ければ速いほどよいわけではありません。モデル形式、対象ハードウェア、レイテンシ/スループット、デプロイ環境、チームの保守力に合っていることが大切です。図を見るときは、ONNX Runtime、TensorRT、OpenVINO を、それぞれ異なる制約下で使うツール箱だと考えてください。
:::

:::tip この節の位置づけ
学習済みモデルは、そのままでは高性能なオンラインサービスにはなりません。  
その間には、たいてい非常に重要なシステム部品がもう一段あります。

- 推論エンジン

これは、モデルのグラフを本当に効率よく、特定のハードウェア上で動かす役割を持ちます。

この授業で答えたいのは、次の問いです。

> **なぜデプロイでは、よく「モデルの重みをそのまま読み込んで推論する」のではなく、先に推論エンジンを通す必要があるのか。**
:::

## 学習目標

- 推論エンジンがデプロイの流れの中でどんな役割を持つか理解する
- いろいろな推論エンジンが、どんなハードウェアや場面に向いているかを大まかに区別する
- 実行できるサンプルを通して、「レイテンシ、スループット、適応性」という3つの指標を理解する
- エンジンを選ぶときの、最初の判断基準を身につける

---

## 一、推論エンジンは何をしているのか？

### モデルそのものではない

モデルが答えるのは：

- ネットワーク構造とパラメータは何か

推論エンジンが答えるのは：

- この構造を、対象デバイス上でどうすればより効率よく実行できるか

### ふつう、何をしてくれるのか？

代表的なものは次の通りです。

- グラフ最適化
- 演算子の融合
- メモリ計画
- バックエンドの kernel 選択

### たとえ話

モデルはレシピのようなものです。  
推論エンジンは厨房のオーケストレーションシステムのようなものです。

同じレシピでも、  
違う厨房で違う手順で作れば、速度も品質も変わります。

---

## 二、なぜ推論エンジンにはたくさん種類があるのか？

### ハードウェアが違うから

よくある対象環境には、次のようなものがあります。

- 汎用 CPU
- NVIDIA GPU
- Intel CPU / NPU
- エッジデバイス

### 最適化の目的が違うから

重視されるものは、場合によって違います。

あるものは次を重視します。

- 使いやすさ

別のものは次を重視します。

- 極限の性能

### だから「絶対最強のエンジン」はない

より適切な聞き方は、次のようになります。

- このモデル、このハードウェア、この目的なら、どのエンジンが合っているか？

---

## 三、まず小さな例で「エンジン選択」を理解しよう

この例は、実際に ONNX Runtime や TensorRT を動かすものではありません。  
ただし、次のことをとてもわかりやすく模擬します。

- さまざまな場面での、異なるエンジンのレイテンシ、スループット、適合スコア

```python
engines = [
    {"name": "onnxruntime", "latency_ms": 32, "throughput_qps": 31, "hardware_fit": 8},
    {"name": "tensorrt", "latency_ms": 14, "throughput_qps": 70, "hardware_fit": 10},
    {"name": "openvino", "latency_ms": 26, "throughput_qps": 38, "hardware_fit": 9},
]


def score(engine, prefer_low_latency=True):
    latency_score = 100 / engine["latency_ms"]
    throughput_score = engine["throughput_qps"] / 10
    hardware_score = engine["hardware_fit"]

    if prefer_low_latency:
        return round(latency_score * 0.5 + throughput_score * 0.2 + hardware_score * 0.3, 2)
    return round(latency_score * 0.2 + throughput_score * 0.5 + hardware_score * 0.3, 2)


latency_first = sorted(
    [{**e, "score": score(e, True)} for e in engines],
    key=lambda x: x["score"],
    reverse=True,
)

throughput_first = sorted(
    [{**e, "score": score(e, False)} for e in engines],
    key=lambda x: x["score"],
    reverse=True,
)

print("latency_first:")
for item in latency_first:
    print(item)

print("\nthroughput_first:")
for item in throughput_first:
    print(item)
```

### このコードは何を教えているのか？

このコードが伝えたいのは、次のことです。

- エンジン選びは、単一の指標だけで順位をつけるものではない

たとえば、次をより重視するなら：

- 低レイテンシ

あるいは、次をより重視するなら：

- 高スループット

最終的な順位は変わるかもしれません。

### なぜ「TensorRT のほうが速い」とだけ覚えるより役に立つのか？

実際の判断では、見るべきことは速度だけではないからです。

- 既存の流れに組み込めるか
- 対象モデルに対応しているか
- その性能差のために、複雑さを増やす価値があるか

---

## 四、よくある推論エンジンの大まかな違い

### ONNX Runtime

より汎用的な選手、と考えるとわかりやすいです。  
よくある長所は次の通りです。

- エコシステムが広い
- 互換性が高い
- 使い始めやすく、バランスがよい

### TensorRT

NVIDIA エコシステムで高性能を狙うルートに近いです。  
よくある特徴は次の通りです。

- GPU 圧倒的に強い
- 調整の余地が大きい
- エンジニアリングの難易度はやや高い

### OpenVINO

Intel エコシステムや特定ハードウェアへの適合に強いです。  
よくある特徴は次の通りです。

- 一部の CPU / Intel デバイスでよい性能を出しやすい
- 特定のデプロイ環境に向いている

### どう選ぶのか？

「どれが一番人気か」ではなく、まず次を考えてください。

- 自分のハードウェアは何か
- 自分のモデル形式は何か
- レイテンシを重視するのか、それとも保守しやすさを重視するのか

---

## 五、推論エンジンはデプロイ結果に何を直接影響するのか？

### レイテンシ

ユーザーが最初に感じるのは、次のことです。

- 速いかどうか

### スループット

サービス側がより気にするのは、次のことです。

- 同時にどれだけのリクエストをさばけるか

### リソース利用率

たとえば、次のような点です。

- VRAM をより節約できるか
- CPU の利用率がより適切か

### 保守の複雑さ

性能が高いルートは、  
ときに次の意味でもあります。

- エクスポートが複雑になる
- デバッグが難しくなる
- プラットフォームへの依存が強くなる

---

## 六、よくある誤解

### 誤解1：推論エンジンは「ライブラリを変えて動かすだけ」

違います。  
多くの場合、次のものが変わります。

- グラフの実行方法
- 最適化戦略
- ハードウェアの使い方

### 誤解2：最速のエンジンが最良のエンジン

互換性が低く、調整が複雑で、デプロイのハードルが高いなら、  
「最速」だからといって最適とは限りません。

### 誤解3：先にエンジンを決めてからハードウェアを見る

より自然な順番は、たいてい逆です。

- 先にハードウェアと目標の制約を見る
- それからエンジンを選ぶ

---

## まとめ

この節で一番大事なのは、いくつかのエンジン名を覚えることではなく、デプロイの見方を身につけることです。

> **推論エンジンは、モデルとハードウェアの間で効率よく実行できるようにするシステム層です。その価値は、単に「動かす」ことではなく、「より速く、より省資源で、より適した形で動かす」ことにあります。**

この理解ができると、あとでサービス化やエッジデプロイを学ぶときに、ぐっと自然に感じられるはずです。
