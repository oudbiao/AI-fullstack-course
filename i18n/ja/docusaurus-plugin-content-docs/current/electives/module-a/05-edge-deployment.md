# E.A.5 エッジデバイスへのデプロイ

:::tip この節の位置づけ
エッジデバイスへのデプロイで、最も見落とされやすいのは次の点です。

- これは「クラウドのサービスを小さい機械に移す」ことではない

エッジ環境でよくある制約には、次のようなものがあります。

- メモリが少ない
- 消費電力に厳しい
- ネットワークが不安定
- アップデートやトラブル対応が難しい

つまり、本当に身につけるべきなのは次のことです。

> **制約のあるデバイス上で「十分に使える」システムを作ることであって、デスクトップ機のような完璧な形を目指すことではありません。**
:::

![エッジデプロイの制約と判断フロー](/img/course/elective-edge-deployment-constraint-map-ja.png)

## 学習目標

- エッジデプロイとクラウドデプロイの根本的な違いを理解する
- メモリ、消費電力、レイテンシ、オフライン対応の4つの観点から方案を評価できるようになる
- 動かせるサンプルを通して、デバイス選定とモデル適合の考え方を理解する
- エッジデプロイで優先順位をどう判断するかを身につける

---

## 一、エッジデプロイは何がそんなに難しいのか？

### デバイスのリソースは通常、サーバーよりずっと少ない

よくある制約は次のとおりです。

- 利用できるメモリが少ない
- CPU / GPU の計算性能が限られる
- 電力予算が限られる

つまり、クラウドでは「標準でOK」な方案でも、エッジではそのままでは現実的でないことが多いのです。

### ネットワークが必ずしも安定していない

エッジデバイスは、次のような場所に置かれることがよくあります。

- 工場
- 店舗
- カメラノード
- 車載やモバイルの場面

ネットワークが不安定でも、システムは最低限の機能を保たなければなりません。  
だからこそ、エッジデプロイでは次の点がとても重要です。

- ローカル推論
- キャッシュ
- オフライン fallback

### 運用コストが高くなりやすい

サーバーなら、落ちても遠隔再起動、段階的リリース、スケール拡張ができます。  
でもエッジデバイスは、問題が起きたときの対応コストが高くなりがちです。

そのため、エッジシステムでは次の点がより重視されます。

- 安定性
- 予測しやすさ
- 手間が少ないこと

---

## 二、エッジデバイスを選ぶとき、まず何を見る？

### メモリ予算

これは最初の関門です。  
モデル、ランタイム、入力キャッシュ、サービス本体のどれもメモリを使います。

### 消費電力予算

エッジデバイスは「動く」だけでは不十分で、次の点も見る必要があります。

- 長時間安定して動作できるか
- 放熱に耐えられるか

### 目標レイテンシ

タスクによってレイテンシ要件は大きく異なります。

- 入退室認証：よりリアルタイム性が必要かもしれない
- バッチ集計：どれくらい遅くてもよい

### オフラインが必須か

もし場面として、

- ネットが切れても動作し続ける必要がある

なら、モデルや依存関係の設計はかなり変わります。

---

## 三、まずはデバイスとモデルの適合例を動かしてみよう

次の例では、以下をシミュレートします。

1. デバイス側のリソース制約
2. モデル側のデプロイ要件
3. 動作可能な組み合わせの自動抽出

```python
devices = [
    {"name": "jetson_nano", "memory_gb": 4, "power_w": 10, "offline_required": True},
    {"name": "raspberry_pi_5", "memory_gb": 8, "power_w": 8, "offline_required": True},
    {"name": "industrial_box", "memory_gb": 16, "power_w": 35, "offline_required": True},
]

models = [
    {"name": "tiny_classifier", "memory_need_gb": 1.2, "power_need_w": 4, "latency_ms": 25},
    {"name": "medium_detector", "memory_need_gb": 6.0, "power_need_w": 12, "latency_ms": 90},
    {"name": "large_vlm", "memory_need_gb": 18.0, "power_need_w": 40, "latency_ms": 250},
]


def can_deploy(device, model, latency_target_ms=120):
    return (
        device["memory_gb"] >= model["memory_need_gb"]
        and device["power_w"] >= model["power_need_w"]
        and model["latency_ms"] <= latency_target_ms
    )


for device in devices:
    candidates = [model["name"] for model in models if can_deploy(device, model)]
    print(device["name"], "->", candidates)
```

### このコードで一番大事なポイントは？

数式そのものではなく、  
「どんな順番でふるい分けるか」という考え方を作れることです。

1. まず、リソース的に載るかを見る
2. 次に、消費電力に耐えられるかを見る
3. 最後に、レイテンシ基準を満たすかを見る

### なぜ「入る」ことと「デプロイに適している」ことは違うのか？

たとえば、モデルはなんとか読み込めたとしても、  
次のような問題があるかもしれません。

- レイテンシが高すぎる
- 消費電力が大きすぎる
- 長時間動かすと不安定になる

その場合、それはまだ良い方案とは言えません。

## 四、エッジデプロイでよく使う最適化の方向

### モデルを小さくする

よくある方法は次のとおりです。

- 量子化
- 小さいモデルへの蒸留
- より軽量なアーキテクチャ

### ランタイムを最適化する

たとえば、次のような工夫があります。

- 適切な推論エンジンを選ぶ
- 不要な前処理を減らす
- 入力解像度を制限する

### システム全体を最適化する

たとえば、次のような工夫があります。

- 結果をローカルにキャッシュする
- オフライン時は重要機能だけ残す
- バックグラウンドログやデバッグの負荷を下げる

---

## 五、エッジデプロイでよくある落とし穴

### 誤解1：先にモデルを選んでからデバイスを考える

より安定しやすい順番は通常、次のとおりです。

- まずデバイス制約を見る
- そのあとでモデルとエンジンを選ぶ

### 誤解2：単発の推論だけ測って、長時間運用を測らない

多くのシステムは、

- 1回は問題なく動く
- 30分連続で動かすと遅くなる、または熱くなる

ということがあります。

### 誤解3：エッジデバイスはいつもネットにつながると思う

実際の現場では、  
オフライン対応は「あると便利」ではなく、基本要件であることが多いです。

## 小まとめ

この節で一番大事なのは、エッジの視点を持つことです。

> **エッジデプロイは、まずリソースと安定性の問題であり、その次にモデル性能の問題です。**

メモリ、消費電力、レイテンシ、オフライン対応をまとめて見ることで、  
はじめて現実的な方案になります。
