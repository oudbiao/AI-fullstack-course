# デプロイ統合プロジェクト

:::tip この節の位置づけ
これまでのこの選択科目では、いくつかの要素を学びました。

- C++ の基礎
- リソース管理
- モデル最適化
- 推論エンジン
- エッジデプロイ
- モデルのサービス化

この節でやるのは、これらの要素を組み合わせて、きちんと説明できて、かつ見せられるプロジェクトにすることです。
:::

![デプロイ統合プロジェクトの納品クローズドループ図](/img/course/elective-deployment-project-delivery-loop-ja.png)

## 学習目標

- 「モデルデプロイの能力」を、見せられるプロジェクトとしてまとめられるようになる
- デプロイ系プロジェクトと学習系プロジェクトの、見せるポイントの違いを理解する
- 実行可能な例を通して、プロジェクトの骨組みを作る
- 性能、リソース使用量、安定性をどう見せるべきかを知る

---

## 一、デプロイ系プロジェクトで何を見せるべきか？

### 1.1 「モデルがどれだけ強いか」ではなく、「システムがどれだけ安定しているか」

デプロイプロジェクトで一番説得力があるのは、たいてい次のような点です。

- 1回のデモがどれだけすごいか

ではなく、

- レイテンシはどれくらいか
- スループットはどれくらいか
- メモリ使用量はどうか
- デプロイ環境はどうなっているか
- ロールバックや監視を考慮しているか

### 1.2 初心者に向いているプロジェクト題材

ポートフォリオにしやすい題材の例は、たとえば次のようなものです。

> **軽量画像分類サービス：ローカル推論、バッチ処理、エッジデバイス適応に対応。**

この題材がよい理由は、

- 入力が明確
- 出力が明確
- さまざまなデプロイ方式を比較できる
- 最適化とサービス化を見せやすい

### 1.3 なぜこの題材は「超大規模な汎用システム」よりよいのか？

デプロイ系プロジェクトで一番危険なのは、スコープを広げすぎることだからです。  
機能を広く薄くするより、わかりやすく、測れて、見せられることのほうが大切です。

---

## 二、デプロイ統合プロジェクトに最低限必要なモジュールは？

少なくとも、次のものを入れるのがおすすめです。

1. モデル準備  
   たとえば、エクスポート、量子化、形式変換
2. 推論実行  
   たとえば、推論エンジンを選ぶ
3. サービスインターフェース  
   たとえば、HTTP かローカルのバッチ処理インターフェース
4. 指標の集計  
   レイテンシ、スループット、メモリ
5. デプロイ説明  
   環境、ハードウェア、起動方法

時間に余裕があれば、さらに次も追加しましょう。

- エッジデバイス版
- カナリアリリースやバージョン切り替え
- より充実した監視

---

## 三、まずはプロジェクトの骨組み例を動かしてみよう

この例は実際にサービスを起動するものではありません。  
ただし、デプロイプロジェクトで最も大事な設計の形を、直接表しています。

```python
from dataclasses import dataclass, field


@dataclass
class DeploymentProject:
    name: str
    model_name: str
    target_device: str
    engine: str
    modules: list
    metrics: dict
    risks: list = field(default_factory=list)


project = DeploymentProject(
    name="lightweight-image-classifier-serving",
    model_name="tiny_classifier.onnx",
    target_device="raspberry_pi_5",
    engine="onnxruntime",
    modules=[
        "preprocess",
        "inference_engine",
        "postprocess",
        "batch_scheduler",
        "http_api",
        "metrics_exporter",
    ],
    metrics={
        "p95_latency_ms": 85,
        "throughput_qps": 18,
        "peak_memory_mb": 620,
    },
    risks=[
        "batch size too large increases latency",
        "edge device thermal throttling",
        "version rollback not automated yet",
    ],
)

print(project)
```

### 3.1 この例が役立つ理由

デプロイプロジェクトでは、よく次のようになりがちです。

- 「いろいろ最適化しました」

でも、次のことがはっきりしません。

- デプロイ対象は何か
- どのデバイスで動かすのか
- コアとなるモジュールは何か
- どんな指標で成果を示すのか

このような構造化された骨組みは、これらをきちんと整理させてくれます。

### 3.2 なぜデプロイプロジェクトには必ず指標が必要なのか？

指標がなければ、  
最適化に本当に価値があったのか、他の人には判断しにくいからです。

少なくとも、次の指標は明確にしましょう。

- レイテンシ
- スループット
- メモリまたは VRAM

## 四、デプロイプロジェクトのおすすめの見せ方

### 4.1 まず問題と目標を説明する

たとえば、

- 軽量な分類モデルをエッジデバイスにデプロイしたい
- オフライン推論が必要
- 目標レイテンシは 100ms 未満

### 4.2 次にシステム構成を説明する

たとえば、

- モデル形式
- 推論エンジン
- サービス構成
- 対象ハードウェア

### 4.3 最後に結果とトレードオフを説明する

たとえば、

- なぜ TensorRT ではなく ONNX Runtime を選んだのか
- なぜ batch をこれ以上大きくしなかったのか
- なぜ、より強い量子化をまだしていないのか

この部分に、その人のエンジニアとしての判断がよく出ます。

---

## 五、プロジェクトで見落としやすい3つのこと

### 5.1 環境の再現性

他の人がすぐに再現できないと、

- 依存関係のバージョン
- 起動コマンド
- 入力サンプル

こうした情報があっても、プロジェクトの説得力はかなり下がります。

### 5.2 ベースライン比較

できれば、次の点を示しましょう。

- 最適化前はどうだったか
- 最適化後に何が改善したか

### 5.3 失敗例

たとえば、

- あるデバイスでは消費電力が高すぎる
- 大きな batch にするとテールレイテンシが悪化する

こうしたことを書いておくと、実際に工程上の取捨選択をしたことが伝わります。

---

## 六、よくある誤解

### 6.1 誤解その1：デプロイプロジェクトは動けば十分

デプロイプロジェクトでより強く見るべきなのは、

- 性能
- 安定性
- 再現性

です。

### 6.2 誤解その2：画面だけ見せて、指標を見せない

この種のプロジェクトでは、UI より指標のほうが大切です。

### 6.3 誤解その3：最初からクラウド、エッジ、モバイルの全部をカバーしようとする

多くの場合、よりよい進め方は、

- まず 1 つの対象デバイスを徹底的に作り込む

ことです。

---

## まとめ

この節で最も大事なのは、デプロイプロジェクトの見方を身につけることです。

> **デプロイ統合プロジェクトの価値は、モデルそのものがどれだけ複雑かではなく、対象デバイス、エンジンの選択、サービス構成、性能指標、そしてエンジニアリング上の取捨選択を、1つのきれいな流れとして説明できるかにあります。**

この流れがはっきりしていれば、この種のプロジェクトはポートフォリオとしてとても適しています。
