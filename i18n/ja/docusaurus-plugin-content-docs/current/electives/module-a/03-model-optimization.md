# E.A.3 モデル最適化技術

![モデル最適化ロードマップ](/img/course/elective-model-optimization-map-ja.png)

![モデル最適化のトレードオフダッシュボード](/img/course/elective-optimization-tradeoff-dashboard-ja.png)

:::tip 図の読み方
最適化の前に、まずボトルネックがどこにあるかを見極めましょう。VRAM、レイテンシ、スループット、精度、ハードウェア適合性、保守コストは、たいてい互いに影響し合います。図を見るときは、単に「もっと速くできるか」ではなく、「どの指標を、どんな代償で改善するのか」を考えてください。

:::tip この節の位置づけ
モデル最適化は、いちばん名詞が並びやすい分野です。

- 量子化
- 剪定
- 蒸留
- 融合

でも、実際のデプロイでの問題は、もっと素朴です。

> **いったい何を節約したいのか。VRAM、レイテンシ、スループット、それともハードウェア適合コストですか？**

この問いを先に整理しておくと、最適化手法が「最適化のための最適化」になりません。
:::

## 学習目標

- 代表的なモデル最適化手法が、それぞれ何を解決するのかを理解する
- 最適化はたいてい精度を性能と引き換えにするのであって、ただ得をするわけではないと理解する
- 実行可能な例を通して、量子化とコストのトレードオフを理解する
- デプロイ視点で最適化の優先順位を判断できるようになる

---

## 一、モデル最適化は単一目標ではない

### よく最適化したいものは、実は4つ

- モデルサイズ
- 推論レイテンシ
- スループット
- デバイス適合性

### なぜ「速い」ことが、いつも「良い」とは限らないのか？

なぜなら、多くの最適化は trade-off だからです。

- 小さくなるが、精度が落ちる
- 速くなるが、デバッグが複雑になる
- メモリは節約できるが、後処理が難しくなる

### たとえ話

モデル最適化は、スーツケースの整理に似ています。  
何でも詰め込むのではなく、次の間で取捨選択します。

- 体積
- 重量
- 実用性

---

## 二、もっともよく使われる5つの最適化ルート

### 量子化

高精度の重みを、より低い精度に圧縮します。  
通常の目的は次の通りです。

- VRAM を減らす
- スループットを上げる
- エッジデバイスに適しやすくする

### 剪定

重要でない重み、チャネル、または層を取り除きます。  
通常の目的は次の通りです。

- 計算量を減らす

### 蒸留

より小さなモデルに、大きなモデルの出力を学習させます。  
通常の目的は次の通りです。

- できるだけ能力を保つ
- そのうえでデプロイコストを下げる

### 演算子融合

複数ステップの計算を実行グラフ内でまとめます。  
通常の目的は次の通りです。

- メモリの読み書きを減らす
- 実行効率を上げる

### バッチ処理とスケジューリング最適化

モデル自体を変えるのではなく、  
動かし方を変えます。  
通常の目的は次の通りです。

- スループットを上げる

---

## 三、まずは量子化誤差の例を見てみる

この例でやることは、とてもシンプルです。

- 浮動小数点の重みを、より粗い刻みに量子化する
- 量子化前後の誤差を計算する

```python
import numpy as np

weights = np.array([0.12, -1.87, 3.44, -0.03], dtype=np.float32)


def fake_quantize(values, scale):
    return np.round(values * scale) / scale


q8_like = fake_quantize(weights, scale=16)
q4_like = fake_quantize(weights, scale=4)

print("original :", weights)
print("q8_like  :", q8_like)
print("q4_like  :", q4_like)
print("q8 mae   :", np.mean(np.abs(weights - q8_like)))
print("q4 mae   :", np.mean(np.abs(weights - q4_like)))
```

### このコードから得られる、いちばん大事な気づきは何ですか？

量子化は「無料の圧縮」ではありません。  
誤差が発生します。

だから今後、次のような表記を見たら：

- `int8`
- `int4`

まず「もっと省ける」と考えるだけでなく、  
次も確認しましょう。

- 精度低下はどれくらいか

### なぜ、より低いビット数は難しくなりやすいのか？

表現できる範囲が粗くなるからです。  
圧縮を強くするほど、元の細かい情報は失われやすくなります。

---

## 四、蒸留が「デプロイしやすい方法」とされる理由

### なぜなら、単にモデルを縮めるのではなく、モデルを置き換えるから

蒸留の本質は、元のモデルを直接書き換えることではなく、  
より小さな生徒モデルを訓練することです。

### どんな場面に向いているのか？

次のような場面に向いています。

- リクエストパターンが比較的安定している
- タスクの境界がはっきりしている
- 訓練コストをかけてでも、デプロイの利点を得たい

### 量子化との違い

- 量子化：同じモデルのまま、精度を下げる
- 蒸留：より小さな新しいモデルに置き換える

---

## 五、最適化の順番はどう考えるべきか？

### 先に技を入れるのではなく、まず原因を調べる

先に次を確認します。

- どこが遅いのか
- どこが高いのか
- どこでメモリ不足になるのか

### とても実用的な順番

1. まず実行時レイヤーの最適化を行う  
   たとえば batching、キャッシュ、スケジューリング
2. 次にリスクの低いモデル最適化を行う  
   たとえば int8 量子化
3. 最後に、より踏み込んだ方法を検討する  
   たとえば大きな剪定、蒸留、構造変更

### なぜこの順番が安定なのか？

多くの問題は、そもそもモデル本体にないからです。  
実行時の調整が不十分なままモデルだけいじっても、  
効果が小さいことがよくあります。

---

## 六、よくある誤解

### 誤解1：最適化とは、できるだけ小さくすること

違います。  
大事なのは次です。

- 事業上受け入れられる精度の範囲で、できるだけ節約する

### 誤解2：量子化すれば必ず速くなる

必ずではありません。  
次の点も見なければいけません。

- ハードウェアが対応しているか
- エンジンが十分に最適化されているか

### 誤解3：すべてのモデルに同じ最適化手法が使える

モデル、ハードウェア、業務要件が違えば、  
最適なルートもよく変わります。

---

## まとめ

この節でいちばん大切なのは、最適化手法の名前を用語集のように覚えることではなく、  
デプロイでの判断軸を作ることです。

> **モデル最適化の本質は、サイズ、レイテンシ、スループット、ハードウェア適合性の間でバランスを取ることであり、単に「もっと強く圧縮する」ことではありません。**

この見方ができるようになれば、今後の具体的なデプロイでも、より適切な手段を選びやすくなります。

## 練習

1. 例の `scale` を大きくしたり小さくしたりして、誤差の変化を観察してください。
2. なぜ量子化は「元のモデルを圧縮する」ことに近く、蒸留は「より小さなモデルに置き換える」ことに近いのでしょうか？
3. あなたの課題が主にスループット不足で、VRAM 不足ではないとしたら、まずどの種類の最適化を見るべきでしょうか？
4. ある最適化が「本番投入する価値がある」と判断するには、どう確認しますか？
