# AIプロダクト設計の考え方

![AI 製品の意思決定 4象限図](/img/course/elective-ai-product-decision-matrix.png)

![AI 製品の実験と指標のループ図](/img/course/elective-ai-product-experiment-metrics-loop.png)

:::tip 図の見方
プロダクトの判断は、「このAI機能はすごい」で止まってはいけません。図を見るときは、ユーザーの問題、仮説、MVP、成功指標、リスクの境界、実験のフィードバック、反復をまとめて見ることで、その機能を本当に続ける価値があるかを判断できます。
:::

:::tip この節の位置づけ
AIプロダクトで最も起こりやすいミスの1つは、次の考え方です。

- 先に「モデルで何ができるか」を考える

そうではなく、まず考えるべきなのは、

- ユーザーは本当は何を解決したいのか

これを間違えると、技術的には派手でも、プロダクトとしては薄いプロジェクトになりがちです。

そのため、このレッスンで解決したいのは次のことです。

> **AI機能を「すごいアイデア」から「検証できるプロダクト価値」に変える方法を学ぶこと。**
:::

## 学習目標

- AIプロダクト設計と純粋な技術実装の違いを理解する
- ユーザーの問題、成功基準、コスト、リスクの4つの観点から案を判断できるようになる
- 実行できる例を通して、最小限のプロダクト優先順位づけの考え方を身につける
- 「先に判断し、あとで機能を作る」というプロダクト思考を身につける

---

## 一、AIプロダクト設計で最も大事な問題は何か？

### 1.1 「できるか」ではなく、「やる価値があるか」

多くのAI機能は、理論上は作れます。  
でも、プロダクトとしてより重要なのは次の点です。

- ユーザーが本当に困っているか
- コストが許容できるか
- リスクを管理できるか
- 使い続けやすいか

### 1.2 たとえ話

技術実装は、たくさんの料理が作れることに似ています。  
プロダクト設計は、次を決めることに似ています。

- 今日どの料理を作るか
- 誰に食べてもらうか
- コストを回収できるか

### 1.3 つまり、AIプロダクト設計は「技術を弱めること」ではない

むしろ、技術を次のものに従わせることです。

- ユーザーの問題
- ビジネス上の制約
- リスクの境界

---

## 二、AIプロダクトを設計するときによく見る4つの観点

### 2.1 ユーザー価値

本当に、ユーザーが頻繁に直面する明確な問題を解決しているか？

### 2.2 コスト

次のものを含みます。

- モデル呼び出しコスト
- エンジンニアリングの保守コスト
- 人手による確認コスト

### 2.3 リスク

次のものを含みます。

- 誤回答のリスク
- コンプライアンスのリスク
- ブランドのリスク

### 2.4 使いやすさ

次のものを含みます。

- 待ち時間
- 説明可能性
- 出力が安定しているか

## 三、まずはプロダクト案の優先順位をつける例を見てみよう

次の例では、とてもシンプルな方法で、  
いくつかのAIプロダクト案を最初に並べ替えてみます。

```python
ideas = [
    {"name": "AI 助教", "value": 9, "cost": 6, "risk": 4, "ux": 8},
    {"name": "AI カスタマーサポート", "value": 8, "cost": 5, "risk": 5, "ux": 7},
    {"name": "AI コードレビュー", "value": 7, "cost": 4, "risk": 6, "ux": 6},
]


def score(item):
    return (
        item["value"] * 0.45
        + (10 - item["cost"]) * 0.2
        + (10 - item["risk"]) * 0.2
        + item["ux"] * 0.15
    )


ranked = sorted(
    [{**item, "score": round(score(item), 2)} for item in ideas],
    key=lambda x: x["score"],
    reverse=True,
)

for item in ranked:
    print(item)
```

### 3.1 この例で何をつかんでほしいか？

プロダクトの判断は、通常1つの軸だけでは決めません。  
あるアイデアが「価値が高い」としても、それだけで一番やるべきだとは限りません。

なぜなら、次の点も一緒に見る必要があるからです。

- コスト
- リスク
- 使いやすさ

### 3.2 なぜ「なんとなくこの方向は面白い」より役立つのか？

それは、次を明確にしないといけなくなるからです。

- どんな基準で判断しているのか

感覚だけに頼らなくてよくなります。

## 四、AIプロダクト設計で特にハマりやすい3つの落とし穴

### 4.1 落とし穴1：モデルの能力から考えてしまい、ユーザーの問題から考えない

たとえば、

- 「大規模モデルがあるから、どこかの場面に入れたい」

こういう発想だと、たいていあまり刺さらないプロダクトになります。

### 4.2 落とし穴2：機能だけ見て、リスクとコストを見ない

デモではとても見栄えが良くても、もし次のようなら、実際のプロダクトにするのは難しくなります。

- 1回あたりの呼び出しコストが高すぎる
- リスクが高すぎる
- 人手でのフォローが重すぎる

### 4.3 落とし穴3：体験の問題を全部フロントエンドの問題だと思う

AIプロダクトの体験は、かなりの部分が次の要素で決まります。

- 出力の安定性
- 待ち時間
- 説明のしやすさ

これは純粋なUIの問題ではなく、プロダクト全体の設計問題です。

## 五、実用的なプロダクト判断の順番

### 5.1 まず、ユーザーがどこで困っているかを確認する

「モデルで何が書けるか」を先に考えず、  
まずはユーザーのどの工程が一番大変かを考えます。

### 5.2 次に、AIはルールや従来の手順より本当に良いかを考える

すべての問題にAIが必要なわけではありません。  
次のような場合もあります。

- ルールで十分
- データベース検索で十分
- フォームのワークフローで十分

### 5.3 最後に、どのモデルを使うかを考える

モデル選定は、たいてい最初の問題ではありません。  
案が固まった後の実装上の問題です。

---

## 六、プロダクト設計で大事な2つの出力

### 6.1 成功基準

たとえば、次のようなものです。

- ユーザーのタスク完了率が上がる
- 平均待ち時間が下がる
- 人手での処理量が減る

### 6.2 失敗の境界

たとえば、次のようなものです。

- どの場面では必ず人に引き継ぐか
- どの出力は自動で通してはいけないか
- どの機能はまだ公開しないか

これによって、プロダクトはより安定し、より現実的になります。

---

## 七、まとめ

この節で最も大事なのは、プロダクトの視点を持つことです。

> **AIプロダクト設計は、まず問題定義とトレードオフの判断であり、その次にモデル選定と機能実装が来る。**

この判断軸がしっかりしていれば、プロダクトを作るときに「機能はすごいけれど、価値は薄い」という状態に陥りにくくなります。
